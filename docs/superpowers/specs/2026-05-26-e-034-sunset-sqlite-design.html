E-034 — Sunset SQLite; Postgres-only substrate

Delete every SQLite read/write in the repo. Migrate orchestrator run state, analytics signals, and experiment readouts to Postgres. One DB, three schemas, Drizzle everywhere. Clean break — no data migration. Standardize tests on truncate-all beforeEach; force the suite serial.

Pattern
One substrate, one driver, one migration tool. The opposite of "use the right tool for each surface."

1. Context — three SQLite surfaces, two Postgres surfaces, one repo

The repo has been progressively migrating off SQLite for months. Today the substrate is split:

Package / surface Substrate Driver Migration tool
@engineerdad/store — 8 business entities Postgres postgres.js + Drizzle Drizzle (pnpm push)
@engineerdad/orchestrator — step_results (claim-check) Postgres raw postgres.js raw SQL files (001_step_results.sql, 002_input_refs.sql)
@engineerdad/orchestrator — runs + run_steps (state) SQLite node:sqlite raw SQL file (001_orchestrator.sql)
@engineerdad/analytics — events, creatives, signals, bandit SQLite node:sqlite raw SQL file (001_init.sql)
@engineerdad/experiment — readout reader SQLite (read-only over analytics) node:sqlite —

The split was incremental and rational at each step. Store moved first (ADR-021) because Notion needed a successor and the entity model is relational. step_results moved next (ADR-022) because the claim-check pattern needed JSONB and rich payload semantics SQLite couldn't give cleanly. The orchestrator run state and analytics tables stayed on SQLite because they "worked fine." They do. But the dual-substrate surface is now its own tax:

  • Two connection idioms. A new test file has to guess: do I prime a temp SQLite, or truncate a Postgres table?
  • Two migration tools. pnpm store:push and pnpm orchestrator:push for PG; applyMigrations() baked into getDb() for SQLite.
  • Two type conventions. SQLite stores JSON as TEXT and timestamps as INTEGER ms; Postgres uses JSONB and TIMESTAMPTZ. Every cross-substrate join is hand-marshaled.
  • Vitest config carries scar tissue. external: ["node:sqlite", /^node:/] exists only because the SQLite branch needs it.
  • Two snapshot/seed strategies in tests. Most recently surfaced in 41c84d3 (gitignore the per-test SQLite fixture) — a workaround that becomes irrelevant once the substrate unifies.
The deeper pattern. E-034 is the inverse of ADR-022 / ADR-023 / ADR-024 — instead of adding a new mechanism to handle a new boundary, it deletes a mechanism that has outlived its purpose. The motivation is not capability (Postgres can do everything SQLite did here) — it's removal of optionality. One substrate means one mental model.

2. Decisions — locked-in shape

Decision Choice Why
Migration scope Clean break, no data migration Local SQLite is dev fixture only; nothing in it is irreplaceable. data/engineerdad.sqlite is deleted from disk; no committed rows to lift.
Schema layout One DB, three schemas (public / orchestrator / analytics) Matches the existing orchestrator.step_results precedent. One DATABASE_URL, one connection pool, clean namespace separation.
Migration tool Drizzle everywhere — also port step_results from raw SQL One pattern. The raw SQL files (001_step_results.sql, 002_input_refs.sql) are deleted in commit 2; Drizzle owns the schema.
Test isolation Truncate-all in beforeEach, globally; store + orchestrator-postgres tests migrate to the same helper Standardize one pattern. Drops the owned-id namespacing brittleness as test count grows.
Test parallelism Force sequential whole-suite (sequence.concurrent: false + poolOptions.forks.singleFork: true) Truncate-all is only safe when nobody else is writing. Wall-clock cost is small — PG-touching tests are the gating tier already.
Commit shape Seven staged commits on one branch; squash on merge Each commit leaves pnpm -r build + pnpm test green. Bisect during development; linear history on main.
ADR ADR-025 (postgres-only) written in commit 7 Captures the why for posterity. Superseded sections in ADR-008/021/022 get a footnote pointer.

analytics.creative_signals
  id          bigserial PRIMARY KEY            -- was INTEGER AUTOINCREMENT
  variantId   text NOT NULL
  channel     text NOT NULL
  platform    text
  kpiName     text NOT NULL

Type-lift decision. Epoch-ms INTEGER columns stay as bigint, not timestamptz. Bandit decay math and time-window queries in packages/analytics treat ts as an opaque number and do millisecond arithmetic; converting to timestamptz would force every caller to EXTRACT(EPOCH ...). JSON TEXT columns DO lift to JSONB — query ergonomics win clearly here.

// packages/shared/src/test-helpers/truncate-pg.ts
import postgres from "postgres";

let sql: ReturnType<typeof postgres> | undefined;
function client() {
  if (!sql) {
    const url = process.env.DATABASE_URL;
    if (!url) throw new Error("DATABASE_URL not set; truncatePg() needs Postgres");
    sql = postgres(url, { max: 2 });
  }
  return sql;
}

export async function truncatePg(): Promise<void> {
  await client().unsafe(`
    TRUNCATE
      public.briefs,
      public.scripts,
      public.authority_articles,
      public.creative_variants,
      public.experiments,
      public.performance_reports,
      public.hypotheses,
      public.learnings,
      orchestrator.runs,
      orchestrator.run_steps,
      orchestrator.step_results,
      analytics.meta_insights,
      analytics.creatives,
      analytics.events,
      analytics.angle_tags,
      analytics.creative_signals
    RESTART IDENTITY CASCADE
  `);
}

export async function closeTruncatePg(): Promise<void> {
  if (sql) { await sql.end({ timeout: 5 }); sql = undefined; }
}

5. Test strategy

5.1 The pattern, everywhere

import { beforeEach, afterAll } from "vitest";
import { truncatePg, closeTruncatePg } from "@engineerdad/shared/test-helpers";

beforeEach(async () => { await truncatePg(); });
afterAll(async () => { await closeTruncatePg(); });

Every PG-touching test file uses the same two lines. No owned-id arrays. No temp file paths. No "did I remember to seed schema X?" — pnpm db:push is a one-time setup; truncate restores empty state per test.

5.2 Whole-suite serialization

Truncate-all is only safe when no other test is mid-write. The root vitest.config.ts enforces this:

// vitest.config.ts
export default defineConfig({
  test: {
    pool: "forks",
    poolOptions: { forks: { singleFork: true } },   // one worker process
    sequence: { concurrent: false },                 // sequential within file
    // ...
  },
});
Cost. The full suite runs serially as of commit 1. Wall-clock impact is modest because the pre-existing PG-touching tests (store, orchestrator-postgres) are already the gating tier, and they'd serialize anyway under the new helper. Pure-unit packages (packages/shared, packages/notion-bootstrap) could in principle override with their own per-package vitest config that re-enables parallelism — out of scope for E-034, file as follow-up if anyone notices.

5.3 DB lifecycle

  • Local dev: pnpm store:up (Docker Postgres) creates both engineerdad and engineerdad_test. One-time pnpm db:push applies all three schemas to both. Tests truncate between runs — never re-push.
  • CI: spin Postgres service; run pnpm db:push once; run pnpm test. No fixture file copying.
  • Removed: all better-sqlite3-flavored fixture priming. The data/ dir holds no test artifacts after commit 7.

8. Rollback

One-way migration. SQLite is fully gone; there's no driver-flip switch. If something blows up post-merge, the path is:

  1. Revert the squash commit on main.
  2. pnpm install to restore better-sqlite3 via the lockfile.
  3. Re-create data/engineerdad.sqlite by running pnpm test once (the SQLite migrations recreate schema; existing rows are lost regardless).

The rollback path is acceptable because (a) the local SQLite holds dev-fixture data only, (b) the branch is heavily tested before merge, and (c) the substrate already moved twice (store, step_results) without rollback being needed.
