Branch-DB MCP Routing

Date: 2026-05-27 Branch: feat/brain-experiment-params accepted

1. Context

Every MCP server in .mcp.json launches with --env-file=.env, which hardcodes DATABASE_URL to the live engineerdad database. .env.local (where pnpm db:sandbox writes the branch sandbox URL) is ignored by the MCPs even though it is the canonical per-developer override file and is already gitignored.

This forces every /loop run, every store write, every analytics ingest from a feature branch to land in production data. The split was originally documented as “sandbox is for tests & migrations; runs go live,” but in practice branches now want to exercise the loop against real-shaped data (a snapshot of live) before merging — without the cost of corrupting live state on partially-built code.

Concrete trigger: while resuming run_1779779169 through HG4 on feat/brain-experiment-params, the orchestrator hit Cannot read properties of undefined (reading 'includes') in planDistribution() — a real D1-query field-projection bug. Reproducing or fixing the bug requires re-walking the distribute stage. Today there is no safe way to do that: every retry mutates live data.

2. Goals & non-goals

Goals

  • When a developer is on a feature branch with a populated .env.local, every DB-touching MCP reads the branch sandbox DB instead of live, with zero manual flag-flipping per server.
  • Provide a one-shot way to clone live data into the branch sandbox so the loop runs against real-shaped state.
  • Make it hard to accidentally write to live from a non-main branch.
  • Keep behavior on main with no .env.local unchanged.

Non-goals

  • Continuous live→sandbox replication. Snapshots are explicit, on-demand.
  • Two-way sync. Sandbox is throwaway; data only flows live→sandbox.
  • Production safety hardening of pnpm db:migrate or other intentionally-live tools. They bypass the guard via an explicit env var.
  • Migrating run_1779779169 itself to the sandbox. That run will live out its remaining stages in live (or be abandoned) — the spec ships new behavior for future runs.

3. Approach

Three coordinated changes:

  1. Routing — append --env-file-if-exists=.env.local to every MCP entry in .mcp.json that already loads .env. Node 24's native env-file layering wins on the second file when present.
  2. Snapshot script — new pnpm db:snapshot that pg_dumps live and pg_restores into the current branch sandbox, run inside the existing engineerdad-postgres container.
  3. Safety guard — a shared helper in packages/shared called from the three schema-owning db.ts modules; refuses to connect to live engineerdad when the current git branch is not main, unless explicitly bypassed.
Alternative considered & rejected: a wrapper launcher script (scripts/mcp-launch.mjs) that loads dotenv + runs the guard before exec'ing the MCP. More code, more indirection, no real win over native --env-file-if-exists + an in-process guard at the actual DB-connection seam.

4. Detailed design

4.1 Routing change (.mcp.json)

Every MCP server that currently uses --env-file=.env (12 of them) gets a second --env-file-if-exists=.env.local arg. Pattern:

"orchestrator": {
  "command": "node",
  "args": [
    "--env-file=.env",
    "--env-file-if-exists=.env.local",
    "mcp-servers/orchestrator/dist/index.js"
  ]
}

Servers affected: meta-ads, meta-organic, analytics, orchestrator, corpus, experiment, heygen, media-providers-gemini, media-providers-kie-ai, store, youtube, plus the others that currently use --env-file=.env. The handful that don't load .env today (asset-store, static-renderer, external engineerdad_site) are untouched — they have no DB or env coupling to begin with.

--env-file-if-exists requires Node ≥ 22.7. Repo runs Node 24, so this is fine. If the file is missing, the flag is a no-op (silent), so behavior on a fresh clone of main with no .env.local is identical to today.

Restarting Claude Code is required for MCP re-registration (already noted in CLAUDE.md). Documented again in the updated workflow section.

4.2 Snapshot script (scripts/db-snapshot.mjs)

New ESM script invoked via a new top-level db:snapshot pnpm script. Mechanics:

  1. Resolve source = engineerdad, target = branchSlug() (the existing helper in scripts/lib/db-slug.mjs).
  2. Refuse if on main (nothing to clone to) or if the target DB doesn't exist yet — print “run pnpm db:sandbox first” and exit non-zero.
  3. Inside the engineerdad-postgres container, run pg_dump --no-owner --no-acl --clean --if-exists -Fc engineerdad > /tmp/<slug>.dump.
  4. Inside the same container, run pg_restore --no-owner --no-acl --clean --if-exists -d <slug> /tmp/<slug>.dump.
  5. Re-apply pnpm db:sandbox's push step at the end (Drizzle push is idempotent) so the snapshotted schema definitively matches what's checked in on the branch — covers the case where the branch has schema changes the live DB doesn't.
  6. Delete the temp dump file inside the container.

One progress line per phase. Total runtime expected in seconds at current data volumes.

Why docker exec rather than host-side pg_dump: this session already proved no PG CLI is installed on the host. Container-side execution removes that prerequisite and keeps the script working for anyone with the standard pnpm store:up setup.

4.3 Boot-time safety guard

New helper packages/shared/src/db-guard.ts exporting assertDbSafeForBranch(databaseUrl). Called once per package at the first getDb() invocation, in three places:

  • packages/store/src/db.ts
  • packages/orchestrator/src/db.ts
  • packages/analytics/src/db.ts

Rule:

If the resolved DATABASE_URL resolves to database name engineerdad (no suffix) and the current git branch is not main, throw an error naming the branch and suggesting:
pnpm db:sandbox && pnpm db:snapshot

Branch detection: read .git/HEAD directly via fs.readFileSync walking up from process.cwd() until found. No git subprocess. If .git/HEAD can't be located (production image, CI worker without git, ephemeral container), the guard is a no-op — it is a developer foot-gun blocker, not a runtime invariant.

Escape hatch: env var ALLOW_LIVE_DB=1 bypasses the guard. Used by:

  • pnpm db:migrate (legitimately writes live from a branch)
  • scripts/db-snapshot.mjs (reads live to dump)
  • Any one-off operator script that has explicit intent

Lazy ordering: the guard runs the first time the DB client is constructed (inside getDb()), so a process that boots and never touches the DB never trips the check. Mirrors how DATABASE_URL not set already behaves.

6. Risks & mitigations

RiskMitigation
pg_dump version drift between container and snapshot consumer Both pg_dump and pg_restore run inside the same container against the same server. No host-side PG tools used.
Snapshot includes orchestrator runs created on live; mixing them with new branch runs is confusing Acceptable — snapshot is a point-in-time clone. The webapp run list already namespaces by runId so collisions are impossible (timestamps + suffix). Run state stays consistent.
Guard false-positive in CI / production Guard is git-aware: with no .git/HEAD reachable, it short-circuits. CI usually runs detached HEAD or in containers without .git; covered.
Existing branch sandboxes (no live data loaded yet) silently used as “empty live” Documentation: db:sandbox + db:snapshot are now the standard branch setup. CLAUDE.md updated.
Forgotten ALLOW_LIVE_DB=1 on legitimate scripts → guard throws on a real migration The two scripts that need it (db:migrate, db:snapshot) set it themselves. Manual one-offs accept the friction; error message names the env var.

  • Integration / manual: on feat/brain-experiment-params: run pnpm db:sandbox, then pnpm db:snapshot, restart Claude Code, run /status via the orchestrator MCP — verify it shows the cloned runs from live and that a fresh /loop writes into the sandbox not live.
  • Negative: on main with no .env.local, MCPs continue connecting to live (regression check). Confirm /status matches today.
  • CI: existing test suite already requires DATABASE_URL to point at _test or _sb_*; guard is consistent with that policy.

8. Out of scope & follow-ups

  • The original D1-query field-projection bug in packages/orchestrator/src/stages/distribute.ts — separate fix, but unblocked by this work (we'll be able to re-walk D2 against a snapshot).
  • Compressing or anonymising the snapshot. Single-user dev env; not needed.
  • Auto-running db:snapshot on branch checkout. Manual is fine.
