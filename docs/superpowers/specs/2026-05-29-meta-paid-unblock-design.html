Meta-paid unblock — B-025 + B-026 + B-027 + webapp angle surface

2026-05-29 · target branch feat/meta-paid-unblock · closes B-025, B-026, B-027 · adds webapp angle-visibility surface · unblocks first end-to-end /loop-once that lands PAUSED Meta-paid entities in Ads Manager

1. Context

The end-to-end walk on run_1779895374 (2026-05-28) reached HG3 cleanly after Brain Initiative Phase 0.5 closed B-022/B-023/B-024 — but the run could not produce Meta-paid entities. Three independent gaps block the path from HG3 to HG4:

  1. B-026 — brief-writer renames Brain's canonical angle keys (e.g. epf-shortfall-parent-worry → epf-shortfall-math). mapCellsToVariants joins by exact string equality on brief.angle; every multi-angle experiment silently empties its cells.
  2. B-025 — verifyExperiment halts on the first empty cell. Operator triage (approving variants from fewer angles than Brain proposed) routinely produces empty cells. The loop halts before distribute is reached.
  3. B-027 — three independent defects in the Meta-paid D2a setup path: (i) create_campaign MCP tool missing is_adset_budget_sharing_enabled (Meta API subcode 4834011); (ii) planner emits daily_budget_cents = 0 (missing MYR→cents conversion); (iii) planner emits adset args with no targeting object (Meta requires it).

All three must land for a clean walk to reach HG4 with Meta-paid PAUSED entities ready for human activation (HG4 is the human flip-to-ACTIVE gate per ADR-015). A fourth concern surfaces alongside: the canonical angle taxonomy is invisible in the review UI today, so an operator approving Briefs at HG1 has no way to see Brain's recommendedAngles, no way to see angle coverage across the 12 Briefs, and no preview of the experiment-status implication of their approval choice.

Scope discipline. This spec closes three named bugs and adds the minimum review-UI surface that prevents the operator from stumbling into a broken or unintended single-cell experiment. Out of scope: B-028 (Reel/YT video pipeline, P3 large), B-029/B-030/B-031 (organic compliance, article fanout, Feed→Meta-organic — separate spec, not on the Meta-paid critical path), partial-rework retry contract (filed as E-044, referenced from §3.1.4).

2. Goal & non-goals

2.1 Goal

A multi-angle /loop-once from a clean branch sandbox walks through HG1 → HG2 → HG3 → HG4 with minimal operator approvals (1 Brief per angle, 1 Script per Brief, 1 Carousel + 1 Feed variant) and lands ≥1 Meta-paid campaign in Ads Manager with ≥1 adset per occupied cell, all PAUSED, each adset carrying non-zero daily_budget and a non-empty targeting block.

2.2 Non-goals

  • No video pipeline fix (B-028 / E-004 remain open; Reel + YT-Long variants stay legitimately skippable at HG3).
  • No organic-compliance, article-routing, or Feed→Meta-organic fixes (B-029/B-030/B-031 — separate spec).
  • No partial-rework retry contract (operator-facing UX, deferred to E-044).
  • No multi-factor experiment generalization (E-042/E-043 — angle stays the only factor in this spec).
  • No critic-step pattern (E-033/E-036/E-037/E-038 — deferred until Brain Initiative Phase 3).

3. Design

3.1 Part 1 — B-026: canonical angle taxonomy enforced

3.1.1 The contract

Brain's Decision Memo is the canonical source of the experimental angle taxonomy for the run. brief-writer consumes that taxonomy and MUST stamp brief.angle with one of those strings verbatim. The angle is one column on a Brief; the 12-Brief count is independent of the angle count — variety lives on persona × promise × proof_type × funnel_stage × budget_bucket.

Singular per Brief. The briefs.angle column stays a single text not null. A Brief carrying multiple angles would make mapCellsToVariants ambiguous (which cell does the downstream Variant belong to?), defeating the experiment. The way to span N angles is N Briefs.

3.1.2 Spawn-prompt arg (orchestrator)

Today brief-writer infers recommendedAngles from the Decision Memo prose. That inference is the drift surface. Lift the values into a structured arg in the spawn prompt — same pattern Phase 0 used for experimentParams.

In packages/orchestrator/src/stages/brief.ts (the B1-author spawn step's build()):

  • Read the resolved S1-reason step result via BuildContext.stageInput() (the ADR-024 staging primitive).
  • Extract recommendedAngles: string[] by looking up the factor by name (factors.find(f => f.name === "angle")?.levels) — not by index. Today Brain only emits a single "angle" factor, but E-042/E-043 may add more, and index-based lookup would silently break.
  • Stage the array as a ref; spawn prompt carries the dereferenced strings.

The spawn prompt template gets a new section:

## Canonical angle taxonomy

Brain's Decision Memo recommends these angles for this run:

  {{recommendedAngles}}

You MUST set `brief.angle` to one of these strings VERBATIM on
every Brief you create. No renaming, no abbreviation, no
paraphrasing. If you cannot reach 12 Briefs within these angles
plus the other axes (persona, promise, proof_type, funnel_stage,
budget_bucket), emit fewer Briefs and surface the shortfall in
your return JSON.

3.1.3 Verifier (closes the loop)

In packages/orchestrator/src/verifiers/verify-brief.ts:

  • Read recommendedAngles from the resolved S1-reason step result (same source as the spawn prompt).
  • For every Brief row the worker created, assert brief.angle is in the set. Collect ALL violations (do not short-circuit on first) so the operator sees the full shape of the drift in one halt message.
  • Return { ok: false, problems: ["Brief bf_3 angle 'epf-shortfall-math' not in recommendedAngles", ...] } on any violation.

Skip-when-absent. If recommendedAngles is absent from the memo (the legitimate-skip path Phase 0 allows for cold-start single-angle runs), the verifier does not assert. This preserves the existing escape hatch.

3.1.4 Recovery contract

On verify.ok === false, the existing one-shot conductor retry fires (.claude/commands/loop.md:56–63). The retry re-spawns the entire brief-writer worker — same prompt, but with one critical difference now: recommendedAngles is an explicit structured arg, not an inference, so the drift cause is gone on the retry.

If the retry still produces off-taxonomy Briefs, the run halts and the operator unblocks manually (delete bad Briefs in DB, re-spawn, or patch brief.angle in-place). This matches the recovery contract every other verifier uses today.

Deferred — E-044. Partial-rework retry (verifier returns per-row bad IDs; orchestrator re-spawns brief-writer with "fix only these N Briefs, leave the other M alone" prompt) is a new contract — needs verifier-shape changes, conductor logic, and brief-writer idempotency. File as a separate enhancement when this spec ships. Trigger to re-open: ≥2 cycles where post-fix brief-writer still drifts on retry.

3.1.5 Schema tightening — briefs.angle NOT NULL

In packages/store/src/schema.ts, change angle: text("angle") to angle: text("angle").notNull(). The B-026 contract makes null an error; the column type should reflect that.

Backfill check. Verified 2026-05-29: existing Briefs all carry a non-null angle (brief-writer has always emitted it). Plan must re-verify at migration time: SELECT COUNT(*) FROM briefs WHERE angle IS NULL; on live engineerdad must return 0 before applying the migration. If any null rows are found, patch them in place from the linked Decision Memo before the NOT NULL constraint lands.

3.2.3 Verifier

In packages/orchestrator/src/verifiers/verify-experiment.ts:

const occupied = cells.filter(c => c.variantPageIds.length > 0).length;
const experimentStatus =
  occupied === cells.length ? "full"
  : occupied >= 2           ? "degraded"
  : occupied === 1          ? "single-cell"
  :                           "broken";

if (experimentStatus === "broken") {
  return { ok: false, problems: ["all cells empty: no approved variants mapped to any cell"] };
}
return { ok: true, problems: [], data: { experimentStatus } };

The verifier carries experimentStatus in its data field so the downstream X3-write step can pick it up without recomputing.

3.2.4 X3-write

In packages/orchestrator/src/stages/experiment.ts, the X3-write step that persists the Experiments row now writes experimentStatus on every row (including full). Source of truth is the verifier's data.experimentStatus from the prior X2-verify step.

3.2.5 Distribute (no branching)

All three pass-states route the same way. plan-distribution.ts reads variants and routes whichever cells are populated; empty cells naturally contribute zero variants. No special case for single-cell.

3.2.6 /reflect (downstream consumer)

The Reflect procedure (.claude/commands/reflect.md + brain agent §B-step-2) reads experimentStatus from the prior cycle's Experiments row. On single-cell, Brain skips Hypothesis graduation for that cycle (a single cell is not a comparison; nothing to confirm or refute). On degraded, Brain grades only the populated cells. On full, no change.

3.3.2 Planner budget math

In packages/orchestrator/src/distribute/plan-distribution.ts (line 154 today):

Before:

daily_budget_cents: Math.round(dailyBudgetMyr * cell.allocationPct),

After:

daily_budget_cents: Math.max(1, Math.round(dailyBudgetMyr * cell.allocationPct * 100)),

Two fixes in one line:

  • The missing * 100 — dailyBudgetMyr is MYR, daily_budget_cents is cents. Today the field is named cents but holds MYR-rounded.
  • The Math.max(1, …) floor — if dailyBudgetMyr defaults to 0 (no --daily-budget on the run) or allocationPct rounds to 0 for a tiny cell, the result violates Meta's exclusiveMinimum: 0. The floor at 1 cent makes the call legal even if the budget signal is missing; the operator sees the absurdly small budget in Ads Manager and corrects it human-side.

Budget source-of-truth. Today the planner reads dailyBudgetMyr from run.params only (the parsed --daily-budget=N CLI arg). Change the read order to: (1) Brain's experimentParams.dailyBudgetMyr from the resolved S1-reason step result (the right source — Brain's hypothesis carries the experimental budget), (2) falling back to run.params.dailyBudgetMyr (operator CLI override), (3) falling back to 0 (caught by the Math.max(1, …) floor). Audit packages/orchestrator/src/run-args.ts to ensure the CLI default stays 0 rather than masking missing Brain output with a hard-coded number.

3.3.3 Planner targeting derivation

Same file. Meta create_adset requires a targeting object. The targeting goes on the adset, and today's distribute architecture is one adset per cell (cell = angle × budget bucket) with multiple ads attached (one per variant × language). Verified by reading plan-distribution.ts:149–162: adsetStep(runId, cell, dailyBudgetMyr) builds one adset per cell; creativeStep + adStep hang the per-(variant × language) ads off that adset.

Because every adset carries bilingual ads (EN + MS variants of each Variant), the adset's locales array must include both languages. Andromeda doctrine (creative-as-targeting; audiences stay broad) reinforces this — the targeting block should be minimal, not language-segmented.

function targetingForCell(_cell: AllocatedCell): MetaTargeting {
  return {
    geo_locations: { countries: ["MY"] },
    age_min: 25,
    age_max: 55,
    locales: [LOCALE_ID.en, LOCALE_ID.ms],
  };
}

// Meta locale IDs are integers from Meta's Graph API.
// Look them up at implementation time via authenticated calls:
//   GET /v18.0/search?type=adlocale&q=English   → find "English (US)" id
//   GET /v18.0/search?type=adlocale&q=Malay     → find "ms_MY" / "Malay" id
// Document the source URL + retrieval date in a comment next to the constant.
const LOCALE_ID: { en: number; ms: number } = { en: /* TBD */ 0, ms: /* TBD */ 0 };

The _cell parameter is unused today but kept in the signature: future cell-aware targeting (per-persona age bands once persona becomes an experimental factor — E-042) reads the cell. Wiring the param now avoids a downstream signature change.

Implementation step (locale IDs). The two integer values in LOCALE_ID MUST be looked up via an authenticated Graph API search at implementation time — web search did not yield a trustworthy source. The first plan step in Part 3 should be: hit /v18.0/search?type=adlocale&q=English + q=Malay using the existing META_ACCESS_TOKEN, write the resulting IDs into the constant with a dated source comment, and add a unit test that asserts the constant's shape (both keys present, both non-zero).

Wire site. In adsetStep(), add targeting: targetingForCell(cell) to the args object alongside daily_budget_cents et al.

Why no per-language adset split. Splitting one adset into two per cell (one EN, one MS) would double the adset count, sub-split the per-cell budget, and force a redesign of the experiment's cell topology. The current "one adset per cell with bilingual ads + bilingual locales" preserves the experiment structure and keeps targeting minimal — the doctrinally correct choice. If language ever becomes an experimental factor (E-043), the split happens at the experiment design layer, not the planner.

Why no persona-derived age band. Persona is Andromeda's parent archetype (e.g. concerned-EPF-parent); ages overlap across archetypes (a "young-parent" archetype is not strictly 25–35). Hard-coding age bands per persona would (a) bake one specific Andromeda taxonomy into the planner, and (b) contradict creative-as-targeting doctrine. Future persona-as-experimental-factor (E-042) is the right place to introduce persona-driven targeting; not here.

3.4 Part 4 — Webapp angle-visibility surface

The angle taxonomy is invisible in the review UI today. An operator at HG1 sees 12 Briefs with various angle strings but no anchor to Brain's recommendedAngles, no count per angle, no preview of the experiment-status implication of their approval choice.

Webapp shape (confirmed by directory inspection). The review UI is a unified dynamic-route Next.js app, not per-entity pages:

  • Run detail: apps/webapp/src/app/runs/[runId]/page.tsx
  • Entity list (Briefs / Scripts / CreativeVariants / etc.): apps/webapp/src/app/review/[entity]/page.tsx — one dynamic route, behavior driven by per-entity list config.
  • Entity detail: apps/webapp/src/app/review/[entity]/[id]/page.tsx
  • Per-entity list configs: apps/webapp/src/app/lib/listConfigs/<entity-slug>.ts — currently creative-variants.ts, distributions.ts, experiments.ts, hypotheses.ts, runs.ts, plus default.ts as fallback (Briefs and Scripts currently inherit from default.ts — no dedicated config yet).

Implication: adding an Angle column to Briefs / Scripts means creating new list-config files (briefs.ts, scripts.ts) registered via lib/listConfigs/index.ts. Adding it to CreativeVariants is an edit to creative-variants.ts (one new column entry).

Three additions, smallest-to-largest:

3.4.1 Run header — angle chips

In apps/webapp/src/app/runs/[runId]/page.tsx, render Brain's recommendedAngles as labeled chips at the top of the page. Each chip:

  • Shows the angle key verbatim: epf-shortfall-parent-worry
  • Shows a count of Briefs under it: · 4
  • On hover/click, shows the Decision Memo's rationale text for that angle (extracted from the memo's angles[].rationale field if present).

Source of truth: the resolved S1-reason step result, fetched server-side via read_step_result or directly from the orchestrator.step_results table (the page is a server component; either works).

3.4.2 Briefs / Scripts / Variants — angle column

Add an Angle column to each entity's list config:

  • New apps/webapp/src/app/lib/listConfigs/briefs.ts — columns include title, angle (new), persona, funnelStage, budgetBucket, approvalStatus, runId. Register in lib/listConfigs/index.ts.
  • New apps/webapp/src/app/lib/listConfigs/scripts.ts — columns include title, angle (joined via script.brief → brief.angle), approvalStatus, runId. The join is server-side in the review/[entity]/page.tsx route — extend the existing CreativeVariants script-title-prefix pattern (see page.tsx:50–60) to handle a Brief lookup for Scripts.
  • Edit apps/webapp/src/app/lib/listConfigs/creative-variants.ts — add { field: "angle", label: "Angle", type: "badge", sortable: true }. The angle is derived two FK hops away (variant.script → script.brief → brief.angle); extend the existing route's pre-fetch loop to fetch the joined Brief for each Variant.

The angle field is not native on Scripts or CreativeVariants — it's a denormalized read at render time. Computing it in the route handler keeps the list configs declarative. Optional UX nicety (deferred decision at implementation time): group rows by angle, or add a filter chip. The baseline is the column.

3.4.3 HG1 approval guidance — experiment-status preview

Above the Briefs list (or inline as a sticky toolbar), surface a live preview of the resulting experiment status given the current approval set:

You have approved Briefs from 2 of 3 recommended angles.
Approving as-is will produce a `degraded` experiment.

  • epf-shortfall-parent-worry   · 2 of 4 approved
  • children-fund-rm200-compounding · 2 of 4 approved
  • pcsf-2007-cautionary         · 0 of 4 approved  ← empty cell

Approving Briefs from 1 angle only would produce a `single-cell`
result (routing, not experiment).

Component shape. The Briefs list page is a Next.js server component today (it awaits store.query directly). Approvals fire via server actions that revalidate the page. So the preview has two implementation options: (a) server-side recompute on each approval action — preview state is just a function of the current DB rows, redrawn on revalidate; (b) client component wrapping the approve buttons with optimistic local state for an instant preview update before the server roundtrip. Option (a) is simpler and consistent with the existing approval flow; option (b) is snappier. Default to (a); revisit if the round-trip feels laggy in the §7 walk.

Either way, the computation mirrors the verifier's tri-state logic in §3.2.3 via a shared helper exported from packages/shared so the webapp and the verifier can't diverge. This is the highest-leverage piece of Part 4 — it converts experiment status from an after-the-fact discovery to a deliberate approval choice.

7.8 Failure → replay loop

If any gate fails:

  1. Identify which snapshot precedes the failure (e.g. 06-hg3-approved.sql).
  2. Patch the bug on a feature branch.
  3. Restore: docker exec ... psql -d $SANDBOX_DB -f $SNAP_DIR/06-hg3-approved.sql (drop + recreate the DB first if needed: dropdb -U engineerdad $SANDBOX_DB && createdb -U engineerdad $SANDBOX_DB).
  4. Restart Claude Code (MCP cache invalidates on session start).
  5. Resume from the failing stage with /distribute --run=<id> (or whichever stage failed). No need to re-walk earlier gates.
