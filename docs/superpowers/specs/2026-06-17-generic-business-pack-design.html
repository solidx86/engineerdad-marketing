1Context & problem

The repo is a closed-loop "Marketing OS" — a nine-stage loop (tracking → analytics → synthesize → brief → content → produce → schedule → experiment → distribute) driven by an orchestrator state machine, four agentic cells, and ~15 stdio MCP servers over deterministic packages/ libraries. The engine (orchestrator, store, analytics, MCP adapters, render pipeline) is already largely domain-agnostic. The business — EngineerDad, a Public Mutual unit-trust + PRS consultancy for Malaysian parents — is coupled into the codebase in four places:

  1. The corpus (corpus/) — compliance, courses, proof, knowledge, data, brand templates. Nearly all EngineerDad content.
  2. Config/infra identifiers — the engineerdad live-DB name, the @engineerdad/* package scope across 22 packages, .env, .mcp.json, docker-compose.yml, package.json.
  3. Baked-in enums — PersonaSchema (a hardcoded Zod enum), Lang = "en" | "ms", the PERSONA_AGE_BAND / Meta-targeting block in derive/specs.ts, and the Malaysian-regulator compliance model.
  4. Prompts — house-style.md, bilingual.md, tactical-piliero.md (a mix of generic technique and brand voice), plus corpus/courses/brand-voice.md.

The goal: a person with two businesses — EngineerDad (financial) and, say, a home-automation brand — should be able to clone the repo, swap the business pack, populate the corpus, and run the loop, with zero edits to engine code.

The driving test case. The home-automation business is not financial, may not be bilingual, and has no Malaysian regulator. It directly breaks the three hardcoded enums: PersonaSchema (8-ish Malaysian-parent tokens), Lang (en|ms), and the SC/FIMM/Public-Mutual compliance assumption. Today, supporting it means editing TypeScript and Zod schemas. The whole point of this work is to turn that baked-in doctrine into business-pack data.

2Framing decisions

Settled during brainstorming:

  • Tenancy model: fork-per-business. Not config-switch, not runtime multi-tenant. One repo per business; to start a new one you clone, rename the local repo, swap the pack, and loop. Single-tenant per fork.
  • Future-business shape: same channels, new industry. The marketing-OS engine and channels (Meta paid + organic, YouTube, site handoff) are reused. Personas, compliance regime, knowledge, proof, and brand differ substantially — so the doctrine must loosen from compile-time enums to pack data.
  • Approach: pack boundary + data-driven doctrine, phased. Neutralize engine identifiers once (upstream); formalize a single active business pack; refactor the hardcoded enums to load from the pack. Ship in two phases.

3Goals & non-goals

Goals

  • A clean, documented engine ↔ business-pack boundary. The engine carries no business literals.
  • A single typed BusinessConfig as the one source of business truth a fork edits.
  • Persona, language codes, Meta targeting, compliance, channels, brand all read from the pack — not from code enums.
  • A fork needs no engine-code renaming: identifiers are neutralized once, upstream.
  • A corpus:lint readiness check so a fork knows when its corpus is loop-ready.
  • EngineerDad remains the working reference pack (the main repo is the EngineerDad fork).

Non-goals

  • Runtime multi-tenancy or co-resident config-switching (explicitly rejected).
  • More than two language slots (see §9 — the twin-field shape stays; N>2 is a separate future project).
  • New ad platforms beyond the existing Meta / YouTube / site-handoff set.
  • Rewriting the agentic cells' reasoning — only their inputs (corpus + config) change.

4The engine ↔ pack boundary

Engine (business-agnostic; renamed once upstream): packages/, mcp-servers/, apps/, scripts/, the .claude/ runtime, the orchestrator stages/verifiers, and the engine-owned render procedure files. Carries no business literals. The @engineerdad/* scope becomes a neutral @os/*; the live-DB name and db-guard read from config.

Business pack (the only thing a fork edits):

  • business/config.ts — one typed BusinessConfig object (§5): identity, languages, personas, channels, compliance, brand-voice pointer, corpus root.
  • corpus/ — stays at repo root (config points at it; default ./corpus). All business content (§8). On fork, replace contents.

The engine reads the pack through a tiny loader package (working name @os/business) that imports the active business/config.ts, Zod-validates it, and caches it. Engine modules that hold business constants today instead read off this config.

EngineerDad becomes the reference pack. Its business/config.ts + corpus/ are what ship in the main repo — the repo doubles as the EngineerDad fork and the worked example a new fork copies from.

5The BusinessConfig schema

A sketch of the shape (final types settled during implementation; this is the contract, not the literal source):

interface BusinessConfig {
  identity: {
    slug: string;          // "engineerdad" — drives live-DB name, R2 key prefixes, db-guard
    displayName: string;   // "EngineerDad"
    landingUrl: string;    // https://engineerdad.my  (LANDING_URL today)
  };
  languages: {
    primary: LangSpec;            // { code: "en", label: "English" }
    secondary?: LangSpec;         // { code: "ms", label: "Bahasa Melayu" } | omitted → monolingual
  };
  personas: PersonaDef[];         // replaces the PersonaSchema enum
  channels: {
    metaPaid:    { enabled: boolean; targetingBase: MetaTargeting;
                   igHashtags: string[]; fbHashtags: string[] };
    metaOrganic: { enabled: boolean };
    youtube:     { enabled: boolean; descriptionFooter: string };
    siteHandoff: { enabled: boolean };   // the engineerdad_site cross-repo draft handoff
  };
  compliance: {
    rulesPath: string;                       // default ./corpus/compliance/banned-phrases.yaml
    disclaimers: Record<LangCode, string>;   // the DISCLAIMER map in specs.ts
    regulatorHints: string[];                // REGULATOR_HINTS — "already disclaimed?" probe
  };
  brandVoicePath: string;   // ./corpus/courses/brand-voice.md
  corpusRoot: string;       // ./corpus
}

interface PersonaDef {
  id: string;            // "young_parents_25_35"
  label: string;
  description: string;
  targeting?: { ageMin: number; ageMax: number };  // replaces PERSONA_AGE_BAND
}

interface MetaTargeting {                 // replaces TARGETING_BASE
  countries: string[];                    // ["MY"]
  ageMin: number; ageMax: number;         // 25 / 45
  publisherPlatforms: string[];
  facebookPositions: string[];
  instagramPositions: string[];
}

7Identity neutralization (one-time, upstream)

  • Package scope: @engineerdad/* → @os/* across all package.json names and imports (22 packages). One-time rename; forks never touch it.
  • Live-DB name & guard: db-guard.ts hardcodes "engineerdad" as the live DB; DATABASE_URL defaults to it. Both derive from config.identity.slug (the guard's "is this the live DB?" check reads the configured slug, not a literal).
  • Infra files: docker-compose.yml service/DB names, .mcp.json, root package.json name (engineerdad-marketing → marketing-os), .env.example footer/landing values.

Outcome: a fork needs zero engine-code renaming. Only business/config.ts + corpus/ change. The slug is the single knob that re-points DB name and asset prefixes.

8The corpus

The key insight: the corpus is not uniformly "business content." It is three different ownership classes wearing one folder.

Corpus pathOwnerFork action
compliance/ (rules, banned-phrases.yaml, phrasebank)PackReplace — your regime, or a near-empty no-op file if unregulated.
proof/, knowledge/, courses/ (topics), data/ (datasets + charts)PackReplace wholesale — pure business content.
templates/brand-contract.md, templates/reference-designs/, courses/brand-voice.mdPack (brand/taste)Replace — your palette, voice, visual exemplars.
templates/worker-prompts/ (render-worker.md, reel-render-worker.md, _chart-rules.md)EngineKeep as-is — generic render procedure, not business content.
Coupling smell named, not fixed. Engine procedure files (the render-worker prompts) live under corpus/ and are referenced by path across agents + ADRs. Phase-1 choice: mark them engine-owned in place — a fork keeps them untouched. Relocating them to an engine templates/ dir is deferred (pure path churn for no behavioural gain).

The contract is the frontmatter, not the content

The engine reads frontmatter keys, not prose:

  • knowledge/: cluster · funnel_tier · granularity · source_type · lang_status · related
  • proof/: persona · permission_status · attribution
  • data/charts/*.yaml: id · title_en/ms · chart_type · x_label_en/ms · labels
  • data/datasets/*.json: provenance block (verified_at · source · …)

The shape is the engine contract; the content is the pack. A fork keeps the keys, rewrites the values. Two keys point back at business/config.ts and must agree:

  • persona: tags in proof/ must be ids in config.personas (or all).
  • Bilingual title/label/lang_status fields must match config.languages' declared slots.

New: corpus:lint — the readiness green-light

Today "populate the corpus" is guesswork; nothing tells a fork when its corpus is complete enough to loop. corpus:lint (a new script) will:

  1. Check the required skeleton exists (the six scope dirs + compliance/banned-phrases.yaml + templates/brand-contract.md).
  2. Validate each file's frontmatter against a per-scope schema (Zod / JSON-schema).
  3. Cross-check persona and language tags against business/config.ts.
  4. Flag stale identity leakage — residual engineerdad / Public Mutual / Malaysia strings — so a fork knows it hasn't left reference content behind.

This converts the corpus from a folder you hope is right into a pass/fail checklist — the thing that makes "clone → populate → loop" safe.

Fork workflow for the corpus

empty the Pack-owned dirs (keep skeleton + worker-prompts)
  → author compliance / proof / knowledge / courses / data / brand
  → pnpm corpus:lint        # readiness green-light
  → /ingest-corpus          # build the BM25 index (gitignored; soft-fails empty)
  → /loop

Phase 2 ships business/_template/corpus/ — the empty skeleton with per-scope READMEs documenting the frontmatter contract — so a fork copies a clean skeleton instead of gutting EngineerDad's content file by file.

9Language: slot-based bilingual (the key compromise)

Every entity is twin-shaped: BilingualSchema = {en, ms}, scriptBodyEn/scriptBodyMs, captionEn/captionMs, Record<Lang, …> maps. Truly variable language count would mean renaming twin fields across every schema — a massive, risky change.

Decision: decouple the physical slot from the language it carries.

  • The schema keeps the physical field names en / ms (and …En/…Ms) as slot names: primary slot = the en-named field, secondary slot = the ms-named field.
  • config.languages declares what language each slot renders in (code + label).
  • A monolingual pack omits secondary; the secondary slot mirrors primary (or is left empty, per validation rule TBD-at-impl but defaulting to mirror).
  • A non-EN/BM bilingual pack (e.g. EN + Bahasa Indonesia) keeps the field names and sets secondary.code = "id".
This delivers language flexibility (any one or two languages) without a schema-wide field rename. N>2 languages is explicitly out of scope — a separate future project that would generalize the twin shape to an array.

10Phasing

Phase 1 Boundary + identity → forkable blank-brand

  • Introduce BusinessConfig + the @os/business loader (Zod-validated, fails loud).
  • Neutralize all engine identifiers (scope rename, DB-name from slug, infra files).
  • Lift the specs.ts literals into config; thread config through deriveSpecs.
  • Wire compliance default path + channels to config.
  • Personas become data here (cheap — Zod enum → validated string).
  • Still EN/BM. Outcome: clone, edit config, swap corpus → runs as a different (financial-shaped) brand.

Phase 2 De-enum for cross-industry → passes the home-automation test

  • Parameterize language slot codes/labels; allow monolingual (§9).
  • Make compliance genuinely no-op-able for an unregulated pack.
  • Ship business/_template/ (blank config + corpus skeleton + READMEs) and corpus:lint.
  • Add a /new-business onboarding checklist (doc; optional small scaffold script).
  • Build a thin home-automation skeleton pack as the genericity proof.
  • Migrate the test suite to a test pack fixture (see §11).

11Testing strategy

The current suite bakes in EngineerDad values (engineer_dad_archetype, en/ms, MY targeting). The proof that the engine is genuinely business-clean is: the engine tests run against a synthetic test business pack, not EngineerDad.

  • Introduce a test/fixtures/business/ pack with neutral personas, a single language, and trivial compliance.
  • Engine unit tests (orchestrator, specs derivation, compliance scan, store validation) load the test pack via the loader.
  • EngineerDad's pack gets its own focused tests (its personas validate, its compliance file lints, its corpus passes corpus:lint).
  • corpus:lint ships with tests over good/bad fixture corpora.
  • A "genericity" smoke test: load the home-automation skeleton pack and run plan() through HG1 in a sandbox DB.

12Validation & error handling

  • Config load: BusinessConfig is Zod-validated at first load; a malformed config fails loud with a precise path (no silent defaults that mask a typo).
  • Persona integrity: a Brief whose persona ∉ config.personas fails validation at the store write boundary — the same choke point the compliance scan already uses.
  • Compliance file: missing file → clear error pointing at config.compliance.rulesPath; a present-but-empty file is a valid no-op (documented).
  • Language slots: content written to a slot whose language isn't declared is rejected; a monolingual pack writing to the secondary slot is a validation warning, not a hard fail (mirror-default).
  • Identity leakage: corpus:lint surfaces stale reference strings as warnings (not loop-blocking) so a fork can decide.

14Scope boundaries (YAGNI)

  • No multi-tenant runtime, no co-resident config-switch.
  • No >2 language slots — two-slot shape preserved; N-language deferred.
  • No new ad platforms beyond Meta / YouTube / site-handoff.
  • No relocation of engine-owned worker-prompts out of corpus/ (deferred churn).
  • No change to the agentic cells' reasoning scaffolds — only their corpus + config inputs change.

15Risks & open questions

  • Scope-rename blast radius. @engineerdad/* → @os/* touches every import. Mechanical but wide; best done as an isolated, single-purpose commit with a green build, separate from the logic refactor.
  • Threading config into pure functions. deriveSpecs & friends must stay pure (config injected, not globally read) to preserve testability.
  • Migration to a test pack. Re-pointing the suite off EngineerDad literals is the bulk of Phase 2 effort and the real proof of cleanliness; underestimating it is the main schedule risk.
  • ADR-010 (bilingual EN/BM only). This work supersedes it for the engine: languages become per-pack. EngineerDad's pack still declares exactly en+ms. A new ADR should record the boundary and the slot-based decision (§9), explicitly noting it does not reintroduce a third language slot.
  • Resolved (2026-06-17): the pack lives under a business/ directory — business/config.ts now, business/_template/ as the Phase-2 sibling skeleton. corpus/ stays at repo root to match the "populate the corpus" mental model.
