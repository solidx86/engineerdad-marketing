Agentic Rebuild — Implementation Plan (Roadmap)

For agentic workers This HTML is the human-review plan-of-record. The executable form of each phase is a markdown file with - [ ] checkbox steps, consumed by superpowers:subagent-driven-development or superpowers:executing-plans. Phase 1 is fully detailed in phase-1.html and ready to convert to executable markdown now. Phases 2–7 are scoped here at the task / interface / acceptance-criteria level; each gets its own phase-N.html expanded to bite-sized TDD detail when its predecessors land (their exact code depends on interfaces discovered in execution).

Goal: Rebuild the EngineerDad Marketing OS as a small set of genuinely agentic cells standing on a typed, deterministic orchestrator state machine — replacing eight scripted agents and prose-orchestration with ~4 agentic cells, deterministic stages, and a code-owned run loop.

Architecture: The slash command orchestrates (main session); a typed orchestrator MCP owns the run-state machine (plan / verify / advance); agentic cells and fan-out workers are terminal level-1 spawns. Deterministic transforms and orchestration move out of agent prompts into unit-tested TypeScript. Notion stays the human review surface and gate signal; SQLite is the orchestrator's source of truth.

Tech stack: TypeScript / Node 20+, pnpm workspaces, vitest (TDD), better-sqlite3, @modelcontextprotocol/sdk ^1.18 (stdio), Zod, Claude Code agents + slash commands.

Seven phases, each producing working, testable software on its own. Phase 1 (extract deterministic transforms) is independent, greenfield, and zero-behavior-change — it can land first and alone, killing the recurring G-series drift class. Phases 2–7 then build the orchestrator and reshape the agents on top of it.

3. Phase detail

Phase 1 — Extract deterministic transforms shipped

Objective. Move every deterministic transform currently described in English inside agent prompts into pure, unit-tested TypeScript in packages/shared/src/derive/. Greenfield, zero behavior change, no agent or MCP touched — purely additive. This is the foundation Phases 2–7 import.

Files. Create packages/shared/src/derive/{text,channels,ids,index}.ts + colocated *.test.ts; modify packages/shared/src/index.ts and packages/shared/package.json.

Tasks (10, full TDD detail in phase-1.html).

  1. slugify(title) — kebab-case slug (was media-production Step 8.2)
  2. truncateAtWord(text, max) — word-boundary truncation primitive (Step 5.7a)
  3. readingTime(body) — word-count ÷ 250 → "N min read" (Step 8.2)
  4. topicTag(topic) — uppercase category chip, ≤60 chars (Step 8.2)
  5. extractKeywords(targetQuery, topic) — deduped keyword list, cap 10 (Step 8.2)
  6. variantId(scriptId, format, aspect) — sha256(...)[:12] (Step 5.5 G5)
  7. deriveChannels(format, aspect, funnelStage) + Channel type (Step 5.6)
  8. youtubeCategoryId(notionCategory) — name→ID map (distribution §4b)
  9. metaCtaType(funnelStage, ctaText) — CTA-by-funnel map (Step 5.7a)
  10. Wire-up — barrel index.ts, root re-export, package.json exports, full build

assignSchedule (organic cadence) lands in Phase 4 alongside its consuming schedule stage; deriveSpecs (the composer) lands in Phase 5 alongside creative-director — both belong with their consumers, not here.

Acceptance All 9 functions exist and are exported from @engineerdad/shared/derive; every function has a colocated vitest suite covering happy path + edge cases; pnpm vitest run packages/shared green; pnpm -r build clean (sequential); pnpm sync:agents:check unaffected. No file under .claude/ or mcp-servers/ changed.

Phase 2 — Orchestrator core + /loop shipped

Objective. Build the typed run-state machine and the thin conductor. After this phase, one stage (produce) runs end-to-end under /loop.

Files. packages/orchestrator/ (logic) + mcp-servers/orchestrator/ (thin wrapper) + a SQLite migration + .claude/commands/loop.md & status.md + .mcp.json registration.

Key tasks

  • SQLite migration — runs (run_id, stage, status, params, timestamps) + run_steps (run_id, step_id, stage, status, result JSON, problems, attempts).
  • state.ts — typed read/write over those tables (node:sqlite — Node's built-in DatabaseSync, matching the analytics MCP; not better-sqlite3).
  • engine.ts — plan / verify / advance; plan is a pure function of persisted run state (the resumability property).
  • stages/produce.ts — the StageDefinition (P1 spawn → P2 fanout → P3 write → HG3 gate); verifiers/verifyProduce.ts.
  • mcp-servers/orchestrator/ — stdio server, 3 tools, delegates to the package (ADR-005 thin-adapter); register in .mcp.json.
  • .claude/commands/loop.md — the domain-agnostic conductor (plan → spawn / fanout / write → verify → advance → repeat; stop at gate). status.md — read-only run dashboard.
  • Re-point the granular commands (/produce etc.) at the same conductor, scoped.

Interfaces exposed (locked here, consumed by 3–6)

type StepKind = "spawn" | "fanout" | "write" | "verify" | "gate" | "done" | "halt";
interface Step { kind: StepKind; step_id: string; /* + kind-specific fields */ }

interface StageDefinition {
  id: string;
  entry: (run: RunState) => boolean;
  steps: StepSpec[];          // builders: buildSpawnPrompt / buildUnits / buildCalls
  exit:  (run: RunState) => boolean;
}

orchestrator.plan(runId | { args })          -> Step
orchestrator.verify(runId, stepId, result)   -> { ok: boolean; problems: string[] }
orchestrator.advance(runId, stepId, result)  -> { stage: string; status: string }
Acceptance orchestrator MCP registered and starts clean; given a seeded run with approved Scripts, /loop drives the produce stage to its HG3 gate; verifyProduce unit-tested against fixture run state; a resumability test proves re-entry after a partial failure re-plans only the undone steps; plan never double-creates. pnpm -r build + pnpm vitest run green.

Phase 3 — Collapse distribution + experiment-os shipped

Objective. Routing decisions stay deterministic; the worker derives nothing (ADR-018 intact).

Files. orchestrator/src/stages/{experiment,distribute}.ts + verifiers + the pure planDistribution / allocation functions; the gate-passing mechanism (a check on the gate Step). Agent-file deletion + command re-pointing move to Phase 6 — doing it now, with the stages not yet live and brain.md still referencing them, would leave a broken intermediate state.

Acceptance experiment + distribute exist as unit-tested StageDefinitions; planDistribution, allocation, and the two verifiers are pure + tested; gate-passing works (proven by an integration test); dryRun preserved as a plan flag; idempotency tested (re-run skips back-filled rows); Meta entities still land PAUSED (ADR-015, MCP-enforced). Live-registry wiring + agent deletion: Phase 6.

4. Sequencing & branch strategy

  • Phase 1 is independent and zero-risk — land it first on feat/agentic-rebuild, optionally ship/merge it on its own.
  • Phase 2 is the spine; 3, 4, 5 each depend only on 2, not on each other — they can be done in any order, or in parallel on sub-branches.
  • Phase 6 wires the live loop — it depends on 3, 4, and 5 all being done, since it activates every stage they built and deletes the agents they replaced.
  • Phase 7 is last — it cannot delete brain.md's dispatch sections until every stage that replaced them (2–6) exists.
  • Each phase ends green (pnpm -r build + pnpm vitest run + pnpm sync:agents:check) and is a review checkpoint.
