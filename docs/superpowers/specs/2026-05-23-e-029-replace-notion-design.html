1. Context

E-027 shipped on 2026-05-23 — heavy single-spawn agents fan out into per-Brief / per-Script workers. The first attempt to walk run_1779446750 through the new produce stage hit a different ceiling: the conductor's inline tool-result cap (~32k tokens). P0-scripts queried 6 approved Scripts via the Notion MCP and got back 70k chars — spilled to disk; P1-fanout's plan() response would have been comparable.

The wall is the MCP stdio wire cap, not a Notion API limit. Two correctnesses came out of the post-mortem:

  1. The friction isn't Notion-the-UI — Notion-the-data-source returned the payload to its MCP server cleanly. docs/war-room-wishlist.md still gates against designing a custom UI on the strength of one friction; this is filed as an architectural decision, not as a war-room build.
  2. A local store with a redesigned MCP surface dissolves the wall. Today we ship bulk content over the wire because per-ID reads against Notion cost 100–300 ms each. Local-process per-ID reads are sub-ms. Only with a local store does the cap-honoring surface become economical.

Concurrent payoffs from the same move retire four Notion-shaped frictions: ADR-012's rich_text 2000-char chunking, the filter_properties dance, Notion API rate limits, all eight migrate-* live-workspace coordination scripts.

2. Goal

Replace Notion as the OS's data + human-gate substrate with Postgres running in a Docker container, with the data volume bind-mounted to ./data/postgres/ (gitignored), and a single-user Next.js review app. Redesign the data MCP surface to ship IDs not bulk content, so the wire cap is never approached at v1 scale.

Definition of done:

  • A fresh cold-start /loop-once walks all the way to HG3 on the new infra without any tool result exceeding ~10% of the MCP wire cap.
  • Human approvals at HG1/HG2/HG3 happen in apps/review-ui at http://localhost:3030.
  • The compliance scanner runs synchronously inside the store's create / update paths — same auditable choke point as today, different layer.
  • Postgres comes up via docker compose up -d; schema lands via pnpm store:push; no per-feature migration files.
  • Playwright covers every UIUX-touching surface in the review app (list rendering, detail edit + persist, status flip + persist, markdown side-by-side live preview). Vitest covers the data + MCP layer.
  • mcp-servers/notion/, packages/notion-bootstrap/, and ADR-012 retired. No mcp__notion__* tool calls remain anywhere.
  • pnpm -r build + pnpm test + pnpm test:e2e + pnpm sync:agents:check green.

3. Non-goals

  • 1:1 Notion feature parity — no comments, mentions, sharing, multi-user collab, mobile, real-time presence.
  • Cross-machine sync. Postgres data lives at ./data/postgres/ on the local machine (gitignored). Backup is whatever you do for any local Postgres — pg_dump on demand, Time Machine, etc.
  • Auth. Localhost-only, single user. Postgres binds to 127.0.0.1:5432 only; the review app to 127.0.0.1:3030.
  • Migration of existing Notion content. Clean slate: run_1779446750 is archived in Notion. First run on new infra is a fresh cold-start.
  • Migration tooling at v1. Schema lands via drizzle-kit push against the live container; no per-feature migration files committed. (The door is open to switch to drizzle-kit migrate later if schema-evolution discipline becomes load-bearing.)
  • Replacing the orchestrator state tables. runs + run_steps already live in SQLite (data/engineerdad.sqlite) and stay there — they're orchestration plumbing, not entities. Two stores side-by-side: SQLite for ephemeral run state, Postgres for the 8 entity tables.

6. The redesigned MCP surface

Five tools, all on mcp-servers/store/:

ToolReturnsUsed by
mcp__store__query(entity, filter, fields?) { id, title }[] by default; fields opts in to small index columns. Never bulk content. Stage write-steps (C0-briefs, P0-scripts) and gate checks.
mcp__store__get(entity, id) Full row of one entity. Workers in a fanout — one bulk crossing per worker, within the 200k window.
mcp__store__create(entity, props) { id, ok, problems? }. Runs compliance scan synchronously. Worker writes (Scripts, AuthorityArticles, CreativeVariants).
mcp__store__update(entity, id, props) { ok, problems? }. Fill-only-if-empty mode preserved. Article-enrichment (P4), HG-write-back.
mcp__store__set_status(entity, id, status) { ok }. Review UI server actions.
The cap-honouring contract. query NEVER returns bulk text fields. The only path to read bulk content is get(id). Workers do one get on their assigned unit. The conductor never sees bulk content.

Flat filter DSL (replaces Notion's filter shape)

// Today (Notion shape — verbose, dense, server-side schema-coupled)
{ and: [
  { property: "Run ID", rich_text: { equals: "run_..." } },
  { property: "Approval Status", select: { equals: "Approved" } },
]}

// After (flat object — Drizzle WHERE under the hood)
{ runId: "run_...", approvalStatus: "Approved" }

// AND is implicit. Special operators only where needed:
{ runId: "run_...", approvalStatus: { in: ["Approved", "Rejected"] } }
{ hookCount: { gte: 30 } }

No or at v1 — no stage uses one. Add when needed.

7. The Postgres container

One service in docker-compose.yml. Bind mount to ./data/postgres/ so the volume sits inside the repo (per user direction) but the contents are .gitignored.

services:
  postgres:
    image: postgres:16-alpine
    container_name: engineerdad-postgres
    restart: unless-stopped
    environment:
      POSTGRES_USER: engineerdad
      POSTGRES_PASSWORD: engineerdad
      POSTGRES_DB: engineerdad
    ports:
      - "127.0.0.1:5432:5432"          # localhost-only
    volumes:
      - ./data/postgres/pgdata:/var/lib/postgresql/data    # pgdata subdir — initdb refuses non-empty parent
      - ./scripts/postgres-init:/docker-entrypoint-initdb.d:ro
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U engineerdad"]
      interval: 5s
      timeout: 5s
      retries: 10

Init script (scripts/postgres-init/01-create-test-db.sql): creates the engineerdad_test database used by Playwright. Runs once on first container start.

CREATE DATABASE engineerdad_test;
GRANT ALL PRIVILEGES ON DATABASE engineerdad_test TO engineerdad;

Connection string (.env):

DATABASE_URL=postgresql://engineerdad:engineerdad@localhost:5432/engineerdad
DATABASE_URL_TEST=postgresql://engineerdad:engineerdad@localhost:5432/engineerdad_test

Root scripts:

ScriptEffect
pnpm store:updocker compose up -d postgres — start the container detached.
pnpm store:downdocker compose down — stop. Data persists in the bind-mount.
pnpm store:wipedocker compose down -v && rm -rf data/postgres/pgdata — full reset. The .gitkeep placeholder is preserved.

8. Stage call-site touch-ups

Every stage that calls mcp__notion__query changes its tool name and its filter shape; every stage that reads stepResult for a Notion query response changes its shape from { results: [{ id, properties: { ... } }] } to [{ id, ... small fields ... }]. Builders that previously embedded a property tree into a worker spawn-prompt now embed the id and rely on the worker to call mcp__store__get.

9. Compliance boundary preservation

ADR-005's compliance choke point is intact. The boundary moves from mcp-servers/notion/src/handler.ts to packages/store/src/crud.ts's create and update paths. The scanner itself (packages/shared/src/compliance.ts) is unchanged. Both the MCP path (workers calling create) and the UI path (server actions calling store.create) go through the same function — the boundary is now layer-aware, not transport-aware.

Invariant. No row enters the store without compliance scan. The scan refuses a write on violation. Same shape as today.

10. The review app — minimum viable shape

Next.js 15 App Router with server actions for all writes. Tailwind for layout. No client state library — Next's revalidation handles list refresh after a server action.

Routes (8 entities, same pattern)

/                                — dashboard: in-flight runs + counts per entity
/briefs                          — list view (current run filter)
/briefs/[id]                     — detail + edit + status flip
/scripts, /scripts/[id]          — same
/authority-articles/(...)        — same
/creative-variants/(...)         — same
/experiments, /performance-reports, /hypotheses, /learnings — same

Server actions (one file, ~30 lines total)

// apps/review-ui/src/app/lib/actions.ts
"use server";
import { store } from "@engineerdad/store";
import { revalidatePath } from "next/cache";

export async function updateRow(entity: string, id: string, patch: Record<string, unknown>) {
  const r = await store.update(entity, id, patch);
  if (!r.ok) throw new Error(r.problems.join("; "));
  revalidatePath(`/${entityRoute(entity)}/${id}`);
}

export async function setStatus(entity: string, id: string, status: string) {
  await store.setStatus(entity, id, status);
  revalidatePath(`/${entityRoute(entity)}`);
  revalidatePath(`/${entityRoute(entity)}/${id}`);
}

Field rendering — table-driven

One generic detail component introspects the row's columns and renders by type: text → input, longText → textarea, markdown → markdown editor (textarea + react-markdown preview side-by-side), select → dropdown, multiSelect → checkbox group, jsonb → JSON code editor, relation → searchable picker. Authored once, used 8 times.

11. Postgres-native wins (over the SQLite alternative)

FeatureHow we use it
jsonb columns multi_select fields (proofRefs, channels, keywords, hashtags…) — indexable by element via ? / ?| / ?& operators. Sets the table for E-028 register-level learning queries.
Native uuid + gen_random_uuid() Primary keys; no randomUUID() shim in app code.
Real timestamp + timestamptz createdAt / updatedAt with timezone awareness; clean range queries.
Native postgres.js client Pure JavaScript, zero native build. Kills the better-sqlite3 + CXXFLAGS friction we hit on the first run.
True transactions Multi-row creates (e.g. C1-fanout worker writing N Scripts for one Brief) wrap in store.transaction(async (tx) => …) — atomic.
Real concurrency Next.js server actions + the orchestrator can write concurrently without "database is locked".

13. Test plan

  • packages/store unit tests. CRUD round-trips for each of 8 entities; filter DSL → SQL; compliance scan refusal; fill-only-if-empty semantics. Tests run against the live engineerdad_test database — Postgres container must be up.
  • mcp-servers/store tool tests. Each of 5 tools handles happy path, compliance refusal, not-found.
  • Stage tests. Existing stage tests stay green after the tool/filter/shape flips.
  • Playwright E2E. Five specs cover list (seeded + empty), detail-edit-persist, status-flip-persists-in-list, markdown-side-by-side-live-preview. Each spec drops + recreates the engineerdad_test schema before running.
  • End-to-end validation — the acceptance test. Mint a fresh /loop-once run on the new infra; walk to HG1; approve 12 Briefs in the review UI; resume to HG2; approve Scripts; resume to HG3. No tool result exceeds ~10% of the wire cap.

14. Risks

Docker dependency. pnpm review and any orchestrator walk now require docker compose up -d first. If Docker Desktop isn't running, nothing works. Mitigation: a predev npm hook in apps/review-ui shells pnpm store:up + pg_isready before starting Next.js.
No data in git. This is the explicit user decision but it means: a fresh clone has no data. Cold-start is the only way in unless you carry a pg_dump separately. pnpm store:dump + pnpm store:restore scripts are wishlist material for later if backup pain emerges.
App scope creep. Review app is intentionally minimum-viable: 8 entities, one generic detail page, status flip, edit-and-save. Resist adding comments, mentions, history, search, multi-user. docs/war-room-wishlist.md is the gate for any addition.
Drizzle dialect specificity. pgTable column types (esp. jsonb, uuid, timestamptz) differ from sqliteTable. The schema in packages/store/src/schema.ts is Postgres-specific. If we ever revert to SQLite the schema needs porting.
Doctrine drift. ADR-005's "no MCP mesh" rule holds — mcp-servers/store does not call any other MCP server. The review app is an app consuming packages/store directly, not an MCP.

15. Open questions

  1. Postgres port collision. 5432 is the default. If you already run Postgres locally for other projects, this collides. Plan defaults to 5432 but documents an env override (STORE_DB_PORT → docker-compose.yml).
  2. Backup cadence. No backups at v1. pnpm store:dump is wishlist material — open when backup pain surfaces.
  3. Connection-pool sizing. postgres.js defaults to 10 connections. Sufficient for single-user. Document only — no tuning at v1.
  4. Migrations escape hatch. drizzle-kit push at v1. If schema evolution gets tricky (E-028 register-typed columns?) we revisit; switching to drizzle-kit generate + migrate is a one-task addition, not a rewrite.

16. What this unblocks / what it kills

Unblocks:

  • The wire-cap wall, permanently. The redesigned surface never crosses bulk content.
  • E-028 register-level learning — gains a typed register column + jsonb-indexed hook arrays.
  • Any future stage at any future scale — the store + thin-surface contract holds at v2/v3.
  • Production-shape concurrency. If the OS ever wants multi-window dev or remote access, Postgres is the correct substrate.

Kills:

  • ADR-012 (rich_text 2000-char chunking).
  • All 14 migrate-* live-workspace scripts.
  • Notion API rate-limit hesitations sprinkled across stage code.
  • The spill-file mechanism on the Scripts path. (Other MCP spills continue to exist where appropriate.)
  • The better-sqlite3 CXXFLAGS friction.

Defers:

  • The full war-room. The wishlist file remains the gate for any addition to the review UI.
  • Backup tooling. pnpm store:dump + restore when pain surfaces.
  • Per-feature migration files. drizzle-kit push at v1; migrate when needed.
