Meta-paid Manual Posting Pack + Webapp Migration

Design spec · 2026-05-30 · status: draft for review · branch feat/meta-paid-posting-pack

1. Problem & context

Meta requires business-entity verification before the app can be published — the real, external blocker to creating paid ads via the Marketing API (already recorded: ads_management / app-capability gap on act_943950444894075).

Today the distribute stage (packages/orchestrator/src/stages/distribute.ts) routes Meta-paid by spawning agent workers: D2a-setup (one worker creates the campaign + per-cell adsets) and D2b-route (a fan-out, one worker per row×channel) that call the meta-ads MCP create_campaign / create_adset / upload_* / create_ad_creative / create_ad tools. Those create calls cannot complete while the app is unverified.

The workaround: a manual posting pack. Instead of the agent fan-out hitting the Meta API, the stage renders the already-computed distribution plan into a human-readable spec (campaign / ad set / ad config + budget + creative asset) that the user types into Ads Manager by hand, then backfills the resulting Meta ad IDs. Assets are already in Cloudflare R2 (produced upstream); the pack references those URLs. The pack UI lives in the webapp.

What already exists (load-bearing — reuse, don't rebuild)

  • Full Meta-paid config is computed by plan-distribution.ts: campaign objective: OUTCOME_LEADS; per-cell adset daily_budget_cents = dailyBudgetMyr × cell.allocationPct, optimization_goal: LEAD_GENERATION, billing_event: IMPRESSIONS, bid_strategy: LOWEST_COST_WITHOUT_CAP, targeting = MY, age 25–55, locales en(6)/ms(41); per-ad primary_text / headline / description / cta (EN+BM) from the variant's metaSpec. The pack renders this; it invents no config.
  • Backfill field already exists: the planner's metaBackfill writes { adId } onto CreativeVariants as {en, ms}. Manual backfill writes the same field, so the analytics/insights join is unchanged.
  • Distributions is an append-only audit log with status routed | failed | skipped | dry-run and a skipReason. Manual mode records Meta-paid rows as skipped / "manual posting pack" — no new status enum.
  • Webapp: Next.js App Router under apps/webapp/src/app, Tailwind, a generic entity-review system (review/[entity]) plus bespoke pages (runs/). Server actions in lib/actions.ts, assets via /api/asset/....
  • Organic pack (B-005): scripts/build-posting-pack.mjs renders the approved IG organic queue to R2 HTML. To be migrated into the webapp.

2. Goals & non-goals

Goals

  • Flag-controlled Meta-paid mode: api (existing spawn-and-create path, for when verification clears) vs manual (render a posting pack, no Meta API write). Default manual.
  • A distribute MCP server exposing posting-pack render + backfill (per user direction), thin over a pure render lib that wraps the existing planner.
  • A dedicated per-run webapp page (/posting-pack/[runId]) rendering the pack as a reference table (Campaign → Ad Sets → Ads → Creative) with a per-ad backfill form.
  • Remove HG4 — the human checkpoint moves to the webapp pack + backfill flow.
  • Migrate the organic posting pack into the webapp; retire the R2-HTML script.

Non-goals

  • No automated Meta paid posting (blocked externally). The api path stays as-is, dormant behind the flag.
  • No change to where assets are produced/hosted — they stay in R2.
  • No new campaign/adset/ad config logic — reuse planMetaPaid.
  • No auto-activation, ever (ADR-015). The pack never sets ACTIVE; activation is the human's Ads Manager action.

3. Architecture overview

distribute stage (per META_PAID_MODE)

  api mode (existing, dormant):
    D1-query → D2a-setup (spawn: create campaign+adsets) → D2b-route (fanout: upload/creative/ad)
             → D3a-confirm → D3b-summary → D4-gate(HG4)   ← REMOVED

  manual mode (new, default):
    D1-query → [Meta-paid EXCLUDED from D2a/D2b spawn fan-out]
             → D3b-summary records Meta-paid rows: status=skipped, skipReason="manual posting pack"
             → (no HG4)
    YouTube / Article / Meta-organic units still route normally.

  Pack = a VIEW, derived on demand:
    distribute MCP get_posting_pack(runId)
      → renderPostingPack() wraps planMetaPaid(runId, approvedVariants, cells, dailyBudgetMyr)
      → PostingPackSpec (grouped campaign/adsets/ads + R2 asset URLs)
    webapp /posting-pack/[runId] renders the table
      → user posts in Ads Manager by hand
      → backfill form → distribute MCP backfill_meta_ids()
        → store.update CreativeVariants.adId = {en, ms}  (+ campaigns row for campaign/adset ids)
Key simplification vs first draft: the pack is not persisted state — it is a pure function of the run's approved variants + experiment cells, recomputed on read. Single source of truth = the existing planner. No new store entity, no new status, no payload duplication.

4. Components

4a. packages/distribute (new, small lib)

  • renderPostingPack(runId, variants, cells, dailyBudgetMyr): PostingPackSpec — calls the existing planMetaPaid (imported from @engineerdad/orchestrator or its planner module) and projects the setup + rowPlans ToolStep args into a flat, display-grouped shape (campaign, adsets[], ads[] with EN/BM copy, CTA, link, asset URLs). Pure; no I/O.
  • types.ts — PostingPackSpec (below), backfill input shape.
  • Unit tests: grouping correctness, EN+BM ad pairs present, budget cents surfaced per adset, asset URLs scene-ordered, empty-when-no-Meta-variants.

PostingPackSpec (display shape — derived, not stored)

type PostingPackSpec = {
  runId: string;
  campaign: { name: string; objective: "OUTCOME_LEADS"; special_ad_categories: [] };
  adsets: Array<{
    cellId: string; name: string;
    daily_budget_cents: number; daily_budget_myr: number;  // both, for display
    optimization_goal: string; billing_event: string; bid_strategy: string;
    targeting: { countries: string[]; age_min: number; age_max: number; locales: number[] };
  }>;
  ads: Array<{
    variantId: string; rowId: string; title: string; cellId: string;
    adsetName: string;                       // which adset to attach under
    asset: { urls: string[]; format: string; aspect: string | null };
    en: { primary_text: string; headline: string; description: string };
    bm: { primary_text: string; headline: string; description: string };
    cta_type: string; link_url: string;      // link_url from env LANDING_URL default
    backfill: { adIdEn: string | null; adIdMs: string | null; done: boolean };  // from CreativeVariants.adId
  }>;
};

4b. mcp-servers/distribute (new MCP server)

Thin adapter over the lib + store, structured like mcp-servers/meta-ads (McpServer + stdio + zod). Tools:

  • get_posting_pack({ runId }) — query approved CreativeVariants + the run's Experiments cells, call renderPostingPack, return the spec (backfill state read from CreativeVariants.adId).
  • backfill_meta_ids({ runId, variantId, adIdEn, adIdMs, campaignId?, adsetId? }) — store.update("CreativeVariants", rowId, { adId: { en, ms } }); optionally upsert the campaigns row with metaCampaignId. Idempotent.
  • list_posting_packs() — runs that have approved Meta-paid variants (for a webapp index, optional).

Register in .mcp.json (and .claude/settings.json allowlist if present). Restart Claude Code after registration (CLAUDE.md). Build is sequential (pnpm -r build).

4c. Orchestrator changes (stages/distribute.ts, registry.ts)

  • Flag META_PAID_MODE = "api" | "manual" (env, default manual). A tiny reader (e.g. in run-args.ts or a config.ts), testable.
  • d2aSetup / d2bRoute build(): in manual mode, treat Meta-paid as filtered out — the same mechanism as the existing channelFilter. Setup emits no campaign/adset spawn; the fan-out emits no Meta-paid units. YouTube/Article/Meta-organic unaffected.
  • d3bSummary: in manual mode, still emit Meta-paid audit rows but as status: "skipped", skipReason: "manual posting pack" (reuses plannerSkipReason path; add the reason). verifyDistribute already treats planner-skips as non-failures.
  • Remove HG4: drop d4Gate from distributeStage.steps. The run reaches done after D3b-summary (or proceeds to /reflect). See §5.

4d. Webapp — Meta-paid posting-pack page

  • New route apps/webapp/src/app/posting-pack/[runId]/page.tsx (server component). Fetch via a lib/ helper that calls the distribute MCP get_posting_pack (mirroring how lib/orchestrator.ts wraps MCP/store reads).

5. HG4 removal — implications

HG4 is doctrinally the spend gate (ADR-015, established 2026-05-16). Its check (list_ads → assert none ACTIVE) gates ALL distribution channels, not just Meta-paid. Removing it un-gates the dispatch of YouTube (unlisted) and Meta-organic (scheduled publish) as well.

Why removal is acceptable under the user's instruction:

  • Meta-paid in manual mode does no API spend — there is nothing to gate; the spend decision physically happens in the user's Ads Manager session when they create + (later) un-pause the ads. The gate's purpose moves there intact.
  • No-auto-activate invariant preserved: api-mode creates remain PAUSED; YouTube remains unlisted. Nothing the OS does goes ACTIVE/public by itself.
  • Residual exposure to flag: Meta-organic posts are scheduled publishes — removing HG4 means they dispatch without a pre-publish click. Today's human review of organic content still happens upstream (HG1/HG2/HG3 + organic approval), and the new /posting-pack/organic page remains the working surface. Acceptable, but called out.

Alternative (not chosen): retain a gate only for channels that still write externally (YouTube/organic) and drop it for Meta-paid manual — needs per-channel gating the registry doesn't model. Defaulting to full HG4 removal per instruction.

Docs/doctrine to update: registry.ts comment, ARCHITECTURE.md (loop diagram + gate table + §human gates), CLAUDE.md (loop line), .claude/commands/loop.md, distribute.md, experiment.md (they say "Stops at HUMAN GATE 4"), and an amendment to ADR-015 (+ note in ADR-022) recording that the spend gate is satisfied by manual mode + the webapp pack. ADR-015 is load-bearing — amend, don't silently contradict.

6. Backfill user journey

  1. Run reaches distribute in manual mode; Meta-paid is excluded from the agent fan-out; audit rows = skipped ("manual posting pack"). No HG4.
  2. User opens /posting-pack/[runId] next to Ads Manager.
  3. Create the campaign (objective shown), then each ad set (budget/optimization/targeting shown), then each ad (EN or BM copy + CTA + link, download the R2 asset). Leave everything PAUSED.
  4. Paste each ad's Meta ad ID (EN/BM) into the backfill form → writes CreativeVariants.adId; row shows ✓.
  5. Downstream analytics/insights join on the backfilled adId exactly as in api mode.

9. Risks & open questions

  • HG4 removal scope — confirmed: un-gates YouTube/organic dispatch too (safe states; organic scheduled-publish caveat in §5).
  • Distribute MCP vs in-process — distribute runs in-process; the MCP is an additional read/backfill surface (per user direction). The shared render lib avoids divergence. The lib importing the orchestrator planner creates a distribute → orchestrator package dep; if that's circular/awkward, move the Meta-paid planner helpers into packages/distribute and have the orchestrator import them (cleaner long-term). Decision for the plan phase.
  • campaign/adset id capture — adId is the must-have backfill; campaign/adset IDs are optional metadata (stored on campaigns). Confirm that's enough for later insights, or wire adset id onto a per-variant field if needed.
  • dailyBudgetMyr source — the planner reads it from the Brain memo experimentParams.dailyBudgetMyr / run params; the pack surfaces whatever the run carried. No change.
