ADR-024 — The orchestrator stages worker input; spawnPrompts carry refs

The third and final cap surface. The orchestrator persists per-unit worker input to orchestrator.step_results at fanout-construction time and embeds only the ref in each spawnPrompt. Workers call read_step_result on entry to fetch their bulk input. ADR-022's claim-check mirror, applied to the input direction.

1. Context — a third cap, the mirror surface

ADR-022 / E-031 fixed the worker → conductor cap (a worker's final output too large for the conductor's inline channel). ADR-023 / E-033... no — ADR-023 fixed the orchestrator → conductor cap on the output direction (write-step calls transmitted back through the conductor). Two boundaries down, one to go.

On 2026-05-25, the ADR-023 Phase H.4 cold-start walk on run_1779670351 drove tracking → analytics → synthesize → brief → content through HG2 cleanly — every write step ran inside plan(), every worker output flowed via claim-check refs. At P1-fanout, the orchestrator built spawnPrompts for the 12 creative-director workers. Each spawnPrompt embedded the brief's full hook bank (~30 hooks × bilingual EN/MS text) — about 7 KB of per-unit reference data. Twelve units × 7 KB ≈ 84 KB. The conductor received: 

Error: result (83,795 characters across 46 lines) exceeds maximum allowed tokens.
Tool: mcp__orchestrator__plan

The cap that bit is the same cap E-031 and ADR-023 already addressed. The boundary is new: it's orchestrator → conductor on the input direction. Each fanout unit's spawnPrompt carries the bulk data the spawned worker will operate on. The orchestrator was the source of bulk; the conductor was the cap-sensitive carrier; the worker was the consumer. Three roles, mirror-image of ADR-022, same fix shape.

The deeper pattern. ADR-022, ADR-023, ADR-024 are three local fixes for the same global rule: bulk data does not pass through the conductor's tool channel. The conductor carries refs; bulk lives in orchestrator.step_results. ADR-024 closes the last open direction (orchestrator → worker, via the conductor's transmission).

2. Boundary inventory — three transmission directions

Every bulk-data hop in a closed-loop run sits at one of three boundaries between the orchestrator, the conductor, and a spawned worker. The conductor is the cap-sensitive transmission stop in all three.

# Direction Origin Conductor's role Status
1 Worker → conductor (output) Worker's final message after doing its work Pass output verbatim to verify / advance Fixed by ADR-022. Worker writes its output to step_results; emits {stepResultId}; conductor carries the ref; MCP server dereferences server-side.
2 Orchestrator → conductor (output) Engine's step.calls[] in a write step Execute each call against named MCP tools Fixed by ADR-023. Orchestrator self-executes write steps via library imports of packages/<name>; conductor sees only spawn / fanout / terminal-gate / done / halt.
3 Orchestrator → conductor (input) Per-unit spawnPrompt embedding bulk worker-input data Dispatch Task with the spawnPrompt verbatim Open — this ADR. Orchestrator persists bulk worker-input to step_results at fanout-construction time; spawnPrompt carries the ref; worker calls read_step_result on entry.

Direction #1 is worker-emitted output. Direction #2 is orchestrator-emitted execution work. Direction #3 is orchestrator-emitted dispatch context. The three together exhaust the bulk hops in a fanout cycle.

3. Industry precedent — staged input via shared substrate

The pattern of "lead agent stages context before dispatching subagents" is settled in 2026's production multi-agent systems. The pattern always uses a shared persistence substrate that both writer (lead) and reader (subagent) can name.

Framework Substrate Mechanism
Anthropic multi-agent research system Memory (persistent store) "The LeadResearcher begins by thinking through the approach and saving its plan to Memory to persist the context, since if the context window exceeds 200,000 tokens it will be truncated." Plan is staged before subagents are dispatched. Subagents have their own context windows and can re-read the plan as needed.
Google ADK artifact_service (named, versioned binary store) Agents call save_artifact(filename, data); downstream agents call load_artifact(filename). First-class staging — saving an artifact is an explicit, async, audited operation.
Claude Code Task tool (this OS's runtime) (N/A — by-convention) Docs warn directly: "every token in the prompt is paid for whether the subagent references it or not... scope the prompt to the task and keep the output contract tight." The harness itself bills inline bulk. The recommended pattern is thin prompts; bulk fetched on demand.
LangGraph Send() State store (per-graph) Routing functions can be async and write to state. The Send() per-target payload is technically unbounded but the convention is to pass small dispatch info and have nodes read shared state via the state store.
Apache Airflow XCom (orchestration analog) XCom backend (default: small-message DB; custom: S3 / blob) Large payloads use a custom XCom backend pointing at external storage. Same shape: small ref via XCom, bulk via the backend.
Consensus. Production multi-agent systems (Anthropic, Google ADK) and orchestration analogs (Airflow custom XCom) all converge on the same primitive: a shared persistence substrate that lead and worker can both name. Inline staging is the anti-pattern; ref-passing is the doctrine.

We already have the substrate. orchestrator.step_results shipped with ADR-022. The ref shape (sr_-prefixed ULID) is established. The read tool (mcp__orchestrator__read_step_result) exists. ADR-024 is the doctrine that says: use it from the input side too.

4. The doctrine

When an orchestrator stage step constructs a fanout (or a spawn) whose units require bulk per-unit reference data, the stage's build():

  1. Writes each unit's bulk input to orchestrator.step_results via writeStepResult, with a deterministic (runId, stepId, unitIndex) identity that makes re-planning idempotent.
  2. Embeds the returned stepResultId in the unit's spawnPrompt, alongside a thin task contract (objective, output format, task boundaries — per Anthropic's "four required ingredients").
  3. Includes a small set of unique-per-unit fields directly in the prompt (entity ids, run id) so the worker has handles to fetch any other context it needs.

Worker contract (agent prompts):

  1. First action on entry: call mcp__orchestrator__read_step_result({ stepResultId }) to fetch the bulk input the orchestrator staged.
  2. Operate on the resolved payload. Never assume the prompt itself carries the full reference data.
  3. Persist final output to step_results via write_step_result (per ADR-022) and return only the stepResultId.

What build() is allowed to do (the new discipline)

Making build() async unlocks I/O at step-construction time. The doctrine constrains what's allowed:

AllowedNot allowed
  • Write to orchestrator.step_results (our own substrate)
  • Read from orchestrator.step_results for prior staged inputs
  • Compute / shape the prompt body (pure logic)
  • Call external MCPs (would violate ADR-005 no-MCP-mesh)
  • Write entity rows (Briefs, Scripts, etc.) — that's the compliance scanner's lane, owned by write steps
  • Call external APIs (Meta, OpenAI, etc.) — those are step-execution work, not step-construction
  • Cause observable run-state changes beyond the substrate write

The single allowed write is auditable (every entry in step_results carries run_id, step_id, unit_index, size_bytes, created_at), capped in size by application convention, and addressable via the existing read_step_result tool. The discipline keeps build() from becoming a place where arbitrary side effects accumulate while opening exactly enough room for the staging pattern.

5. The mechanism — async build and the Promise ripple

Today's StepSpec.build is sync: 

// packages/orchestrator/src/types.ts (today)
export interface StepSpec {
  id: string;
  kind: "spawn" | "fanout" | "write" | "gate";
  build: (run: RunState) => Step;
  verify?: (run: RunState, result: unknown) => VerifyResult;
}

After ADR-024: 

// packages/orchestrator/src/types.ts (after)
export interface StepSpec {
  id: string;
  kind: "spawn" | "fanout" | "write" | "gate";
  build: (run: RunState, ctx: BuildContext) => Promise<Step> | Step;
  verify?: (run: RunState, result: unknown) => VerifyResult;
}

export interface BuildContext {
  /** Persist a per-unit input payload; returns the ref to embed in the prompt. */
  stageInput(unitIndex: number | null, payload: unknown): Promise<string>;
}

Three changes:

  • Return type widened to Promise<Step> | Step. Stages that don't need to stage anything keep sync semantics; stages that do return a promise. The engine awaits regardless (await Promise.resolve(syncReturn) is a no-op).
  • New ctx parameter (BuildContext) carries the only piece of effectful capability build() needs: stageInput(unitIndex, payload). The implementation wraps writeStepResult with the current run/step ids closed over. Constraining the capability surface to one named method is what makes the discipline enforceable.
  • Engine call sites in plan() become await spec.build(run, ctx). The orchestrator MCP's runEagerLoop is already async (since Phase B of ADR-023); no signature change there.

Idempotency — re-planning safety

A pure build() is safe to call any number of times for the same state. ADR-024's effectful build() writes to Postgres — running it twice for the same fanout would create duplicate rows. Two mechanisms restore idempotency:

  1. Deterministic identity. stageInput derives the row's primary key from (runId, stepId, unitIndex) rather than a fresh ULID. A re-plan on the same fanout finds the existing row and skips the insert (Postgres ON CONFLICT DO NOTHING). The returned stepResultId is the existing row's id.
  2. Step-completion shortcut. plan() already skips build() for steps marked done in run_steps. A re-plan after the fanout's advance never re-enters build() for that step at all. The idempotency concern only applies during the brief window between build() writing refs and the conductor dispatching workers.

Together: re-planning during the dispatch window finds the existing rows and returns the same refs; re-planning after advance never re-enters build. build() is at-most-once effectively, even though its body contains writes.

The conductor's day (post-ADR-024)

function conductorLoop() {
  let runId, step;
  while (true) {
    ({ runId, step } = mcp__orchestrator__plan({ runId }));
    // step is now <= 2-3 KB even for a 12-unit fanout:
    // every spawnPrompt is "do X, your input ref is sr_01H..."
    // not "do X, here are 7 KB of hooks inline"

    if (step.kind === "spawn")  dispatchTask(step);     // unchanged
    if (step.kind === "fanout") dispatchTasks(step);    // unchanged shape, smaller prompts
    if (step.kind === "gate")   { print(step.message); return; }
    if (step.kind === "done")   { print(step.message); return; }
    if (step.kind === "halt")   { print(step.reason);  return; }

    // verify + advance per ADR-022
  }
}

The conductor's contract is unchanged. The prompts are smaller. The workers do an extra read_step_result on entry.

7. Engine API + MCP envelope shapes

StepSpec.build

build: (run: RunState, ctx: BuildContext) => Promise<Step> | Step

BuildContext

interface BuildContext {
  /**
   * Persist a payload to orchestrator.step_results under the deterministic
   * key (runId, stepId, unitIndex). Re-calls with the same key return the
   * existing row's id (ON CONFLICT DO NOTHING). Returns the stepResultId
   * string to embed in a spawnPrompt.
   */
  stageInput(unitIndex: number | null, payload: unknown): Promise<string>;
}

engine.plan

// Today (sync, would-be name)
function plan(input, registry): { runId, step }

// After ADR-024
async function plan(input, registry): Promise<{ runId, step }>

writeStepResult — new optional idempotencyKey

function writeStepResult(args: {
  runId: string;
  stepId: string;
  unitIndex?: number | null;
  payload: unknown;
  /** When set, derives the row id from this key + run/step; re-calls
   *  return the existing id rather than creating a new row. */
  idempotencyKey?: string;
}): Promise<string>

The idempotencyKey defaults to "${stepId}:${unitIndex ?? -1}" when BuildContext.stageInput calls in. Other callers (worker output writes per ADR-022) keep the existing fresh-ULID behaviour.

No new MCP tools, no new step kind, no schema change

ADR-024 reuses the existing substrate. step_results already has the columns it needs. read_step_result already does the dereference. The only schema delta is an optional unique index on (run_id, step_id, unit_index, payload_kind) to make the idempotency mechanism enforceable at the database layer (belt-and-suspenders with the application-level conflict-handler).

8. Worker prompt protocol — the new entry contract

Workers that consume staged input get a one-line addition to their agent prompt frontmatter (.claude/agents/<agent>.md) and a mandatory first action.

Today (creative-director Single-Script mode)

"Run ${runId}: you are creative-director in Single-Script worker mode.
Your FIRST action: call mcp__store__get({ entity: 'Scripts', id: '...' })
to fetch your Script.
...
HOOK BANK for brief abc123:
[{en: '...', ms: '...', register: 'fear'}, ... × 30 hooks]"

After ADR-024

"Run ${runId}: you are creative-director in Single-Script worker mode.

Your FIRST action: call mcp__orchestrator__read_step_result({
  stepResultId: 'sr_01HZ...' })
to fetch your staged input. It contains: scriptId, briefId, hookBank.

Then call mcp__store__get for the Script and Brief, and operate on
exactly one Script.

Produce 4 distinct CreativeUnits (Reel, Feed, YT-Long, Carousel),
rotating 4 distinct hooks from the staged hook bank. Return
{scriptId, creatives} as your final JSON."

Length: ~280 bytes vs ~7,200 bytes per unit. Twelve-unit fanout drops from 84 KB to ~3.4 KB.

9. Worked example — P1-fanout, before vs after

Before (current behaviour, what blew the cap on 2026-05-25)

// packages/orchestrator/src/stages/produce.ts (today)
const p1Fanout: StepSpec = {
  id: "P1-fanout",
  kind: "fanout",
  build: (run): Step => {
    const scripts = approvedScriptsFor(run);
    return {
      kind: "fanout",
      stepId: "P1-fanout",
      worker: "creative-director",
      units: scripts.map((script) => {
        const hookBank = hookBankForBrief(run, script.briefId); // ~7 KB
        return {
          spawnPrompt: [
            `Run ${run.runId}: you are creative-director...`,
            `...fetch Script id=${script.id}, Brief id=${script.briefId}...`,
            `HOOK BANK for brief ${script.briefId}:`,
            JSON.stringify(hookBank), // ← embeds 7 KB inline
          ].join("\n"),
        };
      }),
    };
  },
  // ...
};

Envelope: 12 × ~7 KB = 83,795 bytes in the plan() response. Hits the harness cap.

After (ADR-024)

// packages/orchestrator/src/stages/produce.ts (after)
const p1Fanout: StepSpec = {
  id: "P1-fanout",
  kind: "fanout",
  build: async (run, ctx): Promise<Step> => {
    const scripts = approvedScriptsFor(run);
    const units = await Promise.all(
      scripts.map(async (script, i) => {
        const hookBank = hookBankForBrief(run, script.briefId);
        const inputRef = await ctx.stageInput(i, {
          scriptId: script.id,
          briefId: script.briefId,
          hookBank,
        });
        return {
          spawnPrompt: [
            `Run ${run.runId}: you are creative-director...`,
            `Your FIRST action: call mcp__orchestrator__read_step_result({`,
            `  stepResultId: "${inputRef}"`,
            `}) to fetch your staged input.`,
            `...produce 4 CreativeUnits, return {scriptId, creatives}...`,
          ].join("\n"),
        };
      }),
    );
    return { kind: "fanout", stepId: "P1-fanout", worker: "creative-director", units };
  },
  // ...
};

Envelope: 12 × ~280 bytes = ~3,400 bytes in the plan() response. Twenty-five× reduction; well under the cap with headroom.

10. Substrate — what reuses, what extends

Reused without change

  • orchestrator.step_results table (Postgres, JSONB payload, ADR-022 schema). Worker input rows live alongside worker output rows; the row's role is implicit from who wrote it (build vs worker).
  • mcp__orchestrator__read_step_result tool. Workers call it the same way they would for any ref.
  • sr_-prefixed ULID format. No change.
  • The resolveRefs helper at the MCP verify/advance boundary — unaffected, since spawn-prompt refs aren't passed back through verify/advance (they're consumed by the worker).

Extended (minor)

  • writeStepResult gains an optional idempotencyKey param. Default behaviour (fresh ULID) preserved for non-build callers.
  • Optional new column payload_kind (text, nullable) on step_results to tag rows as "input" vs "output" for audit clarity. Not load-bearing — derivable from "who wrote it" — but useful for the /status dashboard and for future TTL policy (input rows can be GC'd as soon as the fanout's advance lands; output rows have a longer retention).
  • Unique index on (run_id, step_id, unit_index, payload_kind) to enforce idempotency at the DB layer.

New

  • BuildContext interface in packages/orchestrator/src/types.ts.
  • createBuildContext(runId, stepId) factory in packages/orchestrator/src/exec.ts (close-over writeStepResult with the run/step ids set).
  • Engine's plan awaits build(); signature change ripples through packages/orchestrator + the orchestrator MCP server's runEagerLoop (already async).

13. Out of scope

  • Promoting build() to do general-purpose I/O. The discipline (§4) limits build to orchestrator.step_results writes. Other I/O remains in step execution (write steps, worker tool calls). ADR-024 widens the type signature but not the semantics envelope.
  • TTL / cleanup for input-kind step_results rows. Filed as a follow-up alongside ADR-022's TTL task. Input rows are GC-able sooner (immediately after the fanout's advance), but cleanup design is independent.
  • Cross-run input refs. A future stage might want to stage data once and reuse it across runs. Not in scope; would need its own ADR.
  • Worker → worker direct refs. Workers today never read each other's outputs directly. They could, via read_step_result + a shared ref-id channel, but that pattern is unmotivated and out of scope.
  • Encryption-at-rest for staged input. The substrate is the same Postgres instance as entity rows; same encryption posture; not in scope to harden separately here.

15. Risks & mitigations

RiskMitigation
build() async-ness becomes a slippery slope; future contributors stage Meta API calls or corpus reads in build() Discipline section codified in ADR-024 with explicit allow/deny list. BuildContext exposes only stageInput — no general-purpose tool surface. Adding a new capability requires an ADR amendment.
Idempotency key collides across stages with the same step id (e.g., two stages both have a "C1-fanout"?) run_steps.step_id is already unique-per-run by registry construction. The idempotency key composes runId + stepId + unitIndex; collision would require two registry entries with the same step id in the same stage, which the registry-validation tests already forbid.
Re-plan during the dispatch window dispatches a different set of refs than the conductor expects Idempotency guarantees: a re-plan returns the same refs for the same (runId, stepId, unitIndex). Conductor compares the unit count it received to the unit count it dispatches; mismatch is a halt (no silent partial dispatch).
Engine signature change breaks downstream consumers Phase C lands the signature change once across the workspace: packages/orchestrator + all stage definitions + tests. Build-and-typecheck must be green before Phase D begins. The Promise<Step> | Step union allows sync-return stages to stay sync — the engine awaits unconditionally.
Worker forgets to call read_step_result first and acts on the thin prompt alone Worker agent prompts (.claude/agents/creative-director.md) carry a hard rule with a sample first action; sync:agents keeps the fragment in lockstep. Workers that omit the read produce zero output and fail their verifier (no hooks → empty creative units → verify-produce rejects).
Latency: build() now does writes, slowing every plan() call writeStepResult is a single Postgres insert per unit. P1 (12 units) adds ~12 inserts × ~1 ms = ~12 ms to a plan() that already takes hundreds of ms across the eager loop. Imperceptible.
Input-row pollution of step_results table Optional payload_kind tag separates input from output rows in queries. Future TTL job can sweep input rows aggressively (they're disposable post-advance) while keeping output rows for the run's full lifetime.
