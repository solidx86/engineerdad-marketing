Distribute stage — ADR-22/23/24 compliance + Distributions entity

Date: 2026-05-27 Branch: feat/brain-experiment-params accepted

1. Context

The distribute stage has three coupled problems surfaced while walking run_1779779169 through HG4:

  1. Bug. plan() crashes with Cannot read properties of undefined (reading 'includes') inside planDistribution(). Root cause: D1-query calls mcp__store__query without a fields: projection, so the store returns only {id, title} per row. d2Route.build casts those skinny rows to DistVariant[] and the planner immediately dereferences v.channels.includes(META) on an undefined field.
  2. ADR-024 non-compliance. d2Route is a kind: "spawn" step that embeds the entire DistributionPlan JSON directly into its spawnPrompt via distributePrompt(plan). With the bug fixed, that plan acquires real content (campaign + N adsets + per-row tool sequences + backfills) and the spawnPrompt approaches the conductor's 50 KB cap. ADR-024 mandates per-unit bulk worker-input go through ctx.stageInput() with the prompt carrying only an sr_… ref.
  3. No operational audit trail. Today's distribute stage produces no externally visible record of what was attempted, what landed, what was skipped, or why. D3-confirm re-queries the store and emits a pass/fail verdict against verifyDistribute, but the granular outcome is lost. The webapp has review pages for Briefs/Scripts/CreativeVariants but nothing for distribution attempts.

This spec ships the three together because they're mechanically interlocked: fixing the bug exposes the ADR-024 gap; the per-unit worker contract introduced for ADR-024 is the natural write point for the Distributions audit log; the audit log's row shape constrains what each step's worker reports.

2. Goals & non-goals

Goals

  • The distribute stage runs to completion on real store rows without dereference errors.
  • D2-route conforms to ADR-024: spawnPrompts carry refs, not bulk; build() uses ctx.stageInput() for per-unit input.
  • Every distribution attempt and its ground-truth verification produces a queryable row in a new Distributions entity, surfaced in the review UI.
  • D3-confirm remains the trust-of-record: a D2b worker's self-report is the per-attempt row, but D3's ground-truth re-query writes the row that drives HG4.
  • The IG manual-posting boundary (B-005) is preserved unchanged.

Non-goals

  • Automating IG organic distribution. Stays manual via /posting-pack until E-024 (always-on scheduler) lands.
  • Retrying failed distribution attempts automatically. The Distributions log makes retries trivially auditable but the retry mechanism itself is a follow-up (handled by re-running /distribute; D2a/D2b idempotency keys already guard against duplicates).
  • Cross-run analytics on Distributions data (rollups, dashboards). The list page in §5 is per-run; aggregation is its own concern.
  • Migrating historical run data into the new Distributions table. The entity is forward-looking.

3. Approach

Three coordinated changes, in the order they get implemented:

  1. D1-query field projection & row → DistVariant projection — add explicit fields: to D1's three queries; add a projectVariant(row, cells) / projectArticle(row) layer that translates raw store columns into the DistVariant / DistArticle shapes the planner expects.
  2. Decompose planDistribution into per-channel emitters — pull planMetaPaidSetup out as a standalone function (used by D2a-setup) and planMetaPaidRows / planYouTube / planArticles / planOrganic as the per-channel row producers (used by D2b-route). The existing planDistribution entry point stays as a thin composer over them for tests / future callers.
  3. Split D2-route into D2a-setup (spawn) + D2b-route (fanout) — D2a creates the Meta campaign + N adsets via one worker and stages the captured IDs. D2b fans out one unit per rowPlan, each unit's input staged via ctx.stageInput(i, …), each spawnPrompt carrying only the ref.
  4. New Distributions entity — append-only event log; one row per D2b unit execution (worker self-report) plus one summary row per (variant×channel) and per article from D3-confirm (ground truth).
  5. Review page — /review/distributions?runId=… list view, sort by createdAt desc, filter by channel / status / author step.

The projection is idempotent: if a row already carries a pre-projected field (e.g. test fixtures with rowId set), it passes through; otherwise the helper derives it from the raw column. This keeps the existing distribute tests passing without touching their fixtures.

D3-confirm gets the same field lists.

5.2 plan-distribution.ts — per-channel decomposition

packages/orchestrator/src/distribute/plan-distribution.ts currently exposes one entry point planDistribution() that internally calls planMetaPaid + planYouTube + planArticles + planOrganic and merges their PlanParts. Each of those internal functions today does both setup and per-row work in one pass (e.g. planMetaPaid emits the campaign + adsets + per-row rowPlans + backfills).

We split planMetaPaid into two:

  • planMetaPaidSetup(runId, variants, cells, dailyBudgetMyr) → PlanPart — emits the campaign + adsets only. Used by D2a.
  • planMetaPaidRows(variants, cells) → PlanPart — emits per-row tool sequences (upload, creative×2, ad×2) and per-row backfills. Used by D2b.

The other three (planYouTube, planArticles, planOrganic) have no shared setup phase; they emit only rowPlans + backfills. They stay single-function and are used by D2b directly.

planDistribution() stays exported as a thin composer over the per-channel functions. Existing callers (tests, dry-run paths) continue to work. The function is no longer called from within d2Route.build directly — D2a and D2b call the decomposed pieces instead.

5.3 D2a-setup (spawn)

New step. Replaces the campaign + adsets portion of the old D2-route.

const d2aSetup: StepSpec = {
  id: "D2a-setup",
  kind: "spawn",
  build: async (run, ctx): Promise<Step> => {
    const d1 = stepResult<unknown[]>(run, "D1-query") ?? [];
    const cells = cellsOf(rowsOf(d1[2])[0]);
    const variants = projectVariants(rowsOf(d1[0]), cells);
    const params = run.params as DistributeParams;
    const setupPart = planMetaPaidSetup(run.runId, variants, cells, params.dailyBudgetMyr ?? 0);
    const ref = await ctx.stageInput(null, {
      setupSteps: setupPart.setup,
      dryRun: params.dryRun ?? false,
    });
    return {
      kind: "spawn",
      stepId: "D2a-setup",
      agent: "general-purpose",
      spawnPrompt: setupPromptFor(ref),
    };
  },
  verify: (_run, result): VerifyResult => {
    // Worker reports {campaignId, adsetByCellId}. Non-null pass; HG4
    // ground-truth check on Meta entities is the real backstop.
    if (result === null || typeof result !== "object") return { ok: false, problems: ["D2a: no setup result"] };
    return { ok: true, problems: [] };
  },
};

Worker prompt (setupPromptFor(ref)):

You are the Meta setup worker for the distribute stage.

Your FIRST action: call
  mcp__orchestrator__read_step_result({ stepResultId: "<ref>" })
to fetch { setupSteps, dryRun }.

Procedure:
1. Run every step in `setupSteps` in order. Each step has `captures` — remember
   the result keyed by that label.
2. Honor `needs`: substitute the captured label for the placeholder before
   invoking the step's tool.
3. If `dryRun` is true, walk the steps and report what you would have called,
   without executing.

Return JSON: { campaignId, adsetByCellId: { cellId: adsetId, ... } }.
On empty setupSteps (no routable Meta variants), return { campaignId: null, adsetByCellId: {} }.

Empty-setup case: if setupPart.setup.length === 0 (no routable Meta variants — e.g. experiment declined), the worker receives an empty list and returns the null shape. d2aSetup still dispatches; the worker no-ops in ~1 second. Alternative considered: skip D2a entirely when empty. Rejected because the engine doesn't natively support conditional skipping at the StepSpec level, and a 1-second no-op is cheaper than the special-case logic.

5.4 D2b-route (fanout)

New step. Replaces the per-row work portion of the old D2-route.

const d2bRoute: StepSpec = {
  id: "D2b-route",
  kind: "fanout",
  build: async (run, ctx): Promise<Step> => {
    const d1 = stepResult<unknown[]>(run, "D1-query") ?? [];
    const cells = cellsOf(rowsOf(d1[2])[0]);
    const variants = projectVariants(rowsOf(d1[0]), cells);
    const articles = projectArticles(rowsOf(d1[1]));
    const setup = stepResult<SetupResult>(run, "D2a-setup") ?? emptySetup();
    const params = run.params as DistributeParams;
    const channels = params.channelFilter ?? null;

    // Per-channel row emitters → flat array of (channel, rowPlan, backfill)
    const rowUnits = [
      ...(channelOk("Meta-paid", channels)    ? attachBackfills(planMetaPaidRows(variants, cells))    : []),
      ...(channelOk("YouTube",   channels)    ? attachBackfills(planYouTube(variants))                 : []),
      ...(channelOk("Article",   channels)    ? attachBackfills(planArticles(articles))                : []),
      ...(channelOk("Meta-organic", channels) ? attachBackfills(planOrganic(variants))                 : []),
    ];

    // Substitute setup IDs into row plans (rowPlans that need "adset:cellX"
    // get setup.adsetByCellId[cellX] inlined as a needs-resolved placeholder).
    const resolvedUnits = rowUnits.map((u) => resolveSetupRefs(u, setup));

    const units = await Promise.all(
      resolvedUnits.map(async (u, i) => {
        const ref = await ctx.stageInput(i, {
          channel: u.channel,
          rowPlan: u.rowPlan,
          backfill: u.backfill,
          dryRun: params.dryRun ?? false,
          runId: run.runId,
        });
        return { spawnPrompt: routePromptFor(ref, u.channel) };
      }),
    );

    return { kind: "fanout", stepId: "D2b-route", worker: "general-purpose", units };
  },
  verify: (_run, result): VerifyResult => {
    // Per-unit lenient: each unit returned its worker result. D3-confirm is
    // the trust-of-record. Reject only on null / non-array.
    if (!Array.isArray(result)) return { ok: false, problems: ["D2b: fanout result not an array"] };
    return { ok: true, problems: [] };
  },
};

Worker prompt (routePromptFor(ref, channel)) — uniform across channels:

You are a distribute-route worker for the <channel> channel.

Your FIRST action: call
  mcp__orchestrator__read_step_result({ stepResultId: "<ref>" })
to fetch { channel, rowPlan, backfill, dryRun, runId }.

Procedure:
1. If dryRun, write a Distributions row { status: "dry-run", ... } and return
   what you would have called. Skip steps 2–4.
2. Pre-check: if the variant/article already has its idempotency-key field
   populated (adId for Meta-paid, ytVideoId for YouTube, fbPostId for
   Meta-organic, deliveredAt for Article), write a Distributions row
   { status: "skipped", skipReason: "<reason>" }, return early.
3. Execute rowPlan.steps in order, substituting captures and `$`-prefixed
   placeholders. If any tool call errors, write a Distributions row
   { status: "failed", errorMessage, completedAt } and return with the
   error captured.
4. Run the backfill step (writes adId / ytVideoId / fbPostId / deliveredAt
   back onto the variant or article row). Write a Distributions row
   { status: "routed", outputJson: <captures>, completedAt }.

Return JSON: { rowId, channel, status, outputJson?, errorMessage? }.

Per-unit row shape: (channel × rowId). A dual-channel variant (e.g. channels: ["Meta-paid", "Meta-organic"]) produces two units — one Meta-paid, one Meta-organic — independently dispatched. Each does its own backfill on the same store row's distinct columns (adId vs fbPostId). No coordination needed because the backfilled columns don't collide.

5.5 D3-confirm (write, eager) — summary writes

D3-confirm's existing behavior is unchanged: re-query the store, run verifyDistribute. We add a side effect: after the verify, D3 issues mcp__store__create calls to write summary Distributions rows.

  • One row per (variant, channel) the variant claims. status: "routed" if the ground-truth field is populated (adId for Meta-paid, etc.); status: "failed" if missing.
  • One row per article. status: "routed" if deliveredAt is set; "failed" otherwise.
  • All summary rows carry authorStep: "D3-confirm".

The summary rows are the trust-of-record. The webapp page renders D2b rows and D3 summary rows together; mismatches (D2b says routed, D3 says failed) get a 🚩 marker as the "worker lied" signal.

5.6 D4-gate (gate, terminal)

Unchanged. mcp__meta-ads__list_ads check; HG4 clears only when every ad is ACTIVE.

9. Risks & mitigations

RiskMitigation
D2a worker silently produces wrong adsetByCellId shape; D2b units fail substitution D2a verify rejects non-object results; D2b workers fail loudly when a needs-substitution can't resolve, writing a Distributions row with the error. Test fixture covers malformed D2a output.
Per-unit fanout for many variants opens many Meta API connections in parallel; rate limits The orchestrator engine's fanout dispatcher already serializes per-unit Task spawns (1 worker at a time on this branch's harness setup). Even at full Claude Code parallelism, Meta's per-app rate limit is comfortably above the 4-30 simultaneous units this stage produces.
Summary-row writes from D3-confirm double-count attempts on re-run Distributions is append-only by design. A re-run produces fresh attempt + fresh summary; the list page shows the full timeline. This is the intended UX.
Drizzle migration for new table conflicts with another branch's pending migration Standard pnpm db:generate workflow; pnpm lint:migrations catches drift. Branch sandbox isolates testing (ADR-022 + earlier today's branch-DB MCP routing work).
Distributions table grows unbounded over time Per-run cardinality is bounded (rows ≈ approved-variants × channels-per-variant × attempts). Across all runs at current pace, ~100 rows/week. Retention policy is a follow-up if it becomes an issue; not in scope here.
Worker prompt > 50 KB if a rowPlan is unusually large Each unit's staged payload is one rowPlan (a few KB at most — even a 5-step ad creation × 20 fields ≈ 2 KB). The spawnPrompt itself is a fixed instruction string + a 50-byte ref. ADR-024 compliance is structural, not just probabilistic.

11. Out of scope & follow-ups

  • E-024 (IG always-on scheduler). When that ships, IG distribution gets folded into D2b as a new unit category. Spec will reference this one.
  • Auto-retry of failed Distributions. Today a user re-runs /distribute manually. A retry policy (3 attempts, exponential backoff, with the attempt# field already tracked) is a future enhancement.
  • Cross-run analytics dashboard. Roll-ups across runs (success rate by channel, failure pareto by error type, throughput) read from the same Distributions table. Out of scope here.
  • Drill-down on variant/article detail pages. The Distributions section embedded in the variant page is mentioned but not specced in detail; it's a small extension of this design (same data source, scoped filter).
  • D2-route migration of in-flight runs. Runs already past D1-query at the time of merge will need their D1-query row deleted from orchestrator.run_steps to force re-execution under the new schema. Manual one-off, documented in the plan.
