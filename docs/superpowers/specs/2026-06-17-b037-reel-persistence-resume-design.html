B-037 — Reel persistence + resume

Design spec · 2026-06-17 · status: proposed · ticket B-037 (P1) · milestone v1.5

orchestratormcpagentsstore

Fixes the two defects that strand HeyGen Reels at HUMAN GATE 3 with empty assetFiles, and makes both self-heal rather than require manual psql recovery. Unblocks the E-004 G4 gate (5 production Reels through HG3 at ≥60%), which cannot be reached while every Reel run needs hand-recovery first.

1. Problem

A HeyGen Reel renders correctly server-side but reaches HG3 showing "no asset generated yet" — its CreativeVariants row carries assetFiles: []. Reproduced twice (run_1779895374 on 2026-06-01; run_1781193285 on 2026-06-12), each needing a manual psql recovery. The ticket splits it into two distinct defects:

  • Defect A — persistence gap. HeyGen completes and the worker's return payload carries the correct assetFiles (public R2 URLs, HTTP 200), yet the row stays assetFiles: []. The loss is in DB persistence, not computation.
  • Defect B — no resume after advance. On a HeyGen poll-timeout the worker exits non-fatally leaving renderState: HeygenGenerating. But P2-render verifies + advances as soon as the fanout returns refs, so a later plan() returns the HG3 gate and never re-spawns the Reel. The "resumes next pass" promise in reel-render-worker.md:244 is false once P2-render has advanced.

Severity P1 (silent-bug / data-quality class): under-spec output clears automated acceptance and only a human gate catches it.

2. Root-cause analysis (verified)

2.1 Defect A is an id-key mismatch

ReelWorkerInput (packages/orchestrator/src/produce/reel-worker-input.ts) carries two identifiers:

  • id — the database UUID of the row pre-created by P1a-reels-prepare.
  • variantId — a deterministic 12-hex-char hash from shared::variantId(scriptId,"Reel","9:16").

The worker prompt writes its row keyed on the hash — reel-render-worker.md Step 4a (store.update({ id: input.variantId, … }), line 171) and Step 5 (line 196) — but the row's primary key is the UUID (input.id). The update therefore targets a key that is not the row's PK; the assetFiles/reelHeygenJobId/renderState write does not land on the pre-created row. This is the 2026-06-12 "id-mapping" observation, now confirmed against the source. (Phantom keys durationSeconds/subtitleUrl that aren't real columns are a real but secondary contributor.)

2.2 The orchestrator already holds what it needs to self-heal A — it just doesn't

Static variants persist from the worker payload: renderResultsOf() reads assetFiles out of the P2 payload, deriveSpecs builds the spec, and P3-persist writes it via store.create. Reels do not: P3-persist updates the reel row with fillOnlyIfEmpty: true and relies on the worker's (mis-keyed) write for assetFiles. But the reel payload carries variantId (the hash), and reelRowIdByVariantId() (produce.ts) already maps that hash → row UUID. So the orchestrator can write assetFiles from the payload itself — no worker dependency, symmetric with statics.

2.3 Defect B: resume is doubly broken

  • reelWorkerInput() never populates resumeFromJobId (it's in the doc's input shape but not in the projection), so the worker's documented "resumeFromJobId non-null → jump to poll" path is dead.
  • The worker's other resume signal is its own row's reelHeygenJobId — but that was written to the wrong key (§2.1), so it isn't findable at input.id.
  • P2-render has no verify (only P1-fanout does), so a timed-out reel's non-null ref passes defaultVerify and the stage advances, stranding the reel.

2.4 The binding constraint: ADR-024

build-context.ts documents the ADR-024 rule: a StepSpec.build's only permitted I/O is ctx.stageInput. A build may not query the store or call other MCPs. Consequence: the orchestrator's build cannot read row state to selectively re-spawn only pending reels, nor to populate resumeFromJobId from the row. Resume must therefore be worker-self-detected (the worker reads its own row — which workers are allowed to do), and the build stays pure (re-spawns all units; idempotency lives in the worker).

2.5 How the conductor reacts to a failed verify (the lever)

Step kindWhere verifiedOn ok:false
write (e.g. P5-confirm)inside plan() eager loop (eager.ts)synthesizes a halt → conductor STOPs. No re-spawn.
fanout (e.g. P2-render)conductor calls mcp__orchestrator__verifyper loop.md §4: transient → re-spawn once & re-verify; structural / still-failing → STOP.

verify and advance are decoupled MCP tools; the engine never auto-advances on a pass. This is what lets a P2-render verify "hold the stage open."

3. Goals & non-goals

Goals

  • Defect A self-heals: a completed reel's assetFiles always reach the row, persisted authoritatively by the orchestrator.
  • Defect B self-heals: a poll-timeout reel resumes (re-polls its existing HeyGen job) on the next pass and finishes — no manual recovery, no double HeyGen spend.
  • A reel can never silently reach HG3 claiming success with empty assetFiles — that becomes a loud halt.
  • A genuinely failed reel (HeyGen moderation reject, etc.) flows to HG3 flagged, without halting the stage or looping forever.

Non-goals

  • Pixel/vision compliance of rendered assets (covered by the text-scan + human HG3 posture; see §8). Out of scope.
  • The narrative-flow / composition coherence concern (that is B-039).
  • YT-Long 16:9 (E-004a) — this spec is Reel 9:16 only.
  • A build-level "skip already-Uploaded reels" optimization (forbidden by ADR-024; handled by worker idempotency instead).

4. Reel terminal-state taxonomy

Every layer keys off three reel outcomes. Making this explicit is the backbone of the design.

OutcomeRow signalPayload signalL2 (P2 verify)L3 (P5 verify)
Done Uploaded + non-empty assetFiles assetFiles present pass pass
In-flight / timeout HeygenGenerating (+ reelHeygenJobId), empty assetFiles no assetFiles, no terminal error transient-fail → resume halt (should not reach P5 if L2 held; backstop)
Genuine failure RenderFailed, empty assetFiles renderState:"RenderFailed" / error pass (flagged; do not loop) pass (soft flag to HG3)

The defect-A case — Uploaded with empty assetFiles — is the one outcome that must be impossible to ship: L3 treats it as a hard halt. After L1 it should never occur; L3 is the net.

5. Design — four layers

L1 Orchestrator-authoritative reel persist (kills defect A at the source)

Where: P3-persist in packages/orchestrator/src/stages/produce.ts; reel-render-worker.md.

  • Extend renderResultsOf() (or a reel sibling) to extract assetFiles from each reel payload, mapping payload.variantId (hash) → row UUID via the existing reelRowIdByVariantId().
  • P3-persist writes those assetFiles + renderState to the reel row definitively (not fillOnlyIfEmpty for these fields). Packaging fields keep fillOnlyIfEmpty so human HG3 edits survive. Reels become symmetric with statics.
  • Worker-prompt fix (belt & suspenders, and required for resume): Step 4a / Step 5 key on input.id (the UUID), not input.variantId; drop the phantom durationSeconds/subtitleUrl keys (or add columns — see L4); and emit renderState in the return payload for all three terminal outcomes.

L2 P2-render verify = the resume trigger (engine-native, ADR-024-safe)

Where: new verify on p2Render in produce.ts; reel-render-worker.md Step 0.

  • Add a verify to p2Render reading per-unit payloads. Statics always pass (synchronous renders — no async phase to wait on; their real check is P5). Per reel, apply the §4 taxonomy: Done → pass; Genuine failure → pass (flagged); In-flight/timeout → ok:false with a transient-flavoured message.
  • ok:false → per loop.md §4 the conductor re-spawns the fanout once and re-verifies. If the job has since finished, the resumed worker returns assetFiles → pass → advance. If still generating, STOP with "reel still rendering — re-run /produce". The human re-runs; plan() re-enters the un-advanced P2-render and resumes. No new step kind.
  • Resume is worker-self-detected (§2.4). The build re-spawns all reel units; each worker's Step 0 reads its own row by input.id: Uploaded + assetFiles → short-circuit return (no HeyGen call); HeygenGenerating + reelHeygenJobId → jump to poll (Step 4b) with that job — no new submission, no double-spend; otherwise fresh render. The write-key fix (L1) is what makes the jobId findable at input.id.

L3 verifyProduce backstop (the loud guardrail / A3)

Where: packages/orchestrator/src/verifiers/verify-produce.ts; P5-confirm query + projectVariant in produce.ts.

  • Today isStatic (line 56) excludes reels from the empty-assetFiles check. Extend coverage to reels, gated by reelPipelineEnabled() (consistent with the P1-fanout verify gate) so pipeline-off runs — where reels are asset-less by design — are unaffected.
  • Rule: a reel fails verify (halt) if it has empty assetFiles and renderState ∉ {RenderFailed} (i.e. it claims Uploaded or is stuck HeygenGenerating). A RenderFailed reel with empty assetFiles is legitimate → allowed through as a soft flag (data.flags) for the human at HG3.
  • Add renderState to the P5-confirm CreativeVariants query and to the ProduceVariant projection (it is not read today).

L4 store.update validation + doc correction (defense in depth)

Where: packages/store/src/crud.ts; reel-render-worker.md:244.

  • create/update currently pass caller props straight to drizzle (crud.ts:118) with no validation: unknown columns are silently dropped, ill-typed values throw mid-write (partial corruption). Validate props against the table's real columns and reject unknown / ill-typed keys with a clear error, so the next bug of this class surfaces loudly instead of silently losing data — for every entity, not just reels.
  • Decide the phantom keys: either add duration_seconds / subtitle_url columns to CreativeVariants, or remove them from the worker prompt. (Leaning: drop from the prompt — duration already rides inside assetFiles[].duration; subtitle isn't consumed downstream yet.)
  • Correct reel-render-worker.md:244: the "resumes next pass" claim becomes true under L1+L2 — update the comment to match reality.

6. End-to-end data flow

Happy path

P1a-reels-prepare  create skeleton reel row (PK = UUID), complianceCheck stamped
P2-render (fanout) reel worker: submit HeyGen → write jobId+HeygenGenerating to row[input.id]
                   → poll → download → upload → return payload{ assetFiles, renderState:Uploaded }
   L2 verify       payload has assetFiles → pass (statics pass trivially) → advance
P3-persist (L1)    map payload.variantId → row UUID; write assetFiles+renderState definitively
P5-confirm (L3)    re-query rows incl. renderState; every in-scope reel has assetFiles → ok
P6-gate            HG3

Timeout path

P2-render          reel worker times out → row stays HeygenGenerating(+jobId), payload has no assetFiles
   L2 verify       ok:false (transient) → conductor re-spawns fanout once
                   resumed worker reads row[input.id] → jobId present → jump to poll → job done → upload
   L2 re-verify    payload now has assetFiles → pass → advance
   (if STILL generating after the one retry → STOP "re-run /produce";
    human re-run → plan() re-enters un-advanced P2-render → resume → done)
P3/P5              as happy path

Genuine-failure path

P2-render          HeyGen status:failed → worker writes RenderFailed, returns {error}
   L2 verify       RenderFailed → pass (flagged), do NOT loop → advance
P3-persist         reel row stays RenderFailed, empty assetFiles
P5-confirm (L3)    empty assetFiles + RenderFailed → soft flag (NOT halt) → HG3 shows it flagged
                   human discards or regenerates after script edit

7. Error handling & edge cases

CaseHandling
Reel pipeline OFF (EDOS_REEL_PIPELINE unset)No reel rows pre-created; L2 sees only statics (pass); L3 reel check gated off. Existing asset-less-reel-to-HG3 behaviour preserved.
Double HeyGen spend on re-spawnPrevented by worker Step-0 self-detect: an Uploaded reel short-circuits; a HeygenGenerating reel re-polls its existing job (no new generate_reel).
Re-spawn re-renders all staticsAccepted tradeoff of holding P2 open: static results aren't persisted until P3, so a P2 re-entry re-renders them. Statics are local HTML→PNG (no API cost, deterministic, idempotent). The conductor's single auto-retry usually finishes the reel within the same pass, so re-entry is the exception. Future optimization noted in §10, kept out of scope.
HeyGen genuinely slower than worker-timeout + one retrySTOP at P2 with a clear message; human re-runs /produce later → un-advanced P2 re-enters → resume. Matches the (now-true) "resumes next pass" doctrine.
Payload has assetFiles but row write fails at P3L3 at P5 reads the row; empty + non-RenderFailed → halt. Loud, not silent.
Variant-count matrixA RenderFailed reel row still exists (count = 5), so verifyProduce's 5-per-script matrix check is unaffected.

8. Compliance under the new persist

Compliance is a text scan at the persist boundary: store.create/update run complianceScan(entity, props) before writing and reject on !ok (crud.ts:70, :100); the scan walks text fields including the stringified shotlistEn/Bm that becomes the on-frame text. The render-stage verifies (L2, P5) are not the compliance owner. So "statics always pass L2" weakens nothing.

Reel-flag honesty (design note). Reels are created as a near-empty skeleton at P1a (so complianceCheck is stamped on empty content), then updated with real content at P3. store.update re-runs the scan and rejects banned content, but does not refresh the complianceCheck flag. Under L1, P3 still updates the reel row, so this stays true. Two acceptable resolutions, to pick during planning: (a) refresh complianceCheck on the P3 reel update from the content scan; or (b) re-scan reel content at P5. Either keeps the flag meaningful. A banned-content reel already fails loudly today (the P3 update returns {ok:false} → write-step halt), so this is correctness-polish, not a hole.

The text-scan-vs-rendered-pixels seam (a worker could bake on-frame text diverging from the scanned shotlist; only the human HG3 catches that) is the system's existing accepted posture and is explicitly out of scope for B-037 (closer to E-033/E-037 critic-in-produce). File separately if it should be tracked.

9. Testing strategy

TDD — each layer gets a red test first. All DB-touching tests run against an _test / sandbox DB per the project guard.

LayerRed test
L3verify-produce.test.ts: feed a reel { renderState:"Uploaded", assetFiles:[] } with pipeline enabled → assert ok:false. And { renderState:"RenderFailed", assetFiles:[] } → assert ok:true + a flag. And pipeline-off → assert no reel halt.
L2produce.test.ts: P2 verify over a payload set — Done payload → pass; timeout payload (no assetFiles, no error) → ok:false transient; RenderFailed payload → pass+flag; statics-only → pass.
L1produce.test.ts: given a P2 reel payload with assetFiles, assert P3-persist emits a definitive store.update to the row UUID resolved via reelRowIdByVariantId (not a fillOnlyIfEmpty no-op).
L4crud.test.ts: update with an unknown column → assert it returns {ok:false} / throws a clear error rather than silently dropping. Ill-typed timestamp → rejected pre-write.
IntegrationAn engine integration test that walks P2→P3→P5 with (i) a clean reel, (ii) a timeout-then-resume reel (re-spawn returns assetFiles), (iii) a RenderFailed reel — asserting halt/advance/flag outcomes.
Worker promptNo automated test for the prompt; verify by reading the diff (Step 0 self-detect, Step 4a/5 key = input.id, return payload carries renderState). Empirically re-walk a real reel produce once code lands.

11. Decisions resolved & out of scope

Resolved

  • B mechanism = verify-hold on P2-render (L2), not a dedicated post-advance resume step. Engine-native: reuses the existing fanout re-spawn + "first not-done step re-enters" mechanic; no new step kind or gate-check plumbing.
  • Resume is worker-self-detected, not build-populated — forced by ADR-024 (build I/O = stageInput only).
  • A self-heals via orchestrator-authoritative persist (L1), not by re-persisting through a HeyGen re-poll. The asset is already in the payload; just write it to the right row.
  • Genuine failures flow to HG3 flagged, distinct from timeouts which resume.

Out of scope

  • Pixel/vision compliance of rendered assets (§8).
  • A build-level "skip Uploaded reels / re-render no statics" optimization — blocked by ADR-024; worker idempotency covers correctness, the static re-render cost is local-only and bounded.
  • YT-Long 16:9 (E-004a); B-039 narrative-flow.

Once approved, this proceeds to an implementation plan via the writing-plans skill (HTML + MD per repo convention). Suggested build order (the plan finalizes it): L1 + L4 foundation (reels actually persist; the write layer stops swallowing data) → L3 backstop (now a true net, not a blanket halt) → L2 resume — each TDD'd against the red tests above. Ordering L3 before L1 would halt every reel run in the interim, so persist lands first.
