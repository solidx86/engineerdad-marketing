**Closed 2026-06-02 by ADR-030 (data-first claim binding)** on `feat/data-first-claim-binding`. ### B-033 `v1.5` `P1` `media` — HeyGen 16:9 avatar letterboxes in 9:16 reels (fit not set)
Fixed 2026-05-29 on `feat/heygen-reel-pipeline`. The avatar look is shot 16:9; on a 9:16 canvas HeyGen's default `fit: "contain"` fits-to-width and pads the rest with the background, so the avatar rendered as a ~32%-height band with white bars (visible on run_1779895374's reel). `mcp-servers/heygen-wrapper/src/heygen.ts` sent only `avatar_style: "normal"` + white bg. ### B-032 `v1.5` `P1` `infra` — `pnpm db:migrate` silently skipped orchestrator + analytics migrations — FIXED 2026-05-30
Surfaced 2026-05-30 while rebuilding a fresh live `engineerdad` from the HG1-pre E2E snapshot. `pnpm db:migrate` runs `drizzle-kit migrate` for store → orchestrator → analytics in sequence, but all three `drizzle.config.ts` used drizzle's **default shared journal** `drizzle.__drizzle_migrations`. Once store recorded its 3 migrations there, drizzle-kit judged orchestrator's and analytics' single (idx-0) migrations as already-applied and **skipped them** — reporting "migrations applied successfully" while `orchestrator.*` / `analytics.*` tables were never created. The live DB had no journal at all (bootstrapped via `db:sandbox` push / snapshots, never `migrate`), which masked this. Fix (this commit): give each package a **distinct journal table in the neutral `drizzle` schema** — `__drizzle_migrations_{store,orchestrator,analytics}` — via `migrations: { schema, table }` in each `drizzle.config.ts`. (First attempt put each journal in the package's *own* schema, which then collided with that package's `CREATE SCHEMA` since drizzle-kit pre-creates the journal schema; the neutral-`drizzle`-schema + distinct-table form is correct.) Verified on a throwaway DB: fresh `db:migrate` → all 3 schemas populated (public 9 / orchestrator 3 / analytics 5), migration 0002 present, re-run idempotent. NOTE: the live `engineerdad` rebuilt during this session was schema-applied via raw `psql` of the migration SQL (pre-fix workaround), so it has no journal rows — a later `db:migrate` there would re-attempt 0000 against existing objects; drop + `db:migrate` for a journaled DB if that one is kept.

### B-025 `v1.5` `P1` `orchestrator` — `verify-experiment` halts on any empty cell; no concept of degraded experiment — FIXED 2026-05-29
Surfaced 2026-05-28. `verifyExperiment` (`packages/orchestrator/src/verifiers/verify-experiment.ts:22-24`) reports "cell X: no variants mapped" as a failure if ANY cell in the design has empty `variantPageIds`. This halts the loop whenever the operator approves variants from fewer angles than Brain proposed — i.e., M < N where N = `factors[0].levels.length`. Empty cells happen routinely: operator triage, brief-writer rename (B-026), no scripts approved for a given angle, etc. The verifier needs a tri-state: `full` (every cell has variants — pass), `degraded` (≥2 cells have variants — pass with warning, write `experimentStatus: "degraded"` to the Experiments row), `single-cell` (only 1 cell has variants — pass as routing-not-experiment, write `experimentStatus: "single-cell"`), `broken` (0 cells have variants — fail). Brain's `/reflect` needs to see `experimentStatus` so it knows the prior cycle was a real comparison vs. a single-cell route; otherwise the hypothesis graduation feedback loop becomes lossy. **Fixed 2026-05-29** — verify-experiment tri-state + X3-write persists `experimentStatus` (B-025 commits on `feat/brain-experiment-params`).

### B-024 `v1.5` `P2` `orchestrator` — X3-write filters Hypotheses by missing `pageId` field; testExperiment back-link never written
Fixed 2026-05-28. `x3Write.build` cast the `Hypotheses` rows out of X1-query as `{ pageId?: string }[]` and filtered for `h.pageId`, but the store's `ALWAYS_RETURNED = ["id","title"]` projection returns `id` (not `pageId`). The filter always yielded an empty array, so the `mcp__store__update` calls that stamp `testExperiment: $experiment.id` on each Hypothesis row never executed. The Hypothesis↔Experiment back-link was silently broken on every run that reached X3. Pre-existing Notion-era leftover surfaced by the B-023 final review. Fix: rename the cast + filter + access to use `id`, add a regression test that asserts one `Hypotheses` `mcp__store__update` per X1-returned row with the correct id and `testExperiment: "$experiment.id"`.

### B-023 `v1.5` `P1` `orchestrator` — experiment cells map to zero variants (`factorTags` never populated)
Fixed 2026-05-28. `mapCellsToVariants` matches cells to variants by `variant.factorTags[k] === cell.factorLevels[k]`, but `factorTags` existed only in test fixtures — no store schema column, no projection in X1-query, no writer in P3-persist. Surfaced on run_1779895374, the first run whose experiment stage actually executed (Brain Initiative Phase 0 wired `experimentParams` from the Decision Memo). Every cell ended up with empty `variantPageIds`, which made distribute's `cellIdFor()` return `null` for every Meta-paid variant. Fix: expand X1-query from 3 to 5 store calls (add `Scripts` + `Briefs` with explicit field projections), then walk variant.script → script.brief → brief.{angle, budgetBucket} in a pure `projectExpVariant()` helper called from `allocatedCellsFor()`. Angle-only scope; future generalization tracked in E-042 (+ format + persona) and E-043 (hook_register + language).

### B-022 `v1.5` `P1` `orchestrator` — `designCellsOf` snake_case ↔ camelCase mismatch
Fixed 2026-05-28. The experiment MCP emits `cell_id` / `factor_levels` per its public schema; the orchestrator's `DesignCell` interface expects `cellId` / `factorLevels`. `designCellsOf` cast without translating, so downstream `Object.entries(cell.factorLevels)` threw `Cannot convert undefined or null to object`. ### B-021 `v1.5` `P1` `orchestrator` — `renderResultsOf` strips Feed renders, leaving `asset_files` empty
Fixed 2026-05-28. `renderResultsOf()` in `packages/orchestrator/src/stages/produce.ts` only collected `{variantId, url, sha256}` from `scenes[]` and required `variantId` on every scene. But per `corpus/templates/worker-prompts/render-worker.md`, single-aspect Feed workers intentionally omit `variantId` on `scenes[]` entries (it's only at the top-level `rendered[0].variantId`); dual-aspect Carousels keep it on each scene to disambiguate aspects. Result on run_1779895374: 4 Carousel variants populated 6–7 assets each; both Feed variants got `asset_files = []`, halting P5-confirm with "Asset Files empty". Fix: when a scene lacks `variantId` and the worker has exactly one `rendered[]` entry, default to its variantId; also added a safety net that includes `rendered[]` entries not already covered by scenes. Backfilled the two halted Feed variants in DB by hand to clear the live run.

### B-020 `v1.5` `P1` `orchestrator` — distribute audit log lost on D3a halt
Fixed 2026-05-27. Stage order was `[…, D3a-confirm (write + verify), D3b-summary (write)]`. When D3a-confirm's `verifyDistribute` failed, the loop halted before D3b ever ran — so the `Distributions` audit log was empty exactly when operators most needed it (the failure case). Also surfaced: `D3b-summary` only emitted `status: "routed" | "failed"`, lumping never-attempted variants in with attempted-but-didn't-land. Fix: removed verify from D3a (it's now just a re-query); moved the verify to D3b, where it filters out variants the planner couldn't route at all (no metaSpec, no cellId, no organic schedule, etc.) before delegating to `verifyDistribute`. Added `status: "skipped"` with `skipReason` for plan-time-not-routable rows so the audit log preserves the distinction between failure and never-attempted.

### B-019 `v1.5` `P1` `orchestrator` — `planArticles` emits incomplete `draft_article` args
Fixed 2026-05-27. `planArticles` in `packages/orchestrator/src/distribute/plan-distribution.ts` only emitted `{slug, lang, brief_markdown, faq_markdown}` to `mcp__engineerdad_site__draft_article`, but the MCP's `ArticleSpecSchema` requires 7 more fields: `title`, `description`, `topic_tag`, `reading_time`, `keywords`, `og_image`, `date_published`. ### B-018 `v1.5` `P1` `infra` — subagent MCP permissions allowlist too narrow
