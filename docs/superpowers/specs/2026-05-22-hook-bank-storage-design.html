Hook-Bank Storage

Approach D — carry the hook bank in the run record, not on Notion Script rows

Date
2026-05-22
Status
Design approved — pending spec review
Origin
The run_1779446750 walk: a creative-director spawn spent ~9 minutes crawling a spilled file because an unfiltered Scripts query overflowed the tool-result token cap.
Related
B-016 (verifier audit) · E-027 (content fan-out) · E-028 (hook-level learning)

1. Problem

Every Scripts row in Notion carries a Hook Bank property — a JSON array of ≥30 bilingual hooks ({en, ms, register}). content-writer copies the same bank onto every Script of a Brief. Measured against the 6 approved Scripts of run_1779446750:

LayerSize per Script
Hook-bank content (the JSON of hooks itself)3.4k chars (18 hooks) → 6.6k (30 hooks)
As stored in Notion (rich_text API JSON, double-encoded)8k–15k chars — Notion's wrapper inflates ~2.2×
The whole Script row (all properties)18k–27k chars

The Hook Bank is ~45–55% of every Script row's weight. Six approved Scripts returned 162,451 characters in one query — over the MCP tool-result token cap. The result spilled to a file, and the agent crawled it in ~100-line chunks for nine minutes. The same overflow hit the orchestrator's own HG2 gate-check two steps earlier. Any filter_properties-less query of Scripts busts the cap once a handful of scripts are approved.

2. What the hook bank actually is

Confirmed during brainstorming: the hook bank is machine-only, single-run plumbing.

  • Not human-gated. At HUMAN GATE 2 a human reviews script bodies and the chosen hooks — nobody scrutinises or edits the full 30-hook bank. So it has no reason to live in Notion, which is the OS's human-gate substrate.
  • Single-run, ephemeral. content-writer generates a fresh bank every run; creative-director consumes it minutes later in the same run. Nothing downstream (/reflect, variant render) re-reads the candidate pool. It is never reused across runs.
  • A candidate pool, not a learning record. creative-director uses 4 hooks per Script; the other ~26 are never published and carry no performance signal. Hook-level learning belongs to the used hooks and is a separate concern — filed as E-028.

Two agents need the same bank, so it must be carried from one to the other — but "carried" need not mean "given a database home."

3. Decision

Three options were weighed. Approach D is chosen.

OptionWhere the bank waits between the two agentsVerdict
A — SQLite store A new hook_banks table + two analytics MCP tools (put/get). Rejected — a table and two tools for an artifact that lives minutes. Storage it doesn't earn.
B — Notion Brief row Move the column from Scripts to Briefs (de-duped 6→4). Rejected — bolts machine plumbing onto the human-gated Brief row, clutters HG1, and only shrinks the bloat rather than removing it.
D — pass-through, no storage Inside the C1-write result, which the orchestrator already persists in the run record. Chosen — zero new storage, schema migration, or MCP tool; the bank rides a result the orchestrator was keeping anyway and dies with the run.
Load-bearing assumption — confirmed. RunStepState.result is persisted, and every step is stage-tagged so run.steps accumulates across stages. produce.ts already ships a stepResult<T>(run, stepId) helper, and P2–P5 already read prior step results through it. Approach D reuses an existing pattern — it is not a new mechanism.

4. Data flow

content-writer authors per-Brief hook banks (≥30 hooks each)
   │  ✗ no longer writes a "Hook Bank" column onto Script rows
   └─→ returns them IN the C1-write result:
         hookBanks: [ { briefId, hooks: [ {en,ms,register} ×30 ] } ]   ← full, not a summary
   │
verifyContent → counts the REAL hooks per register from hooks[]        (tightens B-016)
   │
advance → persists the C1-write result into run.steps                  (already happens)
   │
   ···  HUMAN GATE 2  ···
   │
produce / P1-creative.build(run):
   stepResult(run,"C1-write").hookBanks  →  injected into the creative-director spawnPrompt
   │
creative-director: reads banks from its prompt; queries Scripts only for
   script bodies + CTA, now with filter_properties

5. Changes

Seven small, coherent edits. No new database, table, MCP tool, or migration.

#ComponentChange
1 .claude/agents/content-writer.md Stop writing the Hook Bank Script column. The C1-write result's hookBanks field carries the full hooks[] per Brief, replacing the {registerCounts, totalHooks} summary.
2 packages/orchestrator/src/verifiers/verify-content.ts (+ test) Count the real hooks per register from hooks[] instead of trusting content-writer's self-reported registerCounts. Closes the B-016 weakness (this run reported 18, wrote 30).
3 packages/orchestrator/src/stages/produce.ts — P1-creative.build Read stepResult(run, "C1-write") and inject all per-Brief hook banks into the creative-director spawnPrompt. build() throws if the C1-write result carries no banks.
4 .claude/agents/creative-director.md — Step 2 Read hook banks from the spawn prompt, not a Script column. Its Scripts query (for bodies + CTA) gains filter_properties so it stays well under the cap — the workaround folded into the cure.
5 packages/notion-bootstrap/src/schemas.ts Drop the Hook Bank column from the Scripts DB definition. The live-workspace column becomes vestigial (empty); a migrate: to physically delete it is low-priority cleanup, not required.
6 mcp-servers/notion/src/extract-text.ts Remove extractHookBank() and the "Hook Bank" branch — dead code once the column is never written.
7 packages/shared/src/zod.ts + types.ts Remove ScriptSchema.hookBank and Script.hookBank. Planning confirmed the HookSchema / HookBankSchema / Hook cluster is then fully orphaned (nothing imports it) — remove it too. The result's per-Brief bank is typed by a plain HookBankEntry interface in verify-content.ts.

A stale line in the shared tactical-piliero.md prompt fragment — it claims HookBankSchema enforces the hook count — is corrected alongside change #7 and re-synced with pnpm sync:agents.

6. Error handling & edge cases

  • Missing C1-write result. P1-creative.build() must fail loud if stepResult(run, "C1-write") is absent or carries no hookBanks — it should never silently dispatch creative-director with an empty bank set. (Cannot happen in a normal run; produce only runs after content advances — but the guard is cheap and the failure mode is otherwise silent.)
  • All banks passed, not filtered. P1 injects every per-Brief bank from the C1-write result; the creative-director uses the ones matching its approved Scripts and ignores the rest. Filtering in P1 would couple it to the C2-gate result shape for a negligible saving.
  • Resume safety. The banks live in persisted run state, so a run resumed at P1-creative still has them — no re-query, no regeneration.
  • Prompt size. ~4 banks ≈ 24k chars embedded in the spawnPrompt. A prompt is not a capped tool result; this is acceptable and far below any limit.

7. Compliance note

Trade-off, accepted. Removing the Notion column (change #6) means the hook bank is no longer compliance-scanned at the Notion write boundary. Mitigation: content-writer already runs the compliance scanner on hooks at authoring time, and every hook that is actually used is re-scanned when its creative variant lands in Notion. The ~26 unused candidates per bank are never published, so an at-rest scan of them protects nothing.

8. Testing

  • verify-content.test.ts — update fixtures: hookBanks[] now carries hooks[]; assert the verifier counts real hooks per register and rejects an under-30 bank, a missing register, and a register below 3.
  • produce.test.ts — assert P1-creative.build() reads the C1-write result and embeds the hook banks in the spawnPrompt; assert it injects only banks for Briefs with approved Scripts; assert it fails loud when the C1-write result is missing.
  • Full suite green: pnpm -r build (sequential) then pnpm test.

9. Scope boundary

This design fixes storage of the candidate bank. Three adjacent items are deliberately out of scope:

  • B-016 — the broad verifier audit. Change #2 tightens verifyContent; the rest of the audit stays separate.
  • E-027 — content-stage fan-out. Independent; this design works the same whether the content stage is one spawn or a fan-out.
  • E-028 — hook-level learning (structured register on creatives, register-aware selection). The candidate bank carries no learning signal, so it is correctly absent from this design.
