Reel visual scenes: two-track charts/concept-visuals with baked-in emphasis

Design spec · 2026-05-30 · refines ADR-028 (HeyGen-native scene assembly) · scope: Reel pipeline only · status: approved, pre-plan
One-line summary. Make a reel's non-face scenes carry real explanatory power: a two-track visual model (library-grounded data charts + free-form concept visuals), narration that explains and emphasizes one point per visual, support for multiple visual scenes per reel, a reel-specific reference-design set, and a frames-only dev harness so the render loop can be tuned without running /loop or HeyGen.

1. Context & problem

Today (post ADR-028) the reel pipeline assembles a vertical MP4 from a multi-scene HeyGen render: face scenes are the avatar talking; chart scenes are a full-frame chart image with voiceover. Three things limit it:

  • Thin chart narration. Each chart scene gets a single ≤30-word voiceover over a static frame. There is no first-class way to say what the chart explains or what to emphasize.
  • Effectively one chart. The scene array is technically unbounded, but the creative-director is never told to use more than one, and the chart inventory is artificially capped at 3 of the 14 vetted charts.
  • "Chart" is too narrow. Many ideas are better served by a non-numeric visual (a comparison, a labelled diagram, a metaphor) than by a data chart.

A key finding from code review: the render substrate is already general — render_html_to_png({ html }) renders any self-contained HTML, and the static Feed/Carousel worker already authors brand-compliant HTML (charts and free-form cards). So this is mostly an authoring + prompt + plumbing change, not a new rendering capability.

Defect uncovered. reel-render-worker.md Step 2 calls render_html_to_png({ chartRef }) — a parameter the tool does not accept — and never tells the worker to read the chart YAML and build HTML. The reel chart-render path is therefore under-specified today. This spec replaces that step with the static worker's proven recipe.

2. The two-track visual model

face-over-chart is dropped from the shotlist enum (ADR-028 already aliased it to chart). The HeyGen wrapper's own enum may keep it harmlessly.

3. Narration model — one explaining VO per visual, emphasis baked in

Each chart/visual scene gains two authoring fields that steer both the voiceover and the rendered frame:

  • explains — the one-line takeaway the narration must land.
  • emphasize — chart scenes only: what to spotlight/annotate (e.g. "the year-30 gap", "Fund B's +3% alpha bar"). Concept visuals carry their focal point inside the brief, so emphasize does not apply to them.

Voiceover budget: face stays ≤30 words; chart/visual relax to ≤45 words (~18s) so the avatar can actually explain. Total length stays sane via the existing targetSeconds ≥ Σ estimatedSeconds rule — more visual scenes means tighter VO each.

Decision: no multi-beat / animated reveal. One frame, one VO, emphasis pre-applied. HeyGen plays the frame for the VO duration.

4. Multiple visual scenes (the two scenarios)

Both requested shapes are representable today (the scene array is unbounded) and the generate_reel wrapper already accepts N interleaved scenes — no wrapper change needed. What changes is creative-director guidance:

  • Scenario 1 — charts spread: face → chart → face → chart → face/CTA.
  • Scenario 2 — consecutive charts: face → chart → chart → face → face/CTA.

Authoring guardrails: one idea per reel; faceFirstHook stays true (face opens); a face scene must close/CTA; ≤3 visual scenes per reel (chart or concept, interleaved or consecutive); never an all-chart reel.

The creative-director prompt fragment lives in packages/shared/src/prompts/*; run pnpm sync:agents after editing and verify with pnpm sync:agents:check.

8. Reel-render-worker rewrite

Replace Step 2 with the static worker's recipe, adapted for 9:16 + emphasis. The reel-specific tail (upload frames → generate_reel → poll → store) is unchanged and already loops over N scenes.

8.1 New Step 2 — build & render each visual frame (1080×1920)

  • chart: Read corpus/data/charts/<chartRef>.yaml → map paletteEmphasis to brand tokens → build Chart.js HTML via the shared corpus/templates/partials/chartjs-config.js buildChartConfig() + window.__chartsReady signal → apply emphasize (see 8.2) → render caption + full source_citation (data unchanged) → render_html_to_png({ html, width:1080, height:1920, wait_for_charts:true, run_id, variant_id, scene_id }).
  • visual: compose brand-compliant self-contained HTML from visualBrief + explains + onScreenText, bound by brand contract §6 and the no-numbers rule → render (same call, wait_for_charts:false).
  • face: no frame.

8.2 Emphasis mechanism (decision: HTML/CSS overlay, not the Chart.js annotation plugin)

  • Spotlight: in the config the worker already builds, render the emphasized series/category at full hero color and dim the others (reduced opacity / muted token). No new dependency.
  • Callout: a positioned HTML badge/label over the chart naming the emphasized point. Robust and fully stylable; avoids the annotation plugin's CDN + pixel-positioning cost.

8.3 Reuse boundary

Reuse is the buildChartConfig partial + citing render-worker.md §3–§5 for the Chart.js / __chartsReady mechanics. It is not a literal section-extraction: the 9:16 safe-area layout (§9) is genuinely reel-specific. The reel worker keeps its own frame-building section that points at the shared pieces.

8.4 Frames-only harness mode

A short mode-switch subsection (like creative-director's "Worker mode"): when the spawn prompt declares frames-only mode, do Step 2 + the visual-QA pass, return the PNG paths + QA notes, and skip Steps 3–5 (no HeyGen upload, no generate_reel, no store writes). Used by the dev harness (§11).

9. Reel safe-area rule (gap G2)

Gap G2 — reel safe-area ≠ static safe-area. Feed/Carousel cards are never UI-occluded. IG/FB Reels overlay the right action rail, the top, and roughly the bottom 20%. The static worker pins the logo bottom-right and the citation along the bottom — exactly where the platform UI sits. Naive reuse would bury the logo, citation, and emphasis callout under the Reels UI.

Rule (proposed starting margins, tunable): keep all text, logo, citation, and emphasis callouts inside the central column — clear the top ~14%, the bottom ~20%, and the right ~12% of the 1080×1920 frame. Encoded in both the reel-render-worker prompt and the reference-design exemplars.

10. Compliance guardrails (gap G3)

Gap G3 — concept-visual on-frame text is un-scannable. The produce verifier scans assembled caption text for compliance sentinels; text baked into a PNG is invisible to it. Chart frames are safe (vetted YAML). A free-form visual could bake a fabricated stat that nothing catches until the HG3 human sees the PNG.
  • HARD RULE in both creative-director and reel-render-worker prompts: concept visuals carry no numbers/stats. Anything quantitative must be a vetted chart.
  • Brand contract §6 binds all visual HTML (palette-only colors, the two allowed fonts, logo present, no animation).
  • HG3 (rendered-creative review) is the named human backstop — same checkpoint that reviews chartRef selection today.
  • emphasize is visual-only spotlight; it never alters the YAML's numbers or citation.
  • Organic-caption compliance footer is unaffected (it lives in source.scriptBody, not the frame).

11. Reference designs (new §6 of the reel prompt)

A reel-specific "Reference designs" section, mirroring render-worker.md: before composing a frame, Read the matching 9:16 exemplar and match its density, rhythm, and quality bar. It points only at reel exemplars — the existing Feed exemplars are 1:1/4:5 and would teach the wrong rhythm.

The exemplars do not exist yet. The plan authors them the way the Feed exemplars were made: hand-author brand-compliant HTML → render at 1080×1920 → commit. Source HTML is committed alongside the PNG so designs are reproducible and tweakable.

12. Dev harness — frames-only render loop

Mirror the existing scripts/test-static-worker.mjs pattern (assemble a spawn prompt, print a copy-pasteable Task() call that reuses the live MCP stack; no SDK, no new infra).

  • scripts/test-reel-frames.mjs + pnpm test:reel-frames <fixture> — reads the reel prompt + a reel fixture, assembles a frames-only spawn prompt, prints the Task() call (--copy / --open flags). The worker builds + renders every chart/visual frame at 1080×1920, runs visual QA, returns PNG paths, and stops before HeyGen. No /loop, no HeyGen, no cost.
  • scripts/fixtures/reel-worker/*.json (ReelWorkerInput-shaped) — seed: a chart+emphasis scene, a concept-visual scene, and one mixed face/chart/visual reel.
  • Two loops, both covered: tweak the worker prompt → test:reel-frames, eyeball PNGs, edit prompt, re-run. Refine an exemplar → edit its committed src/*.html → re-render (a single render_html_to_png call, no agent).

16. Testing

  • Unit: zod accepts/rejects each sceneType's field combination; conditional VO cap; projection carries the new fields + palette.
  • Harness smoke: pnpm test:reel-frames on each seed fixture produces PNGs at the expected path; visual QA pass against the exemplars.
  • Compliance: a visual brief containing a numeric stat is rejected by prompt rule (manual/HG3); chart emphasis leaves YAML numbers + citation intact.

17. Out of scope

  • YT-Long, Feed, Carousel enrichment (cap-lift aside).
  • Multi-beat / animated chart reveals (synced per-beat emphasis).
  • Fully-inline data charts (LLM-authored numbers) — rejected on compliance grounds.
  • Bilingual on-frame text — reels remain EN-only (ReelWorkerInput.language = "en").
  • Hardening targetSeconds ≥ Σ estimatedSeconds into a zod refine.
