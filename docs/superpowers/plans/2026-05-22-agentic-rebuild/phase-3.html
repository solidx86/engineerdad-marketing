Three decisions baked in 1 — distribute = worker + deterministic plan + verifier. Routing decisions are a pure, unit-tested planDistribution(); a generic general-purpose worker executes the plan (threads Meta's campaign→adset→ad id chain, polls async uploads); verifyDistribute re-reads Notion for back-fill truth — the guarantee (design §10.1). No domain agent, no copy derivation (ADR-018 intact).
2 — gate-passing. A gate Step gains an optional check (an MCP read); the conductor runs it, verify inspects it — cleared ⇒ advance, else STOP. A gate with no check is terminal (always STOP) — HG4's shape this phase. This is what lets a run flow past HG3 into Phase 3's stages.
3 — registry timing (C1-consistent). experiment and distribute ship as tested StageDefinitions + verifiers; the live FIXTURE_REGISTRY is unchanged. Phase 5 wires the ordered live registry. Deleting distribution.md / experiment-os.md and re-pointing their commands moves to Phase 5 — doing it now, with the stages not yet live and brain.md still referencing them, would leave a broken intermediate state.

Goal: Two new orchestrator stages — experiment (pure step-chain) and distribute (worker + verifier) — plus the gate-passing mechanism, all unit- and integration-tested, built against Phase 2's frozen engine.

Architecture: experiment is a deterministic 3-step chain (query → experiment.design MCP → write) with a pure 70/20/10 allocation overlay. distribute is a 4-step stage whose middle step is one agentic cell — a worker executing a pure planDistribution() plan, guarded by verifyDistribute. Gates become verifiable steps.

Tech stack: TypeScript (ESM), node:sqlite, vitest. New deps: none. Reuses Phase 1's youtubeCategoryId from @engineerdad/shared/derive. Execution: TBD (this doc is for review first).

1. Locked interfaces

Phase 2's structural types (RunState, StepSpec, StageDefinition, VerifyResult, the engine surface) are unchanged. Phase 3 makes exactly two additive edits to types.ts and adds the contracts below.

1.1 — types.ts additive edits

// RunStage — two new stage ids (a stage enum inherently grows per stage)
export type RunStage =
  "fixture" | "produce" | "experiment" | "distribute" | "done";

// Step — the gate variant gains an OPTIONAL `check`: an MCP *read* the
// conductor runs to detect whether the human gate has cleared. A gate WITH a
// check is passable (cleared => the run flows on); a gate WITHOUT one is
// terminal (the conductor always STOPs) — HG4's shape until Phase 4.
  | { kind: "gate"; stepId: string; gate: string; message: string;
      check?: { tool: string; args: unknown } }

Extending RunStage is not a breaking change — the engine treats run.stage opaquely (compares it to registry stage ids; "done" is the one special value). Adding an optional check is additive; the one existing gate (produce's P4) gains a Notion check in the same task so the build never goes red.

1.2 — The conductor's gate branch (loop.md)

gate, NO check   -> print step.message ; STOP            (terminal — e.g. HG4)
gate, WITH check -> execute step.check.tool / step.check.args   (an MCP read)
        v = mcp__orchestrator__verify({ runId, stepId, result: checkResult })
        v.ok === true  -> mcp__orchestrator__advance({ runId, stepId,
                            result: checkResult }) ; goto 1      (gate cleared)
        v.ok === false -> print step.message ; STOP              (awaiting human)

A gate becomes just another verifiable step — the "worker" is the human, the "result" is the Notion/Meta approval state. When STOPped, the next /loop re-plans, gets the same gate, re-runs the check — idempotent by construction.

1.3 — plan sets awaiting_gate

When the step plan returns is a gate, it persists run status awaiting_gate (a pure function of the registry — the step's kind); any other step ⇒ active. /status then shows a gated run as awaiting_gate rather than a misleading active.

1.4 — 70/20/10 allocation (experiment/allocation.ts)

export interface MappedCell {
  cellId: string;
  factorLevels: Record<string, string>;
  variantPageIds: string[];
  bucket: "70" | "20" | "10";
}
export interface AllocatedCell extends MappedCell { allocationPct: number; }

/** Attach variantPageIds whose factor tags match factorLevels; derive bucket
 *  from the majority Budget Bucket of matched variants (default "20"). */
export function mapCellsToVariants(
  cells: { cellId: string; factorLevels: Record<string, string> }[],
  variants: { pageId: string; factorTags: Record<string, string>;
              budgetBucket: "70" | "20" | "10" | null }[],
): MappedCell[]

/** Per-cell allocationPct = bucketShare(b) / n_b, where bucketShare = 70|20|10
 *  and n_b = cells in bucket b. A bucket with 0 cells redistributes its share
 *  proportionally across non-empty buckets. Result allocationPcts sum to 100. */
export function applyAllocation(cells: MappedCell[]): AllocatedCell[]

1.6 — planDistribution + DistributionPlan (pure)

export interface ToolStep {
  tool: string;                       // MCP tool name
  args: Record<string, unknown>;       // literal args (spec read verbatim)
  captures?: string;                  // label for this call's result
  needs?: string[];                   // captures the worker must substitute in
  poll?: { tool: string; untilField: string; untilValue: string };
}
export interface RowPlan {
  rowId: string; channel: string;
  action: "route" | "skip"; reason?: string;
  steps: ToolStep[];
}
export interface DistributionPlan {
  runId: string;
  rowPlans: RowPlan[];
  backfills: ToolStep[];              // notion update_page calls (needs[] → ids)
  skipped: { rowId: string; reason: string }[];
  notes: string[];
  dryRun: boolean;
}

export function planDistribution(
  variants: DistVariant[],
  articles: DistArticle[],
  cells: AllocatedCell[],
  opts: { channelFilter?: string[]; dryRun?: boolean },
): DistributionPlan

planDistribution is the whole determinism story for distribute: channel routing, the EDOS_<runId> / <runId>__<cell>__… naming conventions, per-cell budget math, idempotency gates (a variant with Ad ID populated ⇒ skip), gap detection (a channel present but its spec fields empty ⇒ skip + note), and dryRun. It never derives copy — spec fields are read verbatim onto ToolStep.args. Runtime values (campaign_id, adset_id, creative_id, image_hash) are not literals — the producing step captures them, the consuming step needs them, and the worker substitutes. This makes the plan a fully testable data structure.

1.7 — verifyDistribute (pure)

export function verifyDistribute(
  expectedVariants: DistVariant[], expectedArticles: DistArticle[],
  actualVariants: DistVariant[],   actualArticles: DistArticle[],
): VerifyResult

The guarantee. Re-reads-Notion truth: every expected Meta-paid variant must have a non-null Ad ID in actual (both langs, or one + a note); every YouTube variant a YT Video ID; every approved article a Delivered At. Idempotent — a row already satisfied in expected passes. Every unrouted row is a problem.
