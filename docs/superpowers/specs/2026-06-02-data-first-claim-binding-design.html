Data-First Claim Binding

Design spec · 2026-06-02 · branch feat/data-first-claim-binding
P0 Closes B-038 · folds in B-036 · files mode-C (composition drift) as a sibling.
New doctrine — warrants a dedicated ADR. Amends creative-director.md, both render-worker prompts, brand-contract §9.

A quantitative claim ships a chart-backed visual only if a vetted dataset depicts that claim's scenario and numbers. The claim↔data binding is authored where the claim is authored (content-writer), reviewed by the human (HG2), executed — never invented — downstream (creative-director), and enforced deterministically (C1 + P1 verifiers). When no dataset exists, the claim is held until a human-invoked utility derives one from real source material.

1. Problem

The chartRef — the claim↔data binding — is currently born at the creative-director, at produce time, after HUMAN GATE 2. The Script carries proofRefs (prose proof files) but no chart link and no structured claim→dataset binding. So CD reverse-engineers a chart from the script prose, picking by topical adjacency from a fixed ~16-YAML inventory. Nobody ever asserts "this claim is backed by this dataset." The binding is a downstream guess, created last, after the only human gate that could catch a mismatch.

B-038 (the trigger, P0). The TOFU script "A Million in EPF Still Ran Dry at 72" (scriptId 32da66d3…) claims "even RM1.2M at 60 runs out around age 72 at the Adequate lifestyle" — an EPF retirement drawdown. CD bound it to inflation-vs-savings-real-value (RM100k in a 2% savings account eroding to RM59k over 18 years — an accumulation-phase argument). The chart rendered faithfully to its YAML; the defect is the pairing. The static (Carousel) path had no editorial catch — the mis-paired chart shipped into a finished, real-looking asset. For a FIMM-registered consultant this is trust-destroying. B-036 (numbers leaking into a concept visual) is the same class of defect, caught only by render-worker luck.

2. Doctrine — data-first

The bound dataset is authoritative; the narration conforms to the data's actual numbers and scenario — never the reverse. This makes drift structurally impossible: you cannot narrate "RM1.2M to age 72" off a YAML that says "RM100k savings erosion," because the words are read from the dataset, not matched to it afterward.

The full chain becomes machine-traceable:

claim  →  chart YAML  →  dataset (JSON)  →  verified external source

5. The ClaimBinding (new claim_bindings jsonb on Scripts)

{
  claim:    string,                          // "RM1.2M at 60 runs dry by ~72 at Adequate lifestyle"
  kind:     "data" | "qualitative" | "gap",
  chartRef: string | null,                   // vetted id, iff kind === "data"
  figures:  string[],                        // ["RM1.2M","age 60","age 72"] — must trace to the YAML
  takeaway: string,                          // ≤12 words; becomes CD's `explains` anchor
  gapNote:  string | null                    // iff kind === "gap": the dataset that's missing
}

A Script may carry 0..n bindings (zero = pure brand/qualitative — valid). figures is the load-bearing field — it is what the deterministic trace validates.

Kind is determined by one question: does the statement assert a financial number?

  • Yes → data (dataset exists) or gap (held). There is no reword-to-keep-the-number escape — a persuasive figure may not ship without its data.
  • No (genuinely conceptual) → qualitative → concept visual, ships. qualitative is not a downgrade hatch for numeric claims.

6. content-writer changes (authoring · stays Sonnet)

  • New input surface: the chart inventory via a new mcp__corpus__list_charts (mirrors list_proof) returning per chart: id, title, scenario fingerprint, labels, value-ranges, source_citation. Implemented as a live directory read of corpus/data/charts/ (the BM25 index never carries YAML — see §9).
  • New step 4d — claim binding: per Script, enumerate the quantitative claims → match a dataset → bind (data), classify qualitative, or raise a gap. Author the body's numbers from the matched YAML. Never mis-pair.
  • Internal QA (pre-emit, mandatory) — mirrors the existing hook-bank self-check (see §8).

7. creative-director changes (staging, pinned)

  • Reads the Script's claim_bindings (already fetches the Script).
  • May set a scene chartRef only to a chartRef in the Script's data bindings. The Step-3c global chart inventory list in creative-director.md is deleted and replaced with "bound chartRefs only."
  • Data-scene explains := the binding's takeaway; voiceover segments the already-conformant body. CD keeps scene order, format, palette, hook rotation, and where the chart lands.
  • Binding is permission, not obligation — CD may render a bound chart 0..N times; the verifier only rejects unbound chart scenes.

8. Enforcement — three layers

The split: the LLM reasons which statements are claims and picks the chart; code proves the numbers trace and catches misses. Model can't hand-wave figures; code can't read prose.

  1. Layer 0 — content-writer internal QA (pre-emit, mandatory). Re-read body; enumerate every quantitative token; confirm each maps to exactly one binding (uncovered ⇒ not done); for each data binding re-open the chart metadata and confirm figures trace + scenario matches (else downgrade to gap or fix body); confirm no qualitative/concept claim smuggles chart-numbers. Emit only when all pass.
  2. Layer 1 — C1 deterministic validator (verify-content.ts, before HG2): every data binding references a real chart id and its figures trace to the YAML (HARD); a numeric-token coverage scan over the body flags RM/%/figure tokens not covered by any binding and not on the incidental allowlist → escalates to HG2 (SOFT).
  3. Layer 2 — P1 produce verifier (verify-produce.ts, the P0 hard gate, runs over all formats incl. Carousel/Feed): every scene with a chartRef → that ref ∈ parent Script's data bindings, else reject; scene explains references the bound figures/takeaway; concept visual (visualBrief, null chartRef) contains no digits (folds in B-036).

The incidental-number allowlist (claim detection)

A claim = a quantitative assertion about a financial outcome (returns, balances, shortfalls, inflation, fees, compounding timeframes). The coverage scan ignores incidental numbers: the compliance footer, targeting age (25–60), durations (30s), step counts, dates/years, CTA figures.

figures↔YAML trace algorithm

Each required figure must appear in the YAML's labels / values / title / source after numeric normalization (RM/k/M/%/age), with a rounding tolerance. Derived assertions (e.g. "41% loss") are validated against a value computed from the series (within tolerance) or flagged for HG2. A figure that does not trace ⇒ the binding is invalid ⇒ gap or body fix. (B-038's "RM1.2M" is absent from inflation-vs-savings → fails here.)

9. Promotion needs no reindex

Verified in code: reindex (packages/corpus/src/reindex.ts) indexes only .md/.txt/.vtt/.pdf under compliance/courses/proof/knowledge; .yaml and data/ are skipped. Charts are read by path, live (Read corpus/data/charts/<ref>.yaml in _chart-rules.md + both render-worker prompts). So promotion = write the git-tracked file (the git commit is itself a human-gated checkpoint); list_charts reads the directory live. reindex matters only if an ingested prose source lands in corpus/courses or corpus/proof.

10. Gap policy — held, data-first

A gap claim parks (held) at HG2 while its non-gap siblings flow through to produce. There is no ship-with-an-unbacked-number path. A gap resolves only by: (a) the chart-author derives the dataset+chart (→ data), or (b) the human rejects the chart and the claim stays held / the script is revised at authoring. A gap whose numbers cannot be grounded in any real source means the claim was never factual → it is dropped (a deeper data-first win).

11. Gap-fill — the /chart-gap utility (human-invoked, out-of-loop)

Not a pipeline stage and not auto-triggered. A human-invoked utility, like a maintenance command. The human brings the authoritative source; the agent only transcribes + structures it — zero fabrication.

content-writer flags gaps → gap-scripts HELD at HG2 → non-gap scripts flow on ──►

   ┄┄ later, human-paced, OUTSIDE the loop ┄┄
  YOU run:  /chart-gap   (+ source: PDF · Excel/CSV · image of a chart · text · URL)
            │
  chart-author subagent (Sonnet/Opus TBD — see open items):
     1. reads the gap flags → understands WHAT data is lacking (scenario + figures)
     2. ingests your source(s): parse PDF/Excel · read/OCR a chart image · fetch URL · read text
     3. persists a DATASET (JSON) first  → then derives N chart YAMLs from it
            (both stamped with honest provenance; image-derived = verification_status: unverified)
            │
   ──[ HUMAN reviews dataset + chart YAML(s) + rendered preview → approve / edit / reject ]──   (CLI / file-staging)
            │ approve
     promote: write corpus/data/datasets/<id>.json + corpus/data/charts/<id>.yaml  (git-tracked; NO reindex)
     → UPDATE the held Script's claim binding  (gap → data, chartRef set) · re-run C1 figures-trace
            │
     gap-script now DATA-bound → approvable at HG2 → CD → produce → ship ✅
  • New slash command /chart-gap backed by a new chart-author subagent. Tools: Read (PDF/image/text), WebFetch/WebSearch, Bash (Excel/CSV parse), Write (candidate dataset+chart), static-renderer (preview), store (read gap flags + update bindings). No corpus.reindex.
  • Ingest → dataset → chart: raw facts persist as a reusable JSON dataset; the chart YAML is derived from it and cites it (the provenance contract, §4).
  • Candidate staging: corpus/data/{datasets,charts}/_pending/ + a rendered preview; human approves via CLI confirm. Promotion moves files out of _pending/ and re-binds.
  • Honesty caveat: data read off a chart image is best-effort; verification_status records transcribed-from-figure-unverified; prefer tabular sources; the human is the backstop at review.

16. Risks

  • Coverage scan precision — too strict ⇒ noisy false-positive holds on incidental numbers; too loose ⇒ misses a claim. Mitigated by the allowlist + HG2 human backstop (flag, not hard-block).
  • Sonnet figure-tracing reliability — mitigated by deterministic C1 validator (the hard gate is code, not the model).
  • Image-transcription fidelity — mitigated by verification_status + human review at promotion.
  • Held backlog — gap scripts could accumulate if the human never runs /chart-gap; surfaced in content-writer notes + Brain's Decision Memo self-critique (like proofRatio).
