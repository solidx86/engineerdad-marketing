1. Context — why redesign now

The current review UI (apps/review-ui) shipped with the agentic rebuild (E-029). It's a Next 15 server-rendered tool at localhost:3030 with a left nav auto-generated from ENTITY_NAMES, a dashboard with per-entity counts, and a generic [entity]/[id] form that renders every non-skipped field as an editable <Field> input. It works. Three things are wrong with it.

  1. "Review UI" is too narrow a name. The app is the human surface for everything the OS does — gate approvals, run inspection, memo reading, asset preview, ops. Calling it the review UI was right when it had one job. The job has grown.
  2. The IA is entity-first; the work is run-first. Reviewers think "what happened in run X?" not "show me all CreativeVariants ever." The current nav forces an entity-first lens; the run only exists as a column on a table.
  3. The entity detail page is a flat field dump. A Brief's body (the artifact), its runId (provenance), and its linkedHypotheses (lineage) all render identically as labeled textareas. CreativeVariants has 45+ fields rendered in raw camelCase order. There is no information hierarchy, no read mode, no bilingual toggle, and no asset preview — even though every variant has an asset_files[] array of rendered creative.
The recentering. A run is the unit of marketing thinking: one Decision Memo → 12 Briefs → ~24 Scripts → ~40 Variants. A review happens within that envelope. Make the run the anchor. Entity lists still exist for cross-run browsing; they stop pretending to be the primary lens.

2. Goals and non-goals

Goals

  • Rename apps/review-ui → apps/webapp and the package @engineerdad/review-ui → @engineerdad/webapp, repo-wide.
  • Run-centric IA — new /runs + /runs/[runId] as the anchor surface; entity lists stay under /review/... for cross-run browsing.
  • Surface Brain's Decision Memos as readable documents — both in the Run detail page and on the PerformanceReports entity detail.
  • Per-entity detail layout system — read-by-default, bilingual EN/BM toggle, two-column hierarchy (primary content vs operational metadata), declarative config per entity.
  • Per-entity list config — CreativeVariants gets columns for channels + organic status with filterable chips; Hypotheses and Experiments get custom list configs too.
  • Creative asset preview — IG-style hero + dot navigation rendering of asset_files[], grouped by Script with aspect tabs under each format card.
  • Orchestrator run state — stage timeline by default; debug toggle for full step table + payload viewer.

Non-goals (explicit)

  • Authentication / authorization. Localhost tool. No login, no roles.
  • Run actions (cancel / restart / force-advance gate). Conductor-level surface; webapp stays read + approve only.
  • Live updates (polling, websockets, SSE). Page navigation re-fetches.
  • Bulk operations (approve N variants at once). Each row is a separate action.
  • Cross-entity search. Per-entity list filtering is in scope; full-text search across entities is not.
  • Inline field editing. Whole-form edit toggle per entity layout; per-field inline is heavier and not needed.
  • Cross-run comparison. Single Run detail page is the unit; comparing two runs is a future feature.
  • PDF / image export of memos. Markdown + copy-as-markdown button is enough.
  • Audit log of edits. Would need a new table; not needed now.
  • Mobile responsive design. Desktop-first. The webapp is a desk tool.
  • i18n beyond EN/BM toggles on bilingual fields. UI chrome stays English per ADR-010.
  • Theme switcher / dark mode. Light theme only.

4. Entity detail layout system

The biggest reshape. Today every entity detail page renders the same: a flat dump of every non-skipped field as an editable <Field>, labeled with the raw camelCase name. Briefs (12 fields), Scripts (15), CreativeVariants (45+), Learnings all render identically. Primary content (the brief body, the script, the memo) mixes with operational metadata (runId, createdBy, linkedHypotheses). Everything is in edit mode whether you're reading or writing.

The mental model

  • Read-by-default with an [Edit] button that flips the form to editable. Save commits via the existing saveRow action.
  • Two-column layout: primary content (the artifact) in the main column; operational metadata in a right rail.
  • EN/BM toggle in the page header. Defaults to EN. URL-persisted (?lang=bm). Only affects fields with role bilingual. Edit mode shows both languages so you can author the missing half.
  • Per-entity declarative layout in apps/webapp/src/app/lib/layouts/<entity>.ts. One file per entity, ~30 lines.
  • Status actions (Approve, Reject) live in the read-view header — one-click setStatus calls. No need to enter edit mode just to approve.

Layout config shape

type FieldSpec =

  | { role: 'bilingual'; en: string; bm: string; label?: string }
  | { role: 'fk'; field: string; fk: EntityName; label?: string };

Filter mechanics

  • URL-driven. Filters live in URL params (?channels=Meta-paid,IG-organic&approvalStatus=Awaiting+Approval) so links are shareable.
  • Filter chips at the top of the table. Click ✕ to remove one. "Clear all" resets.
  • AND across filters, ANY (OR) within multiSelect. So channels=Meta-paid,IG-organic means rows whose channels array contains either.
  • No runId filter on lists. Cross-run browsing is the lists' job; scoping by run is what /runs/[runId] does. Avoids two ways to do the same thing.

Channels array filter — implementation note

The current store filter DSL (packages/store/src/filters.ts) supports scalar equality and {in: [...]} for scalar columns, but not jsonb array containment. The channels column is jsonb string[], so "channels has any of [Meta-paid, IG-organic]" can't go through the DSL today. Decision: filter in JS after fetching. The list route applies the scalar filters at SQL, then runs Array.filter(row => row.channels.some(c => selected.includes(c))) in memory.

Why A, not B. Three tests: (1) Single consumer — only the webapp wants this filter today. (2) Volume — ~40 variants per run × ~20 runs ≈ 800 rows, ~3 KB each; Array.filter at this scale is microseconds. (3) API stability — adding containsAny to the DSL widens the public surface and invites the next operator (containsAll, jsonbEquals, …). Defer the DSL change until a second concrete consumer arrives. A 2-line code comment on the JS filter references this decision.

Data composition

The Run detail page is a server component that fans out these reads in parallel via Promise.all:

  1. orchestrator.runs WHERE run_id = ${id}
  2. orchestrator.run_steps WHERE run_id = ${id} ORDER BY updated_at
  3. store.query('Scripts', { runId: id })
  4. store.query('Briefs', { runId: id })
  5. store.query('CreativeVariants', { runId: id })
  6. store.query('PerformanceReports', { runId: id })

  8. store.query('Hypotheses', { runId: id }) (count + titles only)
  9. store.query('Experiments', { runId: id }) (count + titles only)

Server-component render. No client JS for the data path. Tabs, lightbox, language toggle, and the timeline/debug switch are the only client-interactivity boundaries.

SceneViewer component

apps/webapp/src/app/components/SceneViewer.tsx. Inputs: assetFiles: { url, sha256 }[], aspect: '4:5' | '1:1' | '9:16' | '16:9'.

No lightbox

The hero is already at the layout's maximum width. A lightbox to make it "bigger" would be ambiguous (fullscreen overlay? new tab?). The variant detail page (§4) renders the same SceneViewer in its Creative section — that's where original-pixel inspection lives.

Dev asset route

Background: creative_variants.asset_files[].url is file://… in local dev (asset-store writes to data/assets/<runId>/<variantId>/<scene>.<ext>) and https://… in prod (R2 with ASSET_STORE_PUBLIC_BASE set). The browser can't fetch file:// URLs from an <img src>, so dev needs a serving route.

Implementation: a Next route handler at apps/webapp/src/app/api/asset/[runId]/[variantId]/[scene]/route.ts reads from data/assets/<runId>/<variantId>/<scene> and streams the bytes with the correct Content-Type. Validates runId / variantId / scene against /^[a-zA-Z0-9_-]+$/ (same guard the asset-store MCP uses).

Rewrite helper: a small resolveAssetUrl(url) in apps/webapp/src/app/lib/assets.ts. If url.startsWith('file://'), parse the path, match the data/assets/<runId>/<variantId>/<scene>.<ext> shape, and return /api/asset/<runId>/<variantId>/<scene>.<ext>. Otherwise return as-is. Applied wherever a variant's assetFiles[] is rendered.

Tracker: Both the dev route and resolveAssetUrl() are workarounds for the asset-store local-disk path. E-007 (asset-store R2 backend swap) was bumped P3 → P2 on 2026-05-25 because this webapp work surfaces the second consumer pulling on it. Once E-007 ships, both pieces become dead code and get deleted.

export function webappUrl(): string {
  const fromOld = process.env.REVIEW_UI_URL;
  if (fromOld && !process.env.WEBAPP_URL) {
    console.warn("REVIEW_UI_URL is deprecated; set WEBAPP_URL instead.");
  }
  return (process.env.WEBAPP_URL ?? fromOld ?? "http://localhost:3030").replace(/\/+$/, "");
}

RiskMitigation
75 references across 16 files mean grep-and-replace will miss something subtle. The commit sequence above adds the new helper first (additive), then renames, then cleans up. Each step's failures are local and reversible. CI / pnpm -r build must pass at every commit.
