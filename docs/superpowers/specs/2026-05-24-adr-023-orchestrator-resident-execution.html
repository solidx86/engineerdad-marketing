1. Context — a second cap, a deeper pattern

E-031 (ADR-022) shipped Accepted 2026-05-24 and solved one specific class of bug: workers producing fan-out outputs too large for the conductor's inline tool-call channel. Workers now persist their final payload to orchestrator.step_results and emit a 50-byte {stepResultId} ref; the conductor carries refs verbatim; the orchestrator MCP dereferences server-side. The fix was verified end-to-end on run_1779611092 through HG2 — including a 62 KB P1-fanout unit that would have wedged the old inline contract.

A few hours after that landed, the same run continued past HG2 into the produce stage. P3-persist plans the materialization of N CreativeVariants — each carrying dual-language shotlists, ad copy, targeting JSON, organic captions. For four approved scripts × four creative formats, that's sixteen mcp__store__create calls, each ~14 KB of props. The orchestrator MCP's plan() response carried the full set inline — 238,164 bytes: 

Error: result (238,164 characters across 873 lines) exceeds maximum allowed tokens.
Tool: mcp__orchestrator__plan

The cap that bit was not new. The boundary that bit was new. E-031's claim-check applies at worker → conductor. This bug fires at orchestrator → conductor — the moment the conductor asks "what should I do next?" and the orchestrator has to ship the full step.calls[] embedding back. ADR-022's text does not cover this hop, but ADR-022's principle does. The conductor is the cap-sensitive transmission stop in both cases; we just discovered a second tributary feeding bulk through it.

We could file this as a sibling enhancement and mirror ADR-022's specific mechanism at a second boundary. We instead choose to name the underlying doctrine that ADR-022, ADR-018 (mechanical routing), and ADR-005 (thin-adapter doctrine) have all been instantiating — and apply it uniformly going forward.

Boundary B is the gap. The orchestrator builds step.calls[] with full props embedded; the conductor receives them as the answer to plan(); the harness caps the response. Once the response is over ~50 KB the conductor literally cannot see the work it is supposed to do.

3. Industry precedent — this is solved, repeatedly, with two names

A research pass (2026-05-24, see PROMPT §"Industry Research") established that this problem is named, documented, and solved across mature workflow engines. The shapes converge on two families:

3.1 — Claim-check at a new boundary (the ADR-022 lineage)

Apply ADR-022's pattern at boundary B too. Bulk plans become refs the conductor dereferences when it needs to read them.

SystemMechanism
Temporal — External Storage / Large Payload CodecBlobSizeLimitError at 2 MB. DataDog's Large Payload Codec swaps payloads for S3 refs at SDK layer — transparent to caller.
AWS Step Functions — larger state payloadsStates.DataLimitExceeded at 256 KB. Published workaround: state passes S3 keys, not payloads.
Argo Workflows — artifacts vs parametersIssue #2969 "Support large parameters". Solution: switch parameters → artifacts (S3 refs).
Google ADK — artifact handles"Tools accept the artifact identifier name instead of providing the full bytes data in the tools parameter."

3.2 — Direct Service Integration (the new family for us)

A stronger pattern: don't pass the bulk through the executor at all. The state machine invokes the underlying service itself; the executor never sees the work-order, only the result.

SystemMechanism
AWS Step Functions — Direct Service IntegrationsThe state machine invokes DynamoDB, SNS, SQS, Lambda itself, without serializing the payload back through a client. AWS explicitly recommends these over Lambda-wrapped invocations. Native handlers.
Temporal — ActivitiesThe workflow code is the orchestrator; activities run in worker processes but are dispatched by the workflow directly. The orchestrator owns dispatch and result-capture; nothing flows back through an intermediate executor.
Anthropic — multi-agent research systemLead agent "saves its plan to memory" and spawns subagents with only "an objective, output format, guidance on tools/sources, and clear task boundaries." Plan body never re-enters the lead agent's context.
LangGraph — checkpointer with reference stateExplicit doctrine: "A common mistake is storing large binary artifacts directly in the LangGraph state… store large files in specialized storage and keep only the reference URL or metadata."
LangChain — Plan-and-ExecutePlanner LLM has the full plan; executor LLM sees one step at a time. The split is explicit and named.
Arize's published comparison of orchestrator-worker frameworks: "the orchestrator must hold the full task description, all worker results, and enough context to synthesize… at four or more workers, context frequently exceeds window limits." The published response across all mature systems is identical: the LLM should be present at decision points, not at transport points. Mechanical fan-out of pre-decided writes is a transport point.

3.3 — What the research did NOT find

No published pattern recommends pushing bulk through an LLM's tool-call channel. No framework treats the conductor as a sound place to hold bulk. No mature system has "streaming the plan in chunks to the executor" as a documented technique — pagination over plan steps isn't a recognized pattern because step boundaries are semantic, not size-based.

4. The doctrine

ADR-023. The conductor is a reasoning surface — not a transmission surface, not an execution surface.

The conductor's job is to dispatch reasoning workers (spawn / fanout) and to honour human gates. Mechanical work that requires no reasoning belongs inside the orchestrator. Bulk that requires no inspection belongs on the side of the boundary that already holds it.

Concretely the conductor only ever talks to two surfaces: mcp__orchestrator__* (for state + work delegation) and Task (Claude Code's built-in for spawning subagent workers, which the orchestrator cannot invoke itself). Everything else — store, meta-ads, analytics, corpus, youtube, etc. — is reached by the orchestrator via in-process library imports, or by spawned workers via their own MCP access. Never by the conductor directly.

4.1 — What the doctrine generalizes

Prior ADRWhat it constrainedHow ADR-023 generalizes it
ADR-005 — thin-adapter doctrine + no-MCP-meshMCP servers are stdio wrappers over packages/; they don't open client connections to each other.The orchestrator inherits ADR-005's "don't cross MCP boundaries unnecessarily" by importing packages directly. The orchestrator-resident-execution path eliminates the need for refs entirely on the orchestrator → conductor hop — because the bulk never leaves the orchestrator in the first place.

4.2 — The three doctrines, rephrased as one

ADR-005 says: don't make MCP servers gossip with each other; route through libraries.
ADR-018 says: don't make the conductor decide mechanical things.
ADR-022 says: don't make the conductor carry bulk worker outputs.
ADR-023 says: don't make the conductor carry or do anything that isn't reasoning. Everything else lives on the durable side.

5. The three architectural shapes we considered

OptionMechanismIndustry analogVerdict
1 — Plan-ref Mirror ADR-022. When plan() envelope > 50 KB, orchestrator persists it to orchestrator.plans, returns {planRef}. Conductor fetches via read_plan. Temporal codec; Argo artifacts; ADK handles. Works. Same conductor role (transmitter). Adds a reference protocol the conductor must speak. defensible
2 — Streaming Range queries: conductor fetches plan steps incrementally. None recognized. No published precedent. drop
3 — Planner-executes Orchestrator runs its own write-step calls (by importing packages/). Conductor sees only the result summary. AWS Step Functions Direct Service Integrations; Temporal activities; Anthropic memory-handle. This. Removes the boundary; closes the class. chosen

5.1 — Why Option 3, and why total (Scope C) rather than partial (Scope B)

Option 1 patches transmission; Option 3 removes the need for it. The conductor adds no value as a transmitter of pre-decided mechanical calls — it cannot reason about store.create(...) in a useful way, cannot improve the args, cannot retry intelligently. It is shoe leather between two parties (orchestrator and store MCP) that already know each other.

We considered shipping Option 3 incrementally as Scope B — orchestrator self-executes only those write steps whose calls all target a packages/-resident library (today: packages/store only). That gets P3-persist out of the cap path immediately without prework. We are choosing Scope C instead — the full extraction of mcp-servers/<name>/src/ business logic into packages/<name>/ across the remaining six servers — for two reasons:

  • Doctrinal cleanliness. Under Scope B, the conductor's behaviour bifurcates: some write steps it doesn't see, some it executes. The mental model "the conductor sees decisions and only decisions" needs caveats. Scope C lets ADR-023 stand without exceptions.
  • The extractions are owed work anyway. ADR-005's thin-adapter ideal already calls for every MCP server to be a stdio wrapper over a packages/ library. Five of the six aren't (the meta-ads / analytics / corpus / youtube / heygen MCPs hold their business logic inside mcp-servers/<name>/src/). The store MCP is the lone exception that demonstrates the ideal. Scope C funds the alignment.

5.2 — Variant B2 (eager execution) over B1 (lazy)

Inside Scope C there is a choice about when the orchestrator runs the work:

  • B1 (lazy). plan() returns {kind:"write", stepId, executed:false, callCount:N}. Conductor explicitly calls execute_step(stepId). Two tool calls per write step.
  • B2 (eager). plan() for a self-executable write step executes inline and returns {kind:"write", stepId, executed:true, results:[...], stage, status}. The conductor never sees the work to do — only what was done.

We choose B2. The conductor has no information that would let it usefully decide to abort between plan and execute — verification is downstream, and the orchestrator's own verify-fold means the engine catches problems before the conductor sees them anyway. B1 keeps the conductor's role artificially active for no gain; B2 contracts the role to "ask what's next; observe what happened." Cleaner doctrine, simpler loop, fewer round-trips.

6.2 — Conductor's day, after ADR-023

loop {
  step = orchestrator.plan(runId)        // ONE tool call

  switch step.kind {

    case "write":                        // NEVER SEEN — orchestrator did it
    case "gate":                         // NEVER SEEN — orchestrator checked it
        // these never reach the conductor; plan() advances through them
        continue

    case "spawn":
        ref = Task(step.agent, step.spawnPrompt)
        orchestrator.advance(stepId, ref)
        continue

    case "fanout":
        refs = parallel( Task(unit.agent, unit.spawnPrompt) for unit in step.units )
        orchestrator.advance(stepId, refs)
        continue

    case "gate-stop":
        print(step.message)              // human-readable
        stop

    case "done":
        print(step.message)
        stop
  }
}

Compare to today's loop (see .claude/commands/loop.md): the write branch's substitution rules, the $<label> capture syntax, the per-call dispatch, the verify-then-advance two-step — all gone. The conductor's responsibility shrinks to "dispatch reasoning workers; pass refs; honour stops."

6.3 — Orchestrator's day, after ADR-023

The orchestrator MCP's plan() handler gains a loop that runs internally until it reaches a step the conductor needs to see: 

function plan(runId) {
  loop {
    step = engine.plan(runId, LIVE_REGISTRY)

    switch step.kind {

      case "write":
        results = executeWriteStep(step, deps)    // calls packages/* directly
        engine.advance(runId, step.id, results)   // verify folds in
        continue                                  // try next step

      case "gate":
        if step.check {
          checkResult = executeCheck(step.check, deps)  // packages/* read
          if checkResult.passes {
            engine.advance(runId, step.id, checkResult)
            continue                              // gate cleared, advance
          }
        }
        return makeGateStopStep(step)             // human must act

      case "spawn":
      case "fanout":
      case "gate-stop":
      case "done":
      case "halt":
        return step                               // conductor needs to see this

    }
  }
}

The engine, verifiers, and stage definitions are unchanged. executeWriteStep is new code in packages/orchestrator that walks step.calls[], dispatches each {tool, args} to the correct package's function (table-driven), and returns the array of results. The orchestrator MCP imports the new packages and injects them as deps.

7.2 — MCP envelope shapes

The orchestrator MCP's existing plan tool gets a new response shape for steps it executed eagerly. Existing verify + advance tools become vestigial for write / gate-check (the orchestrator already ran them); they continue to exist for spawn / fanout where the conductor's dispatch result needs verification.

Step kind seen by conductorplan() response
spawn{ kind:"spawn", stepId, agent, spawnPrompt } (as today)
fanout{ kind:"fanout", stepId, worker, units:[…] } (as today)
gate-stop{ kind:"gate-stop", stepId, message } (gate that requires a human; check already evaluated and failed inside plan)
done{ kind:"done", message }
halt{ kind:"halt", reason }

The conductor never sees kind:"write" or kind:"gate" (with check). Those are handled inside plan() and the loop spins forward until the next conductor-relevant step.

8.1 — Suggested extraction order

  1. analytics — most write-step traffic (A1, A2, A3, T2 partial). High payoff. Couples with E-030's eventual SQLite-→-Postgres migration: extract structure now so the migration is local to one package.
  2. corpus — small surface; mostly reads. Easy win; unblocks any future write step that needs compliance-block reads at orchestrator-resident time.
  3. meta-ads — second-largest surface. Required for any orchestrator-resident write that touches Meta (T1, T2 partial). HTTPS + PII normalization need careful porting.
  4. experiment — small, pure logic. Trivial.
  5. youtube — OAuth handling is the only tricky bit; defer until a write-step actually needs it from the orchestrator side.
  6. heygen — defer; no current orchestrator-resident write step uses it.

Order 1–4 covers every write step in LIVE_REGISTRY today. Orders 5–6 are pre-emptive and can wait.

8.2 — What stays in mcp-servers/<name>/

The MCP servers don't disappear. Outside callers — Claude Code's main session running /distribute, future scheduled cron jobs, Cowork — still reach these services via stdio MCP. The MCP server becomes the canonical ADR-005 thin adapter: ~30 lines that imports its package, registers tools, dispatches. The business logic moves; the protocol surface stays.

10. Resuming run_1779611092 under the new contract

The wedged run's state is preserved in SQLite (data/engineerdad.sqlite) and Postgres (orchestrator.step_results). Specifically:

  • 11 step_results rows for the run, all clean JSONB objects, covering S1-reason through P2-render (the largest 62 KB).
  • Run is at stage produce, status active, with P2-render completed and P3-persist as the next-due step.
  • 16 CreativeUnit specs are reconstructable from foldCreativePlan over the P1-fanout step_results refs (current code path).

Under ADR-023 + Variant B2: the user resumes by calling /loop run_1779611092. The conductor calls plan(). The orchestrator's plan() handler:

  1. Sees the next step is P3-persist (write).
  2. Builds the 16 calls + trailing query (same code as today).
  3. Invokes executeWriteStep, which dispatches each call through packages/store in-process. Compliance scanner runs per props.
  4. Captures 16 + 1 = 17 results; calls engine.advance; the engine rolls the stage forward.
  5. Sees the next step is HG3 (gate-stop, no automatic check; ADR-014 carve-out variants need human approval).
  6. Returns {kind:"gate-stop", message:"HG3 — review Variants..."} to the conductor.

The conductor's tape for this resume is one tool call. The 238 KB envelope never exists; the work happens where the data already lives.

11. Compatibility — what changes in existing ADRs

11.1 — ADR-005 — no rule change; explicit endorsement of library-import preference

ADR-005's no-MCP-mesh rule remains in force. ADR-023 makes explicit what was previously implicit: inside the workspace, callers should prefer library imports over MCP stdio calls. A status note appended to ADR-005 references ADR-023 as the load-bearing extension.

11.2 — ADR-018 — generalized

ADR-018's "distribution is a pure mechanical router" specializes naturally to "mechanical work is orchestrator-resident." A status note appended to ADR-018 references ADR-023 as the generalization; the original carve-out doesn't change.

11.3 — ADR-022 — narrowed scope, retained substance

ADR-022's claim-check pattern remains in force for worker output (boundary A). The plan-envelope problem (boundary B) is resolved by ADR-023's planner-executes mechanism, not by extending ADR-022 to a second boundary. A status note appended to ADR-022 cross-references ADR-023 as the umbrella doctrine and clarifies that claim-check still applies to worker → conductor specifically.

11.4 — loop.md — substantially trimmed

The current .claude/commands/loop.md body explaining the write branch (substitution rules, $<label> capture, per-call dispatch) becomes obsolete and is removed. The new doctrine reduces the doc by roughly 40%. A new section names ADR-023 as the authority.

  • E-030 — SQLite → Postgres unification. Independent axis. The analytics package extraction (§8.1 step 1) is structured to make E-030 local, but the migration itself is its own ADR.
  • Cross-MCP-mesh option. Not considered; rejected upfront on ADR-005 grounds.

14. Risks & mitigations

RiskMitigation
Package extraction breaks an existing MCP-server consumer (Claude Code main session, /distribute users).Each extraction PR: MCP server stays at the same protocol surface; tests of the MCP layer (mcp-servers/<name>/src/*.test.ts) must pass unchanged. Thin-adapter doctrine ensures the wire surface is preserved.
Compliance scanner is duplicated or skipped in the orchestrator-resident path.The scanner is imported as a library (packages/shared/compliance) and called inside executeWriteStep on every entity write. Tests assert scanner invocation count == entity-write count. PerformanceReports exemption (ADR-022) preserved as a single feature flag.
External HTTP calls inside the orchestrator MCP introduce new failure modes (timeouts, network errors at orchestrator-process level).HTTP clients retain their existing timeout / retry behaviour (moved verbatim from mcp-servers/<name>/ to packages/<name>/). Orchestrator's plan() handler catches and surfaces errors as {kind:"halt", reason} — same shape the engine already uses.
Larger orchestrator MCP process (more deps, more imports, more memory) becomes a contention point.Acceptable for v1.5 single-machine deployment. ADR-006 (future Agent SDK / Cowork migration) will revisit when the orchestrator deploys remotely.
Eager execution (B2) hides intermediate state from the conductor; debugging across a multi-step plan becomes harder.run_steps table still captures every step transition. The mcp__orchestrator__status + mcp__orchestrator__read_step_result tools remain the debugging surface — they don't get hidden, they just aren't called during the happy path.
Some write step uses a tool the orchestrator can't import yet (e.g., a future write step against an external service we haven't packaged).Fallback path: if executeWriteStep encounters a tool not in its deps dispatch table, it returns {kind:"write", executed:false, calls:[...]} as the legacy lazy shape. The conductor's old branch is retained for this fallback, but treated as a smell — every fallback firing files a follow-up to package the dep.
