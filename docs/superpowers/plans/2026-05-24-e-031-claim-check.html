Goal & architecture

Goal: Ship Claim-Check (Hohpe EIP) for the /loop conductor's transmission path so reshape during transmission becomes structurally impossible. Every worker returns a 50-byte {stepResultId} ref; the orchestrator MCP resolves refs to full payloads at the verify/advance boundary; the engine and verifiers see fully-materialized inputs exactly as today.

Architecture: New orchestrator schema in the existing Postgres container. New step_results table with JSONB payload + sr_-prefixed ULID IDs. Two new MCP tools (write_step_result, read_step_result). New resolve.ts helper at the MCP layer. Per-kind enforcement rejects inline for spawn / fanout. Four agent prompts + loop.md updated. Engine, state.ts, verifiers, stepResult<T>() consumers — all unchanged.

The win — default opaque, opt-in transparent.

The conductor physically cannot reshape what it doesn't hold. To reason over worker output (failure-aware retry, meta-orchestration, debugging) the conductor must take an explicit, auditable read_step_result tool call. Implicit reshape during transmission becomes structurally impossible; audited reasoning remains possible. This is exactly Google ADK's LoadArtifactsTool shape: materialize when needed, never by accident.
