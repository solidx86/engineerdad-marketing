1. Context — the bug the architecture forces

The closed-loop conductor (.claude/commands/loop.md) drives every /loop-once, /brief, /content, /produce, /distribute run. For fanout steps it dispatches N Task subagents in parallel, then must assemble each worker's final message into a single JSON result array and pass that array to mcp__orchestrator__verify and __advance. The conductor's contract is transmission middleman: read worker output verbatim, pass it through, never reshape.

.claude/commands/loop.md codifies the rule:

Do not summarise or reshape worker output before handing it to verify — the verifier is the judge, not you.

The architecture violates the rule in two distinct ways:

Architectural defect 1 — the harness 50 KB inline cap.

The Claude Code Task tool returns the subagent's final message as result. When that message exceeds ~50 KB, the harness persists the full content to ~/.claude/projects/.../tool-results/toolu_*.json and surfaces a 2 KB preview. The conductor literally cannot see the full output. Reproduced 2026-05-24 / run_1779557517 / P1-fanout: 4 creative-director workers returned ~13 KB each; 4 previews surfaced; the verbatim payloads were on disk, the conductor saw heads only.

Architectural defect 2 — hand-assembly reshape.

Even when payloads fit inline, the conductor is an LLM hand-typing JSON across N workers. It drops fields, reorders, truncates. Reproduced in:

  • run_1779549312 / C1-fanout (2026-05-23, the E-031 filing repro): first verify rejected 3/4 workers — workers shipped clean banks, the conductor lost them in transit. Second pass restored.
  • run_1779553270 / C1-fanout advance (2026-05-24): conductor dropped hooks from advance payload — produce stage's hookBanksFromC1 lookup returned empty.
  • run_1779553270 / P0-scripts advance (2026-05-24): conductor dropped brief field — produce stage's per-script brief lookup returned empty.

loop.md rule says don't reshape; the architecture forces reshape by transmission alone. The rule is honour-system; the architecture overrides it. Honest workers get rejected because the conductor loses data on the way to the verifier.

2. Research — this is not a new problem

This shape — large outputs in a multi-agent fanout, coordinator can't hold them all — is a 20-year-old enterprise integration pattern that the LLM/agent community has been re-deriving without naming. The name is Claim-Check (a.k.a. Reference-Based Messaging). Hohpe's EIP catalog, codified by Microsoft Azure: store the large payload in an external store, pass a small reference (a claim ticket) through the channel, the consumer dereferences at the boundary.

How established multi-agent frameworks handle it

FrameworkWhat they doVerdict for us
Anthropic Multi-Agent Research System Lead Researcher persists plan to "Memory" because 200k will be truncated. Subagents return summarized findings, not raw dumps. Our design is stricter — pure ref, no summary. Matches our "never reshape" doctrine exactly.
LangGraph Send() API Documented best practice verbatim: "Keep the Send payload small — send IDs and let each branch fetch its own data." Our design = canonical LangGraph fanout shape.
Google ADK Named primitive: artifact store + {name, summary} handle + LoadArtifactsTool for "ephemeral expansion." Closest published vocabulary. We adopt the structure and the framing.
CrewAI Shared crew memory + auto-injected Pydantic outputs. No ref primitive. Relies on outputs being small. Doesn't scale. Don't copy.
AutoGen / OpenAI Agents SDK tool_call_summary_formatter, nest_handoff_history: trim/compress on handoff. Wrong shape for us — they reshape. Our doctrine forbids reshape.

MCP spec primitive — researched and rejected

MCP Resources (uri, mimeType, size; 2026-07 RC adds ttlMs + cacheScope) are tempting — a server can publish addressable content the client reads. But Resources are designed for server→client exposure of content the server owns, not worker→orchestrator→verifier ephemeral run artifacts. Shoehorning is possible (URI scheme stepresult://run/{runId}/step/{stepId}) but stretches the spec's intent. Decision: opaque IDs in tool args, not MCP URIs. Simpler, identical effect, doesn't bend the spec.

Verdict: on the right track, but adopt the named pattern. We are not reinventing — we are correctly applying a 20-year-old EIP that every major multi-agent framework has converged on. This SPEC and ADR-022 can be the doc that names it for MCP-flavoured orchestration.

3. Goal

Make every fanout and spawn step transmit worker output without:

  1. Tripping the Claude Code harness's 50 KB inline cap.
  2. Allowing the conductor to drop, reorder, or truncate fields.

Definition of done:

  • The blocked run_1779557517 resumes from HG2 and clears P1-fanout end-to-end on the new shape.
  • Cold-start /loop-once walks all four worker stages (S1 brain, B1 brief-writer, C1 content-writer fanout + C2 content-writer article, P1 creative-director fanout) without any worker's final message exceeding 200 bytes.
  • orchestrator.step_results rows match expected fanout cardinality (1 from S1, 12 from B1, 4 from C1, 1 from C2, 4 from P1).
  • run_steps.result in SQLite still contains fully-materialized JSON after dereference — downstream stepResult<T>() reads in stage builders work unchanged.
  • The verifyProduce ground-truth check at P5 passes on the resolved payloads.
  • pnpm -r build + pnpm vitest run + pnpm sync:agents:check green.

4. Non-goals

  • Migrating runs + run_steps to Postgres. That's E-030. step_results in Postgres now anchors the migration; the rest follows in E-030's batch.
  • TTL / cleanup job for step_results. Filed as a follow-up (provisional E-033). v1 keeps rows for the run's lifetime + audit retention. Columns (created_at, size_bytes) ship so a sweep job can target rows later.
  • Reducing per-worker token cost. That's E-032.
  • Changing what workers produce. Hook banks, Scripts, Decision Memos, CreativePlans are unchanged in shape and content. Only the transmission primitive changes.
  • Render-worker (P2-render) integration. Audit during implementation; render-workers already return small payloads (asset URLs after upload to asset-store) — likely no change needed. Folded in only if audit shows otherwise.

5. Current shape (what's wrong)

Today's flow — fanout transmission

conductor (LLM) │ ├─► Task(content-writer, brief A) ◄── full final message (~13 KB JSON) ├─► Task(content-writer, brief B) ◄── full final message (~13 KB JSON) ├─► Task(content-writer, brief C) ◄── full final message (~13 KB JSON) └─► Task(content-writer, brief D) ◄── full final message (~13 KB JSON) │ ╔═════════════════════════════════════╧══════════════════════╗ ║ harness ~50 KB cap exceeded? ║ ║ • yes → persist to ~/.claude/.../toolu_*.json, ║ ║ return 2 KB preview to conductor ║ ║ • no → return full content inline ║ ╚═════════════════════════════════════╤══════════════════════╝ │ conductor hand-assembles array: ▼ result = [worker1Out, worker2Out, worker3Out, worker4Out] │ mcp__orchestrator__verify({ runId, stepId, result }) ───► engine.verify mcp__orchestrator__advance({ runId, stepId, result }) ──► engine.advance │ upsertStep(run_steps.result = JSON.stringify(result))

Two failure modes from this shape:

  • Harness-cap failure mode. If any worker returns > ~50 KB, the conductor sees only a preview. Hand-assembling the full array becomes impossible — the conductor would have to read each persisted file from disk, paste contents into a JSON literal, and pray nothing got mangled. We hit this on 2026-05-24.
  • Hand-typing failure mode. Even within the cap, the conductor is an LLM literally retyping 4×13 KB of JSON. Field drops, ordering changes, escaping bugs. We hit this twice in this session and once filed as the original E-031 repro.

6. Target shape — Claim-Check

The flow

conductor (LLM) │ ├─► Task(content-writer, brief A) │ │ worker authors content, │ │ writes Scripts to entity store, │ │ writes full output to step_results table via mcp__orchestrator__write_step_result │ ▼ │ return { stepResultId: "sr_01HZ..." } ◄── ~50 bytes, always under cap │ ├─► Task(content-writer, brief B) → { stepResultId: "sr_01HZ..." } ├─► Task(content-writer, brief C) → { stepResultId: "sr_01HZ..." } └─► Task(content-writer, brief D) → { stepResultId: "sr_01HZ..." } │ conductor wraps refs verbatim: ▼ result = [ { stepResultId: "sr_01HZ..." }, ◄── ~200 bytes total { stepResultId: "sr_01HZ..." }, { stepResultId: "sr_01HZ..." }, { stepResultId: "sr_01HZ..." }, ] │ mcp__orchestrator__verify({ runId, stepId, result }) │ │ at MCP boundary: resolveRefs(result) → array of full payloads │ ▼ engine.verify(runId, stepId, RESOLVED_RESULT) ── unchanged │ mcp__orchestrator__advance({ runId, stepId, result }) │ │ at MCP boundary: resolveRefs(result) → array of full payloads │ ▼ engine.advance(runId, stepId, RESOLVED_RESULT) ── unchanged │ upsertStep(run_steps.result = JSON.stringify(RESOLVED))
The win — default opaque, opt-in transparent.

The conductor physically cannot reshape what it doesn't hold. To reason over a worker's output (failure-aware retry, meta-orchestration, debugging), the conductor must take an explicit, auditable read_step_result tool call — visible in the tool-call log, unlike implicit reshape during transmission. Reshape during transmission becomes impossible by construction. Audited reasoning remains possible. This is exactly Google ADK's LoadArtifactsTool shape: materialize when needed, never by accident.

Design decisions, at a glance

DecisionChoiceWhy
Substrate Postgres, new orchestrator schema namespace Future-proof for E-030. JSONB suits the 50 KB–5 MB payload range. Transactional. Anchors the orchestrator-in-Postgres migration.
Worker write path New MCP tool: mcp__orchestrator__write_step_result Bypasses mcp__store__create's compliance scanner — step_results are intermediate output, not marketing copy. Keeps the entity-only schema clean (no rogue step_results entity). Owned by the same MCP that owns plan/verify/advance.
Conductor read path New MCP tool: mcp__orchestrator__read_step_result Opt-in transparency. Future conductor-as-reasoner work (retry analysis, meta-orchestration, human debugging) hooks in here without further architectural change.

7. Schema

-- packages/orchestrator/src/migrations/postgres/001_step_results.sql

CREATE SCHEMA IF NOT EXISTS orchestrator;

CREATE TABLE orchestrator.step_results (
  id          TEXT        PRIMARY KEY,         -- ULID with "sr_" prefix
  run_id      TEXT        NOT NULL,
  step_id     TEXT        NOT NULL,            -- e.g. "C1-fanout", "P1-fanout", "S1-reason"
  unit_index  INTEGER,                         -- NULL for spawn; 0..N-1 for fanout unit
  payload     JSONB       NOT NULL,
  size_bytes  INTEGER     NOT NULL,            -- octet_length(payload::text) at insert time, for audit / future TTL
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX step_results_run_step_idx
  ON orchestrator.step_results (run_id, step_id);

Why these columns

  • id as TEXT (not UUID) — we want the typed sr_ prefix to survive at SQL level for grep/audit.
  • unit_index nullable — natural shape: spawn (one worker) has no index; fanout (N workers) carries 0..N-1.
  • (run_id, step_id, unit_index) intentionally not unique — retries are valid (a flaky worker can be re-dispatched and write a new row; the conductor carries whichever ref the latest worker returned). Most-recent ref wins; older rows stay for audit.
  • size_bytes for audit and future TTL/sweep. v1 doesn't sweep.
  • JSONB not TEXT — query-shape support if we ever need to grep payloads for debugging.

New: mcp__orchestrator__write_step_result

export const WriteStepResultInputSchema = z.object({
  runId:      z.string().min(1),
  stepId:     z.string().min(1),
  unitIndex:  z.number().int().nonnegative().optional(),
  payload:    z.unknown(),
});

// Handler in index.ts:
server.tool(
  "write_step_result",
  "Persist a worker's final output to orchestrator.step_results. Returns { stepResultId }.",
  WriteStepResultInputSchema.shape,
  async (args) => {
    const id = await writeStepResult({
      runId: args.runId,
      stepId: args.stepId,
      unitIndex: args.unitIndex,
      payload: args.payload,
    });
    return { content: [{ type: "text", text: JSON.stringify({ stepResultId: id }) }] };
  },
);

New: mcp__orchestrator__read_step_result

export const ReadStepResultInputSchema = z.object({
  stepResultId: z.string().regex(/^sr_/),
});

// Handler — thin pass-through to loadPayload:
server.tool(
  "read_step_result",
  "Read a worker's persisted output by stepResultId. Auditable opt-in transparency.",
  ReadStepResultInputSchema.shape,
  async (args) => {
    const payload = await loadPayload(args.stepResultId);
    return { content: [{ type: "text", text: JSON.stringify({ payload }) }] };
  },
);

Changed: VerifyInputSchema / AdvanceInputSchema

const StepResultRef = z.object({ stepResultId: z.string().regex(/^sr_/) });

export const VerifyInputSchema = z.object({
  runId:  z.string().min(1),
  stepId: z.string().min(1),
  result: z.union([
    StepResultRef,                    // spawn: single ref
    z.array(StepResultRef),           // fanout: array of refs
    z.unknown(),                      // write/gate: conductor-assembled MCP-call results (inline)
  ]),
});

// AdvanceInputSchema mirrors VerifyInputSchema.

Per-kind enforcement happens at the handler: the handler looks up the step's StepSpec.kind in the registry and rejects an inline payload for spawn/fanout, rejects refs for write/gate. The Zod schema is permissive at the type level; the handler is strict at the contract level.

Resolution helper

// mcp-servers/orchestrator/src/resolve.ts (NEW)

import { loadPayload } from "@engineerdad/orchestrator/postgres";

const isRef = (x: unknown): x is { stepResultId: string } =>
  typeof x === "object" && x !== null
    && "stepResultId" in x
    && typeof (x as { stepResultId: unknown }).stepResultId === "string"
    && (x as { stepResultId: string }).stepResultId.startsWith("sr_");

export async function resolveRefs(result: unknown): Promise<unknown> {
  if (isRef(result)) return loadPayload(result.stepResultId);          // spawn
  if (Array.isArray(result) && result.every(isRef)) {                  // fanout
    return Promise.all(result.map((r) => loadPayload(r.stepResultId)));
  }
  return result;                                                       // write/gate: pass through
}

Verify / advance handlers — wired

// mcp-servers/orchestrator/src/index.ts

server.tool("verify", "...", VerifyInputSchema.shape, async (args) => {
  const coerced = coerceResult(args.result);
  const resolved = await resolveRefs(coerced);
  const verifyOut = engine.verify(args.runId, args.stepId, resolved, LIVE_REGISTRY);
  return { content: [{ type: "text", text: JSON.stringify(verifyOut) }] };
});

server.tool("advance", "...", AdvanceInputSchema.shape, async (args) => {
  const coerced = coerceResult(args.result);
  const resolved = await resolveRefs(coerced);
  const adv = engine.advance(args.runId, args.stepId, resolved, LIVE_REGISTRY);
  return { content: [{ type: "text", text: JSON.stringify(adv) }] };
});

The rule update makes that distinction explicit.

12. Future capability this unlocks (conductor-as-reasoner)

The conductor today is purely a transmission middleman. But the moment we ship read_step_result, a more capable conductor becomes feasible without further refactoring. The ref itself is just an opaque ID — anyone holding it can dereference. This unlocks:

  • Failure-aware retry. When verify returns {ok: false, problems: [...]}, a smarter conductor can READ the worker's payload, diagnose the gap, and decide retry vs structural-stop with real evidence. Today's conductor flips a coin.
  • Meta-orchestration. A future stage could spawn N workers, READ their refs, choose the best output (or merge top-K), and only then advance. The current architecture doesn't preclude this — it just requires the conductor to explicitly opt into materialization.
  • Cross-stage reasoning. /reflect already grades prior-cycle Hypotheses against PerformanceReports. With READ, brain can also inspect the actual step_results from a prior content-stage run when calibrating its self-critique.
  • Human debugging. A reviewer pastes a stepResultId into the review UI or a CLI and sees the exact bytes a worker emitted — no log diving, no harness-cap workaround.
In scope today: the read_step_result tool ships now so these capabilities don't require a second contract change. The doctrine in loop.md is explicit that today's conductor must not call READ during normal transmission — it's reserved for the cases above, where the act of reading is itself the work.

14. Risks & mitigations

A worker fails to write step_result but still returns a ref-shaped object (e.g. fabricated ID) loadPayload on a missing ID throws a typed error; the MCP verify handler surfaces it as { ok: false, problems: ["stepResultId not found: sr_..."] }. Conductor retries the same step once per loop.md rules. Contract test covers this.
Postgres unreachable when worker tries to persist Worker's write_step_result call fails before the worker can return a final message. Worker errors out; orchestrator never sees a stale ref. Same failure semantics as a worker that errors today.

  • Migrating runs + run_steps from SQLite to Postgres — E-030.

16. Verification

  1. Unit tests: pnpm --filter @engineerdad/orchestrator test, pnpm --filter @engineerdad/mcp-orchestrator test. Cover resolve / pass-through / missing-ref / fanout array / per-kind enforcement.
  2. Migration: pnpm orchestrator:push applies 001_step_results.sql. Verify with docker exec engineerdad-postgres psql -U engineerdad -d engineerdad -c "\dt orchestrator.*".
  3. Sync agents: pnpm sync:agents. Confirm the 4 worker prompts pick up the new "write step_result before returning" instruction.
  4. Live cold-start walk: wipe DB, run /loop-once end-to-end through HG1 → /content → HG2 → /produce → HG3. Confirm:
    • Every worker's final message ≤ 200 bytes (final JSON = {stepResultId}).
    • orchestrator.step_results rows: 1 from S1, 12 from B1, 4 from C1, 1 from C2, 4 from P1.
    • run_steps.result in SQLite still contains fully-materialized JSON (downstream stepResult<T>() reads work unchanged).
    • P3-persist's verifyProduce ground-truth check passes — proving the resolved payload feeds the verifier correctly.
    • No harness "Output too large" persist-to-disk events during the run.
  5. Doctrine alignment: manually attempt to pass an inline payload to mcp__orchestrator__verify for a fanout step — confirm the handler rejects with a typed error.
