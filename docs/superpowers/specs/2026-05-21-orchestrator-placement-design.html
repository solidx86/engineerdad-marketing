Orchestrator Placement — Corrected Wiring for the EngineerDad Marketing OS

Design spec  ·  2026-05-21  ·  DRAFT — FOR REVIEW
Supersedes the dispatch model in brain.md §A/§E  ·  Generalises TASKS.md E-021  ·  Origin: brainstorming session, this conversation

The OS was designed as a 4-tier hierarchy — command → brain → leaf agents → workers. The Claude Code runtime supports only 2 tiers: main session → subagents. brain sits at tier 1, one level too deep to dispatch anything — so 11 of 15 commands and 5 of 8 agents are currently unreachable. The fix: move orchestration into the command layer (the main session). brain becomes a pure reasoning leaf; commands spawn leaves and fan-out workers directly. One rule for the whole system — commands orchestrate, everything else is a terminal leaf.

1. The problem

The OS is built from three layers on disk:

LayerCountWhat it isRuns at
.claude/commands/*.md15Slash-command entry points — currently 2-line stubsMain session (level 0)
.claude/agents/*.md8brain + 7 leaf subagentsSubagent (level 1) when spawned
.mcp.json13 serversThe MCP tool layer leaves callstdio processes

The intended topology, per brain.md §A (mode dispatch) and §E (spawn protocols), is a four-tier hierarchy: 

command (thin launcher)  →  brain (orchestrator)  →  7 leaf agents  →  workers

Every command except four does the identical thing — Task(subagent_type=brain). So the real call chain is: 

Main session (level 0)
  └─ Task → brain          (level 1 — brain IS a subagent)
       └─ Task → targeting / content-gen / media-production / ...
                            (level 2 — never executes)

2. The constraint (verified empirically, 2026-05-21)

Hard runtime constraint A subagent cannot spawn a subagent. The spawn tree is capped at exactly two levels: main session → subagent. There is no level 3.

This was tested directly in the brainstorming session: a general-purpose subagent was spawned with Tools: * (the most permissive allowlist possible) and instructed to spawn a child. Its actual runtime toolset came back as Bash, Edit, Read, ShareOnboardingGuide, Skill, ToolSearch, Write — no Task, no Agent. The runtime strips spawn capability from every subagent regardless of frontmatter. This independently confirms TASKS.md E-021's finding ("brain could not spawn distribution").

The whole commands+agents structure was designed before this constraint was known — the design is sound on paper; only the runtime placement is wrong.

Consequence — what runs today

PatternCommandsStatus
Command → leaf agent (direct)/pull-meta, /tracking-test✓ runs
Command runs inline (no subagent)/ingest-corpus, /posting-pack✓ runs
Command → brain → leaf/loop-once, /brief, /brain, /content, /produce, /experiment, /distribute, /analyze, /reflect, /targeting, /post-week (11)✗ dies at hop 2

5 of 8 agents are unreachable — targeting, content-gen, media-production, experiment-os, distribution. No command spawns them directly; only brain is meant to, and brain can't. brain works only as a reasoner (modes that read + write its own DBs); the instant a mode must dispatch, it dies.

3. The decision

Chosen — Approach 2: Brain becomes a reasoning leaf Orchestration moves into the command layer (main session). brain stays a subagent but only reasons — it runs the §12.6 nine-step scaffold and returns a Decision Memo. It dispatches nothing. Each command becomes the orchestrator: it spawns brain for the Memo, then spawns leaf agents and fan-out workers directly.

Two alternatives were considered and rejected:

ApproachWhy not
1 — Command IS the orchestrator
(brain's scaffold pasted into command bodies; brain.md deleted)		 Correct, but the §12.6 reasoning scaffold — the intellectual core of the OS — gets duplicated across 11 command files and rots. Loses brain as one coherent, testable unit.
3 — Minimal per-command fix
(each command spawns its leaf directly; no shared orchestrator)		 Ships fastest, but routing and multi-step sequencing scatter across 15 files with no shared model, and brain's scaffold is left stranded. Under-designed.

Approach 2 is the only option that is both correct and keeps the system coherent. It also matches a decision the project already made: E-021 specifies that media-production should become a spec-returning subagent that dispatches nothing. Approach 2 applies the same rule to every agent — yielding one uniform model for the whole OS.

6. Worker fan-out — the original goal

The brainstorm started from "make the main agents efficient by fanning out workers on demand." Under the broken topology that was impossible — a leaf agent cannot spawn. Under Approach 2 it falls out for free: the command is the main session, and the main session can spawn workers.

  • /produce — fans out one render worker per static Variant. This is E-021, now an instance of the general pattern rather than a special case.
  • /content (future) — content-gen shrinks to a planner that returns a per-Brief spec; the command fans out one authoring worker per Brief (30+ hooks + 3 scripts each). The single strongest fan-out candidate identified in the assessment.
  • /brief (future) — same shape for targeting's 12 Briefs, if measurement justifies it.

The reason for fan-out is context-budget isolation (per ADR-014), not raw speed: each worker's heavy authoring/rendering lives in its own context, so the orchestrator stays small across N units.

7. Human gates — unaffected

HG1–HG4 are Notion approval states, not code branches. The orchestrator simply runs its spawn sequence and stops — returns control to the human — at the gate. Moving the orchestrator from a brain subagent up to the command layer does not change the gates; it makes them cleaner, because a command in the main session naturally ends the turn and hands back to the human, whereas a subagent returning is not the same as the loop halting.

8. Open decisions (for the spec → plan cycle)

  1. Shared orchestration code. Do the 11 rewritten commands duplicate their spawn-sequence logic, or share it via an include directive (the repo already uses <!-- include: --> + pnpm sync:agents for prompt fragments)? Eleven near-identical orchestrators is a maintenance smell.
  2. How brain receives analytics signal. The command spawns analytics before brain. Does it pass the analytics JSON into brain's spawn prompt, or does brain re-read the freshly-populated SQLite via its own read tools? (Brain currently holds mcp__analytics__* read tools — both work.)
  3. Fate of brain.md §G organic-planner. Move wholesale into the /post-week command body, or keep as a small leaf the command spawns?
  4. E-021's five sub-decisions carry forward: media-production spec shape; where brand-contract reasoning lives; who writes the Step 6 Notion rows; fate of corpus/templates/worker-prompts/static-asset.md + scripts/test-static-worker.mjs; cleanup of the Step 5.5 / G11 / 7.0-B language.
  5. ADR. This supersedes the dispatch model in ADR-005 (MCP + agent architecture) and resolves ADR-014's unrunnable worker split. New ADR, or amendments to both?
  6. Compliance-scan boundary. Confirm that with workers returning drafts and the command writing to Notion, the per-DB compliance scan at the notion MCP layer still fires on every page write (it should — the scan is MCP-side, not agent-side).

9. Scope & non-goals

  • In scope: relocating the orchestrator; trimming brain.md to a reasoning leaf; the command-rewrite pattern; the worker fan-out mechanism and its first instance (/produce).
  • Not in scope: changing the closed-loop sequence itself (Tracking → Analytics → Brain → Targeting → … is unchanged); changing any MCP server; changing the Notion schema; the content-gen / targeting fan-out retrofits (follow-on work, noted in §6).
  • Risk: command files grow from 2 lines to substantial orchestrators. Mitigated by Open Decision 1 (shared include). This is unavoidable in any correct fix — the orchestrator must live somewhere in the main session.
