Reel Quantitative-Visual Exemplar Catalogue

Design spec · 2026-05-31 · branch feat/reel-visual-scenes · status: approved, ready to plan

Problem

The reel data-visual lane (a visual scene with a chartRef) has been the source of repeated rendering-quality failures. We just retired two chart reference-designs that embodied the very defects we were fixing — a callout floated over the plot, the built-in Chart.js circle legend crammed inside the canvas — because the worker prompt orders workers to imitate the exemplar, and a picture beats prose.

The lesson: determinism for these frames lives in an approved exemplar, not in prose rules. But we have no exemplars yet for most quantitative layouts, and only one chart type (line) has ever been rendered through the test harness. We want a curated catalogue: one fixture per quantitative variation, the worker explores a layout, a human reviews every render, and the approved render is promoted into corpus/templates/reference-designs/ as a true exemplar that future renders pattern off.

Current state (as built)

  • Three reel fixtures: data-visual.json, concept-visual.json, mixed-reel.json. Only the line chart (compounding-30y) has been exercised.
  • chartjs-config.js supports line | bar only.
  • reel-render-worker.md Step 2 data-visual branch hard-codes buildChartConfig(...) — it assumes a Chart.js chart, no other presentation.
  • Frames-only mode passes the authored HTML inline to render_html_to_png (render.ts:161). The renderer writes only the PNG (render.ts:176); the HTML is never persisted. Disk confirms: test-reel dirs hold only .png files.
  • face scenes render no frame (worker prompt line 120).

Goals

  1. A catalogue of reel fixtures, one per quantitative visual variation, each driving a single reviewable render.
  2. Promotion path: an approved render becomes a reference-design exemplar carrying its source markup (.html + .png siblings).
  3. Persist the worker's authored HTML beside the PNG so promotion can copy both.
  4. Drop the redundant mixed-reel fixture (it is just data + concept concatenated).

Non-goals

  • No schema / data-model change. chartRef stays the only quantitative data lane; numbers + source_citation always come from a corpus chart YAML (compliant, vetted).
  • No chartjs-config.js extension yet. Pie/donut are hand-authored during exploration; an approved winner is folded into the config afterward (separate follow-up).
  • No promote script. Promotion is manual: cp the PNG + HTML, add one README row.
  • No .md sidecars. The README table is the single curation-note surface.
  • Concept-visual lane is unchanged (already has two clean exemplars).
  • Static-worker harness (test:worker) is out of scope.

Fixture shape

A catalogue fixture is a single visual scene, no face scene — face scenes render no frame in frames-only mode, so they are dead weight for exemplar generation. scene.explains carries the layout nudge; numbers and citation come verbatim from the YAML.

Worker-prompt changes (reel-render-worker.md)

1. Presentation latitude (relax Step 2 data-visual branch)

Today the data-visual branch hard-codes buildChartConfig. Relax it: given a chartRef, the worker chooses the presentation that best serves scene.explains —

  • Chart (line / bar via buildChartConfig; pie/donut hand-authored until codified) — HTML legend above the canvas, callout as a row outside the plot (unchanged §5 rules).
  • Comparison table — pure HTML; rows = entities, columns = metrics; emphasize the contrast cell.
  • Stat-callout grid — pure HTML; 2–4 headline numbers in large type, one per cell.

All presentations: numbers + full source_citation verbatim from the YAML (never invented/recomputed), and obey brand-contract §4a (legibility), §8 (density), §9 (self-critique). Legend/§5-chart rules apply only to chart presentations.

2. Persist the authored HTML beside the PNG

After each render_html_to_png call, the worker Writes the exact HTML it authored to the same directory as the PNG, as <scene>.html — i.e. data/assets/<runId>/<variantId>/<scene>.html beside <scene>.png. The render-worker already has the Write tool. This is the single source of the markup needed for promotion; it works in the normal path too, not just the harness.

3. Read exemplar HTML, not just the PNG

Update the "Reference designs" section so the worker Reads the exemplar's .html (the transferable source) in addition to viewing the .png, when an .html sibling exists.

Promotion workflow (manual)

  1. pnpm test:reel-frames <fixture> → spawn the worker → review the PNG.
  2. On approval, with <vid> = variantId, <n> = scene index: 
    cp data/assets/test-reel-<fixture>/<vid>/<n>.png \
   corpus/templates/reference-designs/reel-9x16-<variation>-NNN.png
cp data/assets/test-reel-<fixture>/<vid>/<n>.html \
   corpus/templates/reference-designs/reel-9x16-<variation>-NNN.html
  3. Add one row to the README "Reel exemplars" table (tone, what it demonstrates).
  4. If it's a chart frame, also point the worker prompt's reference-designs list at the new exemplar (keep ≤3 per bucket — the README's existing rule).

README convention update

  • Drop .md from the sibling convention: exemplars are .html + .png only.
  • State that the worker reads the exemplar .html as the primary pattern, the .png as the visual check.
  • Existing .md sidecars can be cleaned up later — out of scope here.

chartjs-config deferral

Pie/donut are rendered by a worker-authored Chart.js config inline during exploration. Once a pie/donut exemplar is approved, fold a pie/doughnut branch into buildChartConfig as a tracked follow-up (TASKS.md). Comparison-table and stat-grid never touch Chart.js.
