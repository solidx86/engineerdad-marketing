DB Migration Policy & Branch Sandbox Isolation

Date: 2026-05-26 ADR: 027 Accepted Builds on ADR-025

1. Context & Footguns

ADR-025 consolidated the Marketing OS onto a single Postgres substrate. That was the right call, but it left three footguns unfixed:

  1. Root db:push targets the live DB. It runs drizzle-kit push against engineerdad (and engineerdad_test) with no branch isolation. Destructive schema ops surface as CLI prompts, not code-level gates.
  2. No branch isolation. All branch dev work and the live record share the same engineerdad DB. Schema divergence between branches is invisible until conflict.
  3. truncatePg() has no DB safety check. It reads process.env.DATABASE_URL with no guard. If the shell has DATABASE_URL pointing at engineerdad, every beforeEach in the 85-file vitest suite truncates the live DB. This has already happened.
Live DB status: engineerdad was wiped by a test run in the prior session. Decision: accept the loss and move on. corpus/ is the source of truth; the live DB is rebuilt by running /loop cold. This policy prevents recurrence.

2. Decisions

2.1 Branch Sandbox DBs

Each working branch gets its own Postgres database named engineerdad_sb_<slug>, where <slug> is derived from the git branch name (lowercase, / and non-alphanumeric replaced with _, truncated at 40 chars).

pnpm db:sandbox — the setup command:

  1. Reads current branch via git symbolic-ref --short HEAD.
  2. Derives and sanitizes the DB name.
  3. Creates the database if it doesn't exist (idempotent).
  4. Runs drizzle-kit push for all three packages against it.
  5. Writes DATABASE_URL=postgresql://engineerdad:engineerdad@localhost:5432/engineerdad_sb_<slug> to .env.local at the repo root.

Env precedence: Vitest (built on Vite) loads .env then .env.local, with .env.local winning. So tests automatically pick up the sandbox URL. Explicit shell exports and CI environment variables override both.

.env keeps its stable fallback DATABASE_URL=...engineerdad. .env.local holds only the sandbox override and is gitignored.

pnpm db:sandbox:drop — hygiene command:

Queries Postgres for all engineerdad_sb_% databases, checks which corresponding branches still exist locally, lists orphans, and drops them after a y/n prompt. Run occasionally; no automation, no risk of accidental drops.

2.2 Versioned Migrations

Hybrid model: drizzle-kit push is retained for sandbox dev (fast iteration, no SQL files needed mid-branch). Versioned SQL is required at commit time when schema.ts changes.

Per-package migration folders: each package gets its own drizzle/ output directory (drizzle-kit's default). No unified folder — the schemas are cleanly separated by Postgres schema (public, orchestrator, analytics) and packages own their migrations independently.

New root commands:

CommandWhat it doesWhen to run
pnpm db:sandboxCreate/update branch sandbox via push; write .env.localOnce per branch, re-run after schema changes during dev
pnpm db:generatedrizzle-kit generate for all three packages → SQL in packages/*/drizzle/Before committing a schema.ts change
pnpm db:migratedrizzle-kit migrate for all three packages against DATABASE_URLAfter merging to main, against live engineerdad DB
pnpm db:sandbox:dropList and drop orphaned sandbox DBsOccasional hygiene

Root db:push is retired. The individual package push scripts ("push": "drizzle-kit push" in each package's package.json) remain as internal primitives called by db:sandbox.

Dev workflow for a schema change:

  1. Edit schema.ts in the relevant package.
  2. pnpm db:sandbox — push to branch sandbox immediately.
  3. pnpm db:generate — produce the SQL migration file.
  4. Commit schema.ts + generated SQL together.

CI lint rule: if any packages/*/src/schema.ts is modified in a commit, the corresponding packages/*/drizzle/*.sql must also be present. Implemented as a lint script (not a git hook — hooks are not portable across Claude Code sessions).

Post-merge to main: run pnpm db:migrate manually against the live DB. Single-dev laptop operation; a human gate here is appropriate.

2.3 truncatePg() Safety Guard

The safety check moves from inside client() to module load — it throws the moment the file is imported, before any test runs.

// packages/shared/src/test-helpers/truncate-pg.ts (module load, top of file)
const _url = process.env.DATABASE_URL ?? "";
const _safe = /_test$/.test(_url) || /engineerdad_sb_/.test(_url);
if (!_safe) {
  throw new Error(
    `truncatePg() refuses to run against '${_url || "(unset)"}'. ` +
    `DATABASE_URL must end with _test or contain engineerdad_sb_. ` +
    `Run \`pnpm db:sandbox\` to set up your branch sandbox.`
  );
}

The existing DATABASE_URL null-check inside client() is retained as a belt-and-suspenders fallback, but the module-load guard catches misconfigurations before any pool is opened.

2.4 CLAUDE.md — Operational Rules

A new "Database workflow" section is added to CLAUDE.md immediately after the existing build/dev commands table. These are the rules Claude Code must follow:

  • Never run the root db:push script — it is retired and must not be re-added. It targets the live DB directly.
  • Before running tests, ensure DATABASE_URL in .env.local points at an _test or engineerdad_sb_* DB. truncatePg() hard-stops the suite if it doesn't.
  • pnpm db:sandbox — run once per branch. Re-run after any schema.ts change during dev.
  • Schema change on a branch — edit schema.ts, run pnpm db:sandbox, then pnpm db:generate before committing. Generated SQL in packages/*/drizzle/ must be committed alongside schema.ts.
  • After merging to main — run pnpm db:migrate against the live DB once.
  • pnpm db:sandbox:drop — run occasionally to clean up orphaned sandbox DBs.

3. ADR-027 Summary

Title: 027-db-migration-policy-and-sandbox-isolation
Supersedes: the migration mechanism described in ADR-025 (which deferred this question).
Tracker: E-035 (to be created)

Consequences:

  • Tests require pnpm db:sandbox on first branch use. Suite hard-stops if DATABASE_URL is wrong — intentional.
  • Schema changes require two commands before commit (db:sandbox + db:generate). Small overhead, large safety gain.
  • Live DB recovery after a wipe: pnpm db:migrate on a fresh engineerdad DB, then cold /loop to regenerate store entities from corpus/.
  • No SQLite regression — ADR-025 is stable, this builds on it.
  • Two branches can run /loop simultaneously against their own sandbox DBs without run_id collision.

4. Out of Scope

  • Backups and durability — separate concern.
  • Remote staging environment — single-machine ops; sandbox isolation is the substitute.
  • Automated migration on merge — human gate is the right call for a one-person operation.
