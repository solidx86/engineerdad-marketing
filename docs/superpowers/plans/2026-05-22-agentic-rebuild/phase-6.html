This is the cutover phase Phases 2–5 built nine stage definitions behind a fixture-only registry (the C1 staging). Phase 6 builds the three remaining stages, assembles the ordered live registry, and points the orchestrator MCP at it — after this phase /loop walks the real 9-stage loop end to end. It also deletes the five agents the stages replaced. The integration risk is real and concentrated here; T10 is a full live smoke.
Three decisions baked in 1 — synthesize / brief / content are agentic-cell stages. Each is a thin spawn of a genuine reasoning/authoring agent (brain, brief-writer, content-writer) plus — for brief and content — a human gate. The spawn-prompt formats are the ones relocated from brain.md §E, now typed in stage files. The verifiers are light: they catch a totally-failed agent run; the human gate is the real judgement.
2 — content is spawn + gate (no fanout yet). The current content-gen agent authors hook banks, scripts, and articles in one pass. Design §8's "content-writer + fan-out workers" is a later refinement; Phase 6 ships content faithful to the agent as it stands.
3 — the loop order is design §8. tracking → analytics → synthesize → brief →[HG1]→ content →[HG2]→ produce →[HG3]→ schedule → experiment → distribute →[HG4]. schedule precedes experiment + distribute (Phase 4 review).

Goal: Build synthesize, brief, content as agentic-cell stages; assemble the ordered live registry; point the orchestrator MCP at it; upgrade HG4 to a passable Meta check; delete the five replaced agents — so /loop conducts the whole loop.

Architecture: No engine change. The three new stages are spawn(+gate) StageDefinitions built against Phase 2's frozen engine. The registry swap is a one-line change in the orchestrator MCP.

Tech stack: TypeScript (ESM), vitest. New deps: none. Execution: TBD — this doc is for review first.

1. Locked interfaces

1.1 — RunStage (additive)

export type RunStage =
  "fixture" | "tracking" | "analytics" | "synthesize" | "brief"
  | "content" | "produce" | "schedule" | "experiment" | "distribute" | "done";

1.2 — Stage shapes

synthesizeStage  "synthesize"  S1-reason (spawn brain — "run §B 9-step, emit the
                               Decision Memo, do NOT dispatch"; verify =
                               verifySynthesize — Memo has recommendedAngles)
                               — no gate; flows straight into brief.

briefStage       "brief"       B1-write (spawn brief-writer — build passes runId
                               + the Memo's recommendedAngles/personas/
                               topCreatives/hypothesisIds; verify = verifyBrief)
                               B2-gate  (gate HG1 — check: Briefs Approved query)

contentStage     "content"     C1-write (spawn content-writer — build passes
                               runId; verify = verifyContent)
                               C2-gate  (gate HG2 — check: Scripts Approved query)

Each spawn build reads the prior stage's step result to assemble its prompt — the same pattern as distribute's D2. The spawn-prompt formats are brain.md §E's, now typed.

1.3 — Verifiers (light, structural)

verifySynthesize(memo)   ok when the spawn result carries a non-empty
                         recommendedAngles[] (brain produced a usable Memo).
verifyBrief(result)      ok when result.angles[] is non-empty (brief-writer
                         created Brief rows). Cold-start tolerant.
verifyContent(result)    ok when result.scripts[] is non-empty.

These catch an agent run that produced nothing. The acceptance judgement is the human gate (HG1 / HG2) — a light verifier here, a real gate after.

1.4 — HG4 upgrade — distribute D4-gate

D4-gate gains a `check` (it was a terminal STOP in Phase 3):
  check: { tool: "mcp__meta-ads__list_ads", args: { runId } }
  verify: gate clears when every ad for the run is ACTIVE — i.e. a human has
          un-paused them in Meta Ads Manager (ADR-015: activation is human-only).

1.5 — LIVE_REGISTRY (registry.ts)

export const LIVE_REGISTRY: StageDefinition[] = [
  trackingStage, analyticsStage, synthesizeStage, briefStage, contentStage,
  produceStage, scheduleStage, experimentStage, distributeStage,
];

The orchestrator MCP imports LIVE_REGISTRY in place of FIXTURE_REGISTRY; FIXTURE_REGISTRY + fixtureStage stay exported for the engine unit tests.
