Agentic Rebuild — Cells, the Orchestrator State Machine, and the /loop Model

Design spec  ·  2026-05-22  ·  REVIEWED — DECISIONS LOCKED
Extends 2026-05-21-orchestrator-placement-design.html  ·  Answers its Open Decisions 1 & 4  ·  Touches ADR-005 (thin-adapter), ADR-014, ADR-018  ·  Origin: design conversation, this session

The OS is agent-first — every component is an agent — but barely agentic — most agents execute fixed scripts. Those are different things. Agent-first is a topology; agentic is a behavioral property: genuine latitude, grounded by deterministic tools and a deterministic verifier. The rebuild: collapse the deterministic "agents" into typed code, keep ~4 genuinely agentic cells, and stand them on a typed run-state machine (the orchestrator MCP) that the /loop command conducts. Fewer agents — each one actually agentic. The deterministic code isn't the opposite of agentic; it's the spine that makes agency safe.
Relationship to the orchestrator-placement spec This does not replace the 2026-05-21 spec — it builds on it. That spec fixed where orchestration runs (the command layer, main session). This spec fixes what form orchestration takes (a typed state machine in code, not prose) and what the agents become (a few agentic cells, not eight scripted ones). "Commands orchestrate, everything else is a terminal leaf" still holds verbatim.

1. Agent-first is not agentic

They answer different questions. Agent-first is a topology — "what are the components made of?" — and the answer was "all agents." Agentic is a behavioral property — "does this component have genuine agency?" — meaning it perceives state, chooses among genuinely open options, acts, observes, and adapts. Agency requires a decision space that is not pre-determined.

They are orthogonal. A component can be an agent structurally and a script behaviorally:

Agentic — real latitudeNot agentic — fixed procedure
Agent-shapedbrain (partially)distribution, analytics, tracking, experiment-os  ←  the bad quadrant
Code-shaped—the MCP servers (correct)

The agents say it themselves: distribution.md — "a router, not a reasoner"; analytics.md — "never compute composites yourself"; experiment-os.md — "both modes are pure orchestration"; tracking fires two fixed events. Of 8 agents, ~3.5 do work that needs an LLM. The other four sit in the bad quadrant: the full cost of LLM nondeterminism, none of the benefit of agency.

Why the bad quadrant exists It is not an accident of agent-first — it is a consequence of it. With everything an agent but nothing deterministic underneath, you cannot safely give an agent latitude — so you script it to death with 700-line procedures (media-production.md) that contain sha256 formulas, "evaluate this assertion verbatim" pseudocode, and inlined regression post-mortems (F2, the G-series). Agent-first without a code spine forces you out of agentic. A 707-line prompt has every rigidity of code and none of its guarantees — no type checker, no test, no stack trace, a different result every run.

2. The agentic cell

An agentic cell is the unit of the rebuilt system. It has five parts plus two properties:

OBJECTIVE   an outcome, machine-checkable — never a procedure.
            "every approved Script has complete, rendered, spec'd Variants"
            not "Step 1 … Step 8".
TOOLBOX     deterministic tools (MCP + pure functions) + agentic sub-units
            (a generator agent; fan-out workers).
FREE LOOP   perceive -> decide -> act -> observe -> repeat. Real latitude over
            the sequence — not a fixed checklist executed by an LLM.
VERIFIER    deterministic code — the acceptance test. The cell loops until it
            passes. This is where the old "Step 7.0 assertion" goes to LIVE.
GATE        the human judgment the verifier cannot encode (approve / reject / edit).

+ CLOCK       what triggers the cell (a run advancing; the calendar).
+ CARDINALITY its relation to a run (1:1 / N:1 / perpetual).

Clock and cardinality decide cell boundaries. Two units with different clocks cannot be one cell — a cell has one objective and one verifier, and a verifier can only assert "done" against one clock. (This is the instrument that resolved the organic question — see §7.)

The two litmus tests

  1. "Could this line be wrong in a way a unit test would catch?" → then it is code, not prompt. sha256 IDs, char-boundary truncation, slug regex, name→ID maps, the Step-7.0 assertion — all become pure functions in packages/shared, tested.
  2. "Does this step make a judgment a reasonable human would argue about?" → if no, it is not an agent. tracking, analytics, distribution, experiment-os dissolve into deterministic stages.
The contract inversion In agent-first-not-agentic, the contract lives in the prompt as instructions the agent must faithfully execute ("evaluate BOTH assertions verbatim"). In an agentic cell, the contract lives in code as the environment's acceptance criteria — and the agent has freedom to satisfy it however it likes. The agent gets more latitude in the middle and zero latitude on the outcome. More agentic and more reliable at once. The deterministic code is the scaffolding that makes agency safe — tools the agent wields, a verifier that grounds it.

The new piece — the orchestrator MCP (a typed run-state machine)

This is the keystone, and the answer to the orchestrator-placement spec's Open Decision 1 ("eleven near-identical orchestrators is a maintenance smell"). Orchestration logic — sequencing, output threading, gate conditions, acceptance tests — is deterministic control flow. It does not belong in eleven prose command files any more than it belonged in one prose brain.md. It belongs in code, exposed as an MCP: 

orchestrator.plan(runId | args)        -> the next `step` (discriminated union)
orchestrator.verify(runId, stepId, x)  -> { ok, problems[] }
orchestrator.advance(runId, stepId, x) -> persist result, move the run's stage

plan is a pure function of persisted run state — "what is the next step" is computed as the diff between current state and the objective. That single property gives the system resumability for free (see §6). The loop is then a generic engine plus one declarative stage definition file per stage. The brain.md §E spawn-prompt formats relocate here as typed buildSpawnPrompt functions — not prose templates.

4. The /loop conductor & the operator model

Every command collapses to one thin, domain-agnostic conductor. Its entire body is the loop below — it never mentions tracking, briefs, HG1, a runId format, or any marketing concept. That absence is the proof the deterministic logic actually left the prompt. 

repeat:
  step = orchestrator.plan(runId)
  switch step.kind:
    spawn  -> Task(step.agent, step.spawnPrompt)          # verbatim
    fanout -> Task × N general-purpose workers, parallel  # verbatim
    write  -> execute step.calls against the named MCP    # verbatim
    gate   -> print step.message · STOP (hand to human)
    done   -> print · STOP
  v = orchestrator.verify(runId, step.step_id, result)
  if v.ok        -> orchestrator.advance(...) ; continue
  if not v.ok    -> [THE ONE JUDGMENT POINT] transient? retry once.
                    structural? print v.problems · STOP.

There is exactly one place the LLM thinks: the verify-fail branch — a complete agentic cell in miniature (objective, toolbox, free loop, deterministic verifier). Everything else is mechanical dispatch, which is correct, because dispatch has no agency.

What it is like to operate

Today the operator is the state machine — remembering the runId, the stage, which of eleven commands comes next. The machine owns that now. The operator surface collapses to /loop (advance the current run to the next gate) and /status (where is everything). The operator does judgment at the gates and nothing else: 

── GATE SPRINT (an afternoon — compute is fast; YOUR review is the bottleneck) ──
 you  /loop
 sys  ▶ tracking ✓ analytics ✓ brain ✓ brief-writer ✓
      "run_8a2f — 12 Briefs await in Notion → [link].  HG1."
 you  [Notion: approve 6, reject 6, edit 2]                     ~10 min  ← judgment
 you  /loop   → content-writer + 6 fan-out workers  → "HG2: 18 Scripts → [link]"
 you  [approve 9]                                               ~20 min  ← judgment
 you  /loop   → produce: creative-director + 11 render workers  → "HG3: 30 Variants → [link]"
 you  [approve 22]                                              ~15 min  ← judgment
 you  /loop   → schedule (deterministic) → distribute  → "HG4: 8 ads PAUSED in Meta"
 you  [Meta Ads Manager: un-pause]                              ~5 min   ← the only non-Notion gate

── QUIET (7 days) ──   /status → "run_8a2f LIVE. Readout Mon 28th."

── READOUT (automatic — no gate; internal writes only) ──
 you  /loop   → analyze ✓ reflect ✓
      "CPA MYR 6.10 (pred 4–7 ✓) · 2 hypotheses confirmed → Learnings · next cycle ready."

You never type a runId, never track a stage, never choose a command. Your whole job is taste and compliance authority at four gates — the irreducibly human part.

5. Anti-rabbit-hole governance

A cell may loop internally until it satisfies its outcome — but the loop and the judge of "done" must be separate. The agent owns the effort; code owns the termination. The agent never adjudicates its own completion and never grants itself more budget. Five layers enforce it:

  1. A machine-checkable acceptance test. The biggest lever. If the outcome cannot be written as a verify() function, the cell has no termination signal and must not loop toward it.
  2. Harness-enforced budget caps. Tool-call budget and the finite context window are enforced by spawn config and runtime — not a prompt instruction. An LLM told "stop after 10 tries" in prose sometimes will not; a harness cap always holds.
  3. Bounded retry at the conductor, not the agent. On verify failure the conductor retries at most once, only for transient causes. The agent does not vote itself another attempt.
  4. No-progress detection. If attempt N+1 returns the same problems as attempt N, stop — retrying is not helping.
  5. The human gate. Nothing ships unreviewed; a rabbit hole's blast radius is wasted compute on one turn, never shipped output.

A rabbit hole needs two conditions: no machine-checkable "done" and no externally-enforced budget. The architecture removes both.

Genuine reasoning lives in exactly three fenced places: creative-director (scene segmentation, hook-register choice — bounded by the Script body, the fixed matrix, a tool-call cap), each render worker (layout within the brand contract — bounded by its context window and the per-run image cap), and the conductor (the one verify-fail judgment). The deterministic spec derivation — the old Steps 5.7 / 5.8 / 8, ~200 lines of prose — now runs as deriveSpecs() inside plan(); the conductor never sees it, it just gets exact write calls.

Verifier — the centerpiece, uncheatable

// orchestrator MCP · deterministic · unit-tested · the cell cannot fake passing it
function verifyProduce(runId): { ok: boolean; problems: string[] } {
  const problems = [];
  const scripts  = notion.approvedScripts(runId);
  const variants = notion.variants(runId);

  // 1 — coverage: every approved Script got its full format matrix
  for (const s of scripts)
    if (variants.filter(v => v.scriptId === s.id).length !== FORMAT_MATRIX.length)
      problems.push(`script ${s.id}: incomplete variant matrix`);

  // 2 — assets: every static AND video Variant has a resolvable rendered asset
  for (const v of variants.filter(needsAsset)) {
    if (v.assetFiles.length === 0)
      problems.push(`variant ${v.id}: Asset Files empty — render gap`);
    for (const a of v.assetFiles)
      if (!assetStore.resolves(a.url, a.sha256))
        problems.push(`variant ${v.id}: asset unresolved / hash mismatch`);
  }

  // 3 — specs: channel coverage complete + compliant (Channels drives this)
  for (const v of variants) {
    if (intersects(v.channels, META)  && !v.metaSpecComplete())
      problems.push(`variant ${v.id}: Meta spec incomplete (EN/BM)`);
    if (intersects(v.channels, META)  && !hasRegulatorPhrase(v.metaPrimaryText))
      problems.push(`variant ${v.id}: Meta Primary Text missing regulator phrase`);
    if (intersects(v.channels, ["Meta-organic"]) && !v.organicSpecComplete())
      problems.push(`variant ${v.id}: organic spec incomplete (Caption/Hashtags EN+BM)`);
    if (v.complianceCheck !== true)            // server-set by the notion MCP scan
      problems.push(`variant ${v.id}: compliance scan did not pass`);
  }

  // 4 — self-consistency (kills B-003 permanently)
  if (sum(variants.map(v => v.estCostMYR)) !== reportedTotal(runId))
    problems.push(`totals.cost drifts from per-variant sum`);

  // 5 — the F2 contract: static variants exist => workers actually ran.
  //     Reads the events table — GROUND TRUTH — not an agent's self-report.
  const staticN = variants.filter(isStatic).length;
  if (staticN > 0 && events.count(runId, "render-worker:done") === 0)
    problems.push(`${staticN} static variants, 0 render workers ran — direct-render bypass`);

  return { ok: problems.length === 0, problems };
}

Two things prose never could. The old Step 7.0 trusted the agent's self-reported workersSpawned count — the agent grading itself. Check #5 reads the events table instead — ground truth the agent cannot author. And the whole thing is a unit test: the contract is regression-proof, not regression-documented (the G-series).

       → verifyProduce → ok:false  → conductor retries fanout once, still fails
       → SURFACE, stop

This works only because plan is a pure function of persisted run state, not a step counter. If Open Decision 1 were answered with prose-includes instead of code, re-entry would re-run P1 and double-create Variants. This point and the orchestrator-placement spec's Open Decision 1 are the same decision.

7. The organic decision

Decided this session Organic content needs production too. Production's objective includes organic artifacts. Organic becomes 1:1 with a run (scope = this run's Variants; accepted drawback: inventory starvation if runs go sparse — unlikely, a run overproduces organic inventory). Organic-as-a-separate-cell dissolves.

Consequences, each traced:

  • Reel video → Production-owned. A Reel needs a video asset regardless of channel; rendering is a property of the artifact, not the channel. Production renders every asset — static PNGs via render workers, Reel video via HeyGen / kie.ai. Reel HeyGen Job ID is a Production-owned render-tracking field.
  • Scheduling is extracted — as a deterministic step, not an agent. Slot assignment, sort, MYT→UTC conversion, the publish-window check: zero open decision space. By litmus test 2 it is not agentic. It is an assignSchedule() pure function plus a write step that runs after HG3 and stamps Scheduled For. It is not placed inside distribution's routing — that would violate ADR-018 (distribution must not derive); it also fixes the existing §4d smell where the router does timezone math.
  • Organic vs paid scheduling — shared concept, different mechanism. Organic = a per-post publish datetime the FB/IG API enforces. Paid = an adset flight window (largely already the experiment's launchWindow + durationDays). One assignSchedule(), two branches keyed off Channels.
  • Organic Status is retired as a gate. With HG3 now approving organic artifacts, a second organic-only approval adds nothing — channel-level opt-out already exists (a human edits Channels at HG3 to drop Meta-organic). Organic Status survives only as a lifecycle field (Scheduled | Published | Failed) the system writes. HG3 is the single content gate for everything.

Net: the cell count drops (no organic cell), one deterministic schedule stage is added, one gate is removed. The decision simplifies the architecture.

10. Decisions & non-goals

All §10 open decisions were resolved on review (2026-05-22):

  1. Run-state store → two SQLite tables, no Notion mirror. runs (run_id, stage, status, params) + run_steps (run_id, step_id, status, result JSON, problems, attempts) in data/engineerdad.sqlite; the existing events table is the audit trail. Sync is asymmetric, not a mirror: SQLite owns orchestration state; Notion owns artifacts + gate signals; the orchestrator only reads Notion approval columns to detect a cleared gate, and verify re-reads Notion for artifact truth (doubling as the divergence detector). Orchestration state is never written to Notion.
  2. Orchestrator → logic package + thin MCP wrapper. packages/orchestrator/ holds the pure logic (stage definitions, plan/verify/advance, verifiers) and is what unit tests target; mcp-servers/orchestrator/ is a thin stdio wrapper. Per ADR-005's thin-adapter doctrine — same shape as analytics/notion. The command can only invoke tools, so the MCP transport is required; the package keeps the logic testable without stdio.
  3. No-MCP-mesh rule retained, restated precisely. The rule is "no MCP server opens a client connection to another MCP server" — not "deterministic code can't call deterministic code." Shared logic goes in a packages/ library; cross-server sequencing goes through the hub (command + orchestrator). Worker output → Notion: the orchestrator computes the write calls, the command executes them against the notion MCP, the compliance scan fires at the notion boundary. Kept because it preserves the auditable choke point the compliance model depends on.
  4. Granular commands retained — /produce, /targeting etc. stay as power-user escape hatches alongside /loop.
  5. One new ADR-020, sourced from both this spec and the orchestrator-placement spec; supersedes the 4-tier dispatch model; retains ADR-005's thin-adapter doctrine, no-MCP-mesh rule, and compliance boundary; resolves ADR-014's worker split. ADR-005 gets a one-line status note pointing to ADR-020 — not a rewrite. Written after the implementation plan locks (an ADR records a settled decision).

Non-goals: changing the Notion schema wholesale (additive only); changing the compliance doctrine (scan stays MCP-side); the v1.5 video-generation pipeline (E-004); changing the closed-loop sequence beyond the §7 organic adjustment.

Next step after review: an implementation plan via the writing-plans skill — sequencing the six migration steps with checkpoints. The worked Production cell (§6) doubles as the first concrete orchestrator stage definition and the template for the rest.
