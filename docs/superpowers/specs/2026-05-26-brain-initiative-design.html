1. Why this programme exists — the data-driven marketing read

Brain quality is the compounding edge of this OS. Every other stage (brief, content, produce, distribute) is mostly a deterministic transformation of Brain's decisions. The thing that gets sharper over time — or doesn't — is Brain's prediction calibration: do Brain's bets land where Brain said they would, often enough that the next cycle's bets are tighter?

Today that compounding doesn't happen for three structural reasons.

(1) The loop has never completed end-to-end on the current branch.

packages/orchestrator/src/stages/experiment.ts:18 flags that "experiment-design inputs Brain writes into run.params" was never wired. Today, X2-design reads run.params as unknown as ExperimentParams, sees undefined, calls mcp__experiment__design({ factors: [] }), and the design library throws. Every cycle halts at the experiment stage. #Hypotheses graded per week = 0. Calibration data accumulates at zero rate.

(2) The reasoner is architecturally mismatched to the system it reasons over.

The current brain agent runs the §B 9-step scaffold end-to-end as one Opus context across four channels (Meta-paid, Meta-organic, YouTube, Authority) — channels that grade independently (per-channel rules in §B step 2) but reason monolithically. A Meta-paid CPA forecast gets unconsciously anchored by Meta-organic engagement signal; an Authority article's GSC prediction borrows uncalibrated confidence from a Meta-paid winner. Four distinct evidence regimes are compressed into one context window.

(3) The adversarial check is too soft.

The current self-critique is a paragraph re-read as paragraph context by the next cycle's monolithic Brain. Prose criticism does not reliably tighten the next prediction's confidence interval. Hypotheses get refuted without the topology learning why in any structured way.

What a data-driven marketer would do faced with these three: ship the measurement first, generate baseline, then upgrade the reasoner against measurable evidence. Not the other way around. This programme is that sequence.

2. North-star metric & per-phase merge gates

North star: graduation_rate = #Hypotheses (Open → Confirmed) per N completed cycles

Single number, monotonic, derived from performance_reports × hypotheses.predictions_history. Each phase contributes by removing a different blocker on that number.

Why graduation_rate, not ci_hit_rate

CI-hit-rate measures the reasoner alone. A Brain that emits CPA MYR 1–100, 95% CI wins CI-hit-rate and tells you nothing — the v3 spec's gate doesn't catch this. Graduation rate forces three things to be simultaneously true: (a) Brain emits a bet specific enough to be graded, (b) the experiment has enough power to confirm/refute it, (c) calibration is honest enough that confirmations are real. CI-hit-rate is retained as the secondary diagnostic.

Why ≥5 replays at the Phase 2 gate (and not 10/20)

5 is the cheap-but-not-cheating middle. If v3 wins 4/5 on graduation rate, binomial p ≈ 0.19 vs. null — informative, not proof. If v3 wins 5/5, p ≈ 0.03 — defensible. 1–2 replays = pure luck; 20+ = months of elapsed time for marginal certainty. Solo-consultant tempo argues for 5 + a written promise to re-evaluate at 10 and 20 post-merge.

Why pre-merge, not "merge then evaluate"

v3 is not a free upgrade.

17× v2's Opus spawn cost per cycle (~$650/year vs. ~$40/year at current scale, out of your pocket) and 7× the prompt files to maintain. Reverting v3 once it's in main is a multi-file, doctrine-touching, 6-prompt unwind — by the time evidence shows v3 isn't better, sunk-cost gravity will rationalize keeping it. Also: critic transcripts compound — bad v3 cycles seed the next cycle's critic with bad priors. The longer we run before checking, the more compounded damage to unwind. The pre-merge gate forces evidence to land while the change is cheap to reverse.

4. Phase 0 — experimentParams (E-035 fold-in)

4.1 Problem

packages/orchestrator/src/stages/experiment.ts:18 flags that Phase 6 (Brain → run.params wiring) never landed. X2-design reads run.params as unknown as ExperimentParams, sees undefined for every field, calls mcp__experiment__design({ factors: [] }), and the design library throws factors must be non-empty at packages/experiment/src/design.ts:34. Every cycle halts there. The only known unblock is hand-editing runs.params JSON in sqlite — a dangerous one-off.

4.2 Decisions

  1. Ownership. Brain emits experimentParams in its Decision Memo. Engine lifts it into the experiment stage via the synthesize step result.
  2. Factor model. One factor ("angle") with levels = recommendedAngles[]. v1 only. Multi-factor (angle × persona) is a Brain-prompt-only upgrade because the schema is factors[] from day one — see E-039 in §11.
  3. No run.params mutation. The experiment stage reads the resolved memo from run.steps[].result for S1-reason. resolveRefs() already dereferences claim-check refs at the MCP boundary.
  4. Legitimate-skip path. When Brain has only one recommended angle (cold-start, single proof archetype), it emits no experimentParams. X2-design and X3-write detect this and return empty calls: []. The run flows past experiment into distribute unchanged.

4.3 Type lineage — packages/shared/src/types/brain.ts (NEW)

Phase 0 creates a real shared type file so v2 has a defined shape that v3 can extend in Phase 2.

export interface ExperimentParams {
  hypothesis: string;
  factors: Array<{ name: string; levels: string[] }>;
  holdConstant: string[];
  primaryMetric: "cpa" | "hook_rate" | "thumbstop" | "ctr";
  dailyBudgetMyr: number;
  durationDays: number;
}

export interface DecisionMemoV2 {
  schemaVersion: 2;
  runId: string;
  memoId: string;
  recommendedAngles: string[];
  personas: string[];
  topCreatives: unknown;        // pass-through from analytics
  hypothesisIds: string[];
  banditAllocation: unknown;    // pass-through from analytics
  experimentParams?: ExperimentParams;   // absent = Brain declined
  notes?: string;
}

Naming convention: camelCase, matching the rest of the memo. The experiment library's snake_case (hold_constant, primary_metric, etc.) translation lives in the X2-design build, not in the schema.

Why multi-factor-shaped on v1: when Brain later learns to reason about angle × persona, the engine accepts the wider shape with no code change. Locking v1 to a flat angles: string[] shape would require synchronized changes across three files later. Forward cost: a 3-token shape (factors: [{...}]) instead of (angles: [...]).

5. Phase 1 — Calibration suite

5.1 Why this phase

v3's merge gate needs measurable evidence. Today, with graduation_rate = 0 from zero cycles, no evidence exists. Phase 0 starts producing cycles. Phase 1 turns those cycles into a deterministic replay harness — the same input always produces the same scores — so v3 can be compared to v2 head-to-head, fairly, without waiting months for live cycle accumulation.

5.2 Snapshot-on-write hook (engine change)

packages/orchestrator/src/stages/synthesize.ts gains a hook at the start of the stage that writes the inputs Brain saw into a deterministic location:

data/snapshots/<runId>/ analytics-top-creatives.json analytics-cost-per-angle.json analytics-decay-curve-<adId>.json (one per top-3 spender) analytics-bandit-allocate.json store-hypotheses-pending.json store-learnings-active.json store-performance-reports-prior3.json corpus-compliance-block-en.txt corpus-compliance-block-ms.txt run-meta.json (runId, window, timestamp, schema_version)

The hook reads from the same MCPs Brain would call but persists the responses before spawning the synth agent. Brain's actual MCP calls then return the same data (read-through cache: if a snapshot exists for this runId, return it; otherwise call MCP and write the snapshot). The cache is keyed on runId so it's a one-shot per cycle, no staleness risk.

Per-cycle disk cost: ~5–20 KB. At 52 cycles/year ≈ 0.5–1 MB/year. Trivially cumulative.

This hook is the deterministic-input foundation. Without it, replays would read live analytics data which mutates daily.

5.3 The suite — tools/eval/calibration-suite/ (NEW)

Directory layout:

tools/eval/calibration-suite/ README.md — how to run, what gates exist replay.ts — replay a snapshot through a Brain version score.ts — graduation_rate + ci_hit_rate scorers gate.ts — exits 0 iff gate criterion holds fixtures/ <runId>/ — symlinks to data/snapshots/<runId>/ reports/ YYYY-MM-DD-<gate>.json — gate result per run

CLI entry points (wired in package.json):

pnpm calibrate replay <runId> [--brain-version v2|v3]
  → runs the named Brain against the snapshot; writes a Decision Memo
    to data/snapshots/<runId>/memo-<brain-version>.json

pnpm calibrate score <runId>
  → loads memo-v2.json and memo-v3.json + the historical outcomes
    (the hypotheses rows for this runId, post-Reflect grading);
    emits a row in reports/.

pnpm calibrate gate phase-2|phase-3
  → loads all reports for the last N=5 runs; checks gate criterion;
    exits 0 on pass, 1 on fail; prints summary table.

5.4 Scorers

graduation_rate(memo, outcomes) — for each Hypothesis the memo emitted, look up the outcome's eventual status (Confirmed / Refuted / Inconclusive). Return #Confirmed / #total. A cycle's graduation_rate is averaged across its emitted Hypotheses.

ci_hit_rate(memo, outcomes) — for each Hypothesis, did the actual outcome land inside the predicted CI band? Returns #hit / #total.

Both scorers are pure functions of (memo, outcomes). Deterministic given the inputs. Two runs of pnpm calibrate score <runId> produce identical output — asserted in tests.

5.5 Phase 1 tests

  • replay.test.ts — deterministic: replay the same snapshot twice, assert identical Decision Memos (modulo the runId, which is the snapshot's own).
  • score.test.ts — golden-file: fixture snapshots with known outcomes, scorers produce expected graduation_rate and ci_hit_rate values.
  • gate.test.ts — gate logic: feed synthetic per-run reports, assert pass/fail per spec.
  • Cold-start handling: a snapshot with no prior Hypotheses scores 0/0 graduation_rate (returns NaN-handled-as-0, not crash).
  • Snapshot hook integration in packages/orchestrator/src/stages/synthesize.test.ts: the synth stage writes the expected files in the expected location.

6. Phase 2 — Brain v3 MoE-Critic (folded-in spec, edited for inheritance)

This section is the v3 spec rewritten to inherit Phase 0's contracts. The v3 design's substance is preserved; the edits are to (a) reference the Phase-0 DecisionMemoV2 instead of assuming one, (b) move experimentParams derivation into the judge's EMIT step, (c) update the merge gate to graduation_rate-as-primary.

6.1 Failure mode v3 addresses

The current monolithic Brain reasons across 4 channels in one context window. Channels grade independently but reasoning doesn't. Cross-channel context bleed leaks into per-channel predictions; the adversarial check (paragraph self-critique) is too soft to tighten CIs reliably across cycles.

6.2 Topology — 5 phases, one debate graph

synthesize stage (under /loop) ├── Phase 1: GENERATOR R1 fanout (4 spawns in parallel, ~30s) │ ├─ brain-channel-paid → ChannelCandidatePacket │ ├─ brain-channel-organic → ChannelCandidatePacket │ ├─ brain-channel-youtube → {noSignal:true} (stub) │ └─ brain-channel-authority → {noSignal:true} (stub) │ ├── Phase 2: CRITIC R1 fanout (parallel per non-stub channel, ~25s) │ ├─ brain-critic[paid] → CalibrationAttack │ └─ brain-critic[organic] → CalibrationAttack │ ├── Phase 3: GENERATOR R2 fanout (parallel; revise attacked moves only, ~20s) │ ├─ brain-channel-paid → revised ChannelCandidatePacket │ └─ brain-channel-organic → revised ChannelCandidatePacket │ ├── Phase 4: CRITIC R2 fanout (parallel; final attack residual, ~15s) │ ├─ brain-critic[paid] → final CalibrationAttack │ └─ brain-critic[organic] → final CalibrationAttack │ └── Phase 5: JUDGE spawn (sequential after fanout, ~40s) └─ brain-judge → reads 4× FinalChannelPacket → runs §B steps 6–9 portfolio-wide → emits Decision Memo v3 (extends V2) → derives experimentParams from chosen actions → writes Hypotheses + PerformanceReports → persists debate_transcripts on PerformanceReport

Up to 17 spawns/cycle in full mode; 11 in stub mode (YouTube + Authority generators stub out and skip critic + R2). Wall-clock ~130s vs. current ~45s — channels parallelize within each phase; the longest path is max(gen R1) + max(critic R1) + max(gen R2) + max(critic R2) + judge ≈ 30+25+20+15+40s.

Key topology properties

  • Debate is intra-channel, not cross-channel. Preserves parallelism. The failure mode being fixed ("channels grade independently but reasoning doesn't") is solved by giving each channel its own reasoner, not by making one reasoner think harder.
  • The judge is the only writer to performance_reports / hypotheses. Single audit trail, single compliance scan, single runId propagation point (preserves ADR-011, ADR-005 thin-adapter doctrine).
  • CalibrationAttack transcripts persist on performance_reports. Next cycle's critics retrieve them as priors. Topology is self-sharpening across cycles — see §6.7.
  • Cold-start safe. Stub channels return {noSignal:true} packets; judge handles them explicitly. No special-casing in the orchestrator.

6.7 Calibration compounding — the self-sharpening mechanism

Every CalibrationAttack persists on the performance_reports.debate_transcripts JSONB. Next cycle's critic does an explicit retrieval:

// brain-critic, Phase 2, grounding step
const prior = await mcp.store.query({
  entity: "PerformanceReports",
  filter: { channel },
  last: 5
});
const myPriorAttacks = prior
  .flatMap(p => p.debate_transcripts)
  .filter(t => t.channel === channel)
  .flatMap(t => t.criticR2Final.attacks);
// grouped by attackType: "this critic for meta-paid has historically tightened CI
//   by avg 38% on ci_too_tight attacks; 76% of those tightenings were correct
//   vs. eventual outcome"

A critic that over-attacks (always says CI is too tight when actuals land inside the original CI) gets evidence of its own miscalibration and self-corrects. The judge precomputes criticCalibrationSummary at emit time (rolled-up per-channel critic accuracy from the last N runs) so the retrieval is one query, not a recomputation.

This is the recursive Reflect layer that monolithic Brain cannot have because there is no division of labor to grade.

6.8 Quiescence — critics may return {attacks: []}

Quiescence allowed.

If the generator's predictions are well-grounded, the critic emits no attacks. Don't manufacture them. This prevents FUD from a misfiring critic and lets a well-calibrated critic explicitly assent. The judge weighs an empty criticR2Final as endorsement, not missed work.

7. Phase 3 — Generic critic step kind (E-033, scoped to synthesize)

7.1 Why this phase

Phase 2's critic agents are coupled to the synthesize stage's bespoke fanout. Phase 3 generalizes them into a first-class conductor step kind that any stage can opt into via its verifier. Synthesize is the canonical implementation; brief / produce / distribute become E-036 / E-037 / E-038, dormant in TASKS.md until their trigger conditions hit (see §11).

7.2 New step kind

packages/orchestrator/src/types.ts gains:

export interface CriticStep {
  kind: "critic";
  stepId: string;
  spawnPrompt: string;
  inputRefs: string[];          // claim-check refs into orchestrator.step_results
  channel?: string;             // optional channel parameterization
}

export type Step = WriteStep | SpawnStep | FanoutStep | CriticStep;

packages/orchestrator/src/exec.ts adds the dispatch arm: a CriticStep is a constrained spawn that takes only inputRefs (no outputRefs until the critic writes one), targets the brain-critic agent (or a parameterized variant), and emits a CalibrationAttack shape.

7.3 Migration of synthesize's v3 critics

Phase 2 ships synthesize's critic phases as fanout steps over brain-critic spawns. Phase 3 replaces those with kind: "critic" steps. Behavioral diff: zero — golden-file test asserts step_results rows are byte-identical pre/post migration. The change is structural: the orchestrator now knows what a critic is.

7.4 ADR-026 final draft (accepted on Phase 3 merge)

Single doctrine doc, two parts:

Part A — MoE-Critic topology for synthesize

  • The synthesize stage uses a 5-phase MoE-Critic debate graph: 4 channel-specialist generators → critics R1 → generators R2 → critics R2 → portfolio judge.
  • Debate is intra-channel only (cross-channel deferred to E-040).
  • The judge is the only writer to performance_reports / hypotheses.
  • Critic transcripts persist on performance_reports.debate_transcripts to make calibration self-sharpening across cycles.

Part B — Generic kind: "critic" step

  • A first-class conductor step kind for verifier→delegate flows where structural verification is insufficient.
  • Initial implementation: synthesize's v3 critic agents.
  • Opt-in by stage: a stage's verifier decides whether to delegate to a critic step; the orchestrator dispatches.
  • Composes with ADR-022 (claim-check transport), ADR-023 (orchestrator-resident execution), ADR-024 (staged input).
  • Future stages (E-036/037/038) adopt independently.
