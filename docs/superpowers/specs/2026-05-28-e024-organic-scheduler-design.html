1. Context

The task is E-024 (TASKS.md): build the always-on organic-post scheduler. The trigger is B-005: Meta's Instagram Content Publishing API has no native scheduled-publish support — scheduled_publish_time is silently ignored and the post goes live immediately. Three IG posts published instantly during the 2026-05-21 smoke test (run_1778486942) before the manual deletes. ADR-019's amendment delegates the fix here.

Today, IG publishing is fail-closed disabled at the MCP tool boundary (mcp-servers/meta-organic/src/ig-guard.ts); approved IG posts go through scripts/build-posting-pack.mjs (the /posting-pack command) as a read-only pack the user manually posts from. E-024 restores a real schedule + cancel window for IG.

Carry-forward from the 2026-05-21 brainstorm (parked in TASKS.md E-024):

  • n8n's Wait node is durable (serializes paused executions to its DB, survives reboot).
  • Two trigger models on the table: poll (cron queries Notion every ~5 min) vs push (/distribute registers a per-post job).
  • A push variant that holds only {variantId, platform, fireAt} and re-reads source at fire-time converges with poll's freshness + cancel-safety.
  • Host decided: self-host on the Mac mini (pmset no-sleep).

3. Design — §1 Architecture overview

A new sibling Node process apps/scheduler/:

apps/
├── webapp/        ← Next.js review UI (next start, port 3030)
└── scheduler/     ← NEW: always-on worker (single Node process)
                     - polls scheduled_jobs every 60s
                     - claims due rows with SELECT FOR UPDATE SKIP LOCKED
                     - executes via direct import of @engineerdad/meta-organic functions
                     - updates row state
                     - runs under launchd on Mac mini, restart-on-crash

Both processes share: the Postgres DB (engineerdad), @engineerdad/store for typed access, the meta-organic publish functions (transport-decoupled plain async fns), and packages/shared/src/compliance.ts for pre-flight scanning.

Webapp owns user-facing concerns: queue view, "Schedule" button, cancel/edit, status badges. Scheduler owns execution: tick, claim, publish, retry, settle. The only coupling is the shared DB table.

Why not inside the Next.js process: Next.js is request-driven. Embedding the tick means UI deploys interrupt the scheduler mid-claim, next dev HMR double-fires intervals, and one process's OOM takes both down. Two launchd plists vs one buys reliable separation.

4. Design — §2 Data model

A new scheduled_jobs table — a thin job ledger. Holds just enough to claim and dispatch, then re-reads the source entity at fire-time (matches the parked 2026-05-21 decision).

// packages/store/src/schema.ts
export const scheduledJobs = pgTable("scheduled_jobs", {
  id: uuid("id").primaryKey().defaultRandom(),

  // Discriminator — first user is 'publish_ig_image' / 'publish_ig_carousel',
  // but the table is general-purpose for future job types.
  jobType: text("job_type").notNull(),

  // The source entity this job acts on (re-read at fire-time).
  variantId: text("variant_id").references(() => creativeVariants.id),

  // Lifecycle
  status: text("status").notNull().default("queued"),
    // queued | claimed | done | failed | cancelled
  fireAt: timestamp("fire_at", { withTimezone: true }).notNull(),
  claimedAt: timestamp("claimed_at", { withTimezone: true }),
  claimedBy: text("claimed_by"),               // worker hostname:pid
  doneAt: timestamp("done_at", { withTimezone: true }),

  // Retry
  attempts: integer("attempts").notNull().default(0),
  maxAttempts: integer("max_attempts").notNull().default(3),
  nextRetryAt: timestamp("next_retry_at", { withTimezone: true }),
  lastError: text("last_error"),

  // Result (set on success)
  publishedPostId: text("published_post_id"),  // Meta's post ID for traceability

  createdAt: timestamp("created_at").notNull().defaultNow(),
  updatedAt: timestamp("updated_at").notNull().defaultNow(),
}, (t) => [
  index("scheduled_jobs_due_idx").on(t.status, t.fireAt),  // hot path for claim
  index("scheduled_jobs_variant_idx").on(t.variantId),
]);

No payload JSONB blob — a snapshot would freeze the caption/asset at insert time, defeating the cancel/edit window. Re-read at fire-time wins.

Relationship to existing fields: the variant already has organicScheduledFor (user intent) and organicPublishedAt (settled state). The new table sits between them as the worklist:

  • HG3 approval → /distribute inserts a scheduled_jobs row with fire_at = variant.organicScheduledFor.
  • Worker claims, executes, sets published_post_id + done_at.
  • Worker writes creative_variants.organicPublishedAt so the existing entity stays the read model.

FB stays out of this table — its /distribute leg keeps calling publish_image_post with native scheduled_publish_time. Only publish_ig_* job types land here initially.

5. Design — §3 Job lifecycle

Insertion (push by /distribute)

// inside distribute stage, IG branch
await store.scheduledJobs.create({
  jobType: variant.format === "Carousel" ? "publish_ig_carousel" : "publish_ig_image",
  variantId: variant.id,
  fireAt: variant.organicScheduledFor,    // user's intended publish time
});
// FB branch unchanged — still calls publish_image_post with native scheduling

Idempotency: unique partial index on (variantId, jobType) where status IN ('queued','claimed') — re-running /distribute for the same variant cannot duplicate a queued job.

Claim (worker tick, every 60s)

BEGIN;
SELECT id, job_type, variant_id
FROM scheduled_jobs
WHERE status = 'queued'
  AND fire_at <= now()
  AND (next_retry_at IS NULL OR next_retry_at <= now())
ORDER BY fire_at ASC
LIMIT 5
FOR UPDATE SKIP LOCKED;

UPDATE scheduled_jobs
SET status = 'claimed', claimed_at = now(), claimed_by = $worker_id, attempts = attempts + 1
WHERE id = ANY($claimed_ids);
COMMIT;

SKIP LOCKED makes multi-worker safe (not needed today but free). LIMIT 5 caps single-tick blast radius.

Execute (per claimed row)

const variant = await store.creativeVariants.get(variantId);    // re-read fresh
if (variant.organicCancelled || !variant.organicScheduledFor) {
  // user cancelled or rescheduled between insertion and now
  await markCancelled(jobId, "variant cancelled or unscheduled");
  continue;
}

await preflightCompliance(variant.organicCaptionMs, variant.organicCaptionEn);
// fail-closed: throws ComplianceError on banned phrase

const postId = jobType === "publish_ig_carousel"
  ? await publishCarouselPost({ ... })
  : await publishImagePost({ ... });

await markDone(jobId, postId);
await store.creativeVariants.update(variantId, { organicPublishedAt: new Date() });

Settle (success / failure)

  • Success — status='done', published_post_id set, done_at=now(), organicPublishedAt written.
  • Compliance failure — status='failed', last_error='compliance: …', no retry; alert via webapp queue red badge. Human edits caption + re-fires /distribute.
  • Transient failure (5xx, network) — leave status='queued'; set next_retry_at = now() + backoff(attempts); exponential 1m / 5m / 30m. After attempts >= max_attempts, flip to failed.
  • Permanent failure (4xx other than rate-limit) — status='failed' immediately.

Cancel / reschedule (operator)

Webapp queue UI exposes Cancel + Reschedule on queued rows. Both are no-ops if the row has already flipped to claimed — the worker has a 60s window and an in-flight claim is uninterruptible; the operator sees a "publishing now" badge.

The ADR-019 10-minute cancel-safety window is restored here as a client-side guard at insertion: /distribute refuses to write a row with fire_at < now() + 10min. Same doctrine, enforced one layer up.
