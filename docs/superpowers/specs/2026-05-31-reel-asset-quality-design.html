Reel & static asset quality — format-aware text density, worker self-critique, and the two-type reel model

Design spec · 2026-05-31 · branch feat/reel-visual-scenes

Problem
Rendered creatives (1) show overlapping elements by chance, and (2) cram too much text — reels especially, where the voiceover should carry the explanation.
Approach
B — fix at the render layer and upstream at the authoring layer. Critique mechanism = worker self-critique (no separate critic agent/step).
Scope
One indivisible change set: density doctrine + self-critique rubric + collapse the reel scene model from three types to two (face | visual), across both the authoring and HeyGen-substrate layers, dropping emphasize.
Blast radius
Prompt files (no restart) plus schema/code/MCP + a new Opus-pinned render-worker agent (needs pnpm -r build + Claude Code restart) + ADR rewrite + tests/fixtures. Two agents change model: creative-director (Sonnet→Opus) and the new render-worker (Opus).
Supersedes
Rewrites ADR-029 in place (three-type → two-type).

1. Why — root causes (verified against the code)

  1. Overlaps "by chance." The reel render worker has no visual-QA pass in its normal path — the PNG-reading QA only runs in "FRAMES-ONLY MODE" (a dev harness). In a real run, frames are built and shipped straight to HeyGen unchecked. The static worker does have a QA loop (render-worker.md §5.5, incl. an overlap check), but it can still miss, and its anti-overlap clearances live only in brand-contract.md §4b which the worker is told to "sanity-check," not score.
  2. Too much text. Nothing caps on-frame text. onScreenText: z.string() in packages/shared/src/zod.ts:174 has no length bound; only voiceover is word-capped (face ≤30 / chart-visual ≤45) and targetSeconds is bounded. So on-frame wordiness is governed entirely by prose in the prompts today — and the prompts apply one flat rule (onScreenText ≤8 words) regardless of whether the surface has a voiceover. Reels (narrated) and Feed/Carousel (silent) need opposite density.
Key consequence: on-frame text lives in the rendered PNG, not the database, so the produce verifier cannot machine-count it. Density is enforced by worker self-critique + the HG3 human gate — the same backstop the existing "no numbers on concept visuals" rule relies on.

4. The two-type reel model

There are two "type" fields at two layers. Both collapse to the same two values.

Layer 1 — authoring (sceneType, written by creative-director)

sceneTypeWhat it isRequired fieldsNumbers on frame?
faceHeyGen avatar (Shoo) talking. Hooks, anecdotes, CTAs. First & last scene must be face.chartRef=null, visualBrief=null, explains=nulln/a (no frame)
visualData visual — chart / graph / comparison table / anything grounded in a vetted YAML.chartRef set, visualBrief=null, explains setYes — the numbers come from corpus/data/charts/*.yaml (vetted).
Concept visual — comparison/flow/metaphor/diagram with no underlying dataset.visualBrief set, chartRef=null, explains setNo — unvetted figures would be a fake-data compliance violation.
The XOR invariant (replaces the old per-type refinements): a visual scene must have exactly one of chartRef or visualBrief non-null — never both, never neither. This is what carries the compliance boundary forward after the type names merge. Refinements key on field presence, not on a type string.

Layer 2 — HeyGen render substrate (kind, inside mcp-servers/heygen-wrapper)

This enum describes the physical render mode, not the content. There are only ever two physical modes: avatar, or full-frame image background. The current names (chart, face-over-chart) are misleading — a concept illustration is also rendered as an image background. After this change:

chartvisualfull-frame image background (the rendered PNG), voiceover only — chart_url arg kept but now means "any image URL"
face-over-chart— (deleted)already dead since ADR-028; no distinct semantics

5. §8 — Text density by format (new section in brand-contract.md)

Principle: match on-frame words to whether a voiceover exists. Reels are narrated → the frame is a visual aid with a concise support line, not a paragraph. Feed/Carousel are silent → the frame must fully self-explain, so they carry a short body block. Over budget → trim words, never shrink type below §4a minimums.

SurfaceMandatory on frameDiscretionary text budgetNotes
Reel facelogonone (avatar fills frame)copy lives in HeyGen captions
Reel visual — data
(chartRef set)		the chart/table/graph, YAML caption_en, full source_citation, logoheadline ≤ 6 words + 1 support line ≤ 12 wordsnumbers OK (vetted); single-stat callout lives here; support line = explains
Reel visual — concept
(visualBrief set)		logoheadline ≤ 6 words + 1 support line ≤ 12 words + ≤ 2 labels ≤ 3 words eachno numbers (compliance, not density); support line = explains
Feedlogo (+ chart caption/citation if charted)headline + body block: 2–3 lines, ~30–45 words totalmust self-explain, no VO; body = condensed body
Carousel cardkicker, N / total indicator, logo2-line headline + body block: 2–3 lines, ~30–45 wordscard 1 (hook hero) may be headline-only; cards 2..N carry the body

Binds the rubric:

  • "Headline" = the hook / onScreenText in var(--font-head) at the §4a floor.
  • Reel mandatory chart caption + source citation are never counted against the headline/support budget and are never truncated.
  • Word counts = whitespace-split tokens (same definition as the schema's reelWordCount), counted per language (EN frame and BM frame each independently in budget).
  • This table is exactly what §9 rubric item 3 scores against.

6. §9 — Self-critique rubric (new section in brand-contract.md; both workers run it)

After rendering a PNG, the worker Reads it and writes a one-line, evidence-cited observation for each item. Any fail → fix & re-render (within the retry budget). A "looks good" with no evidence is not acceptable.

  1. No overlap — no text-on-text, text-on-chart, or headline-descender collisions. Cites the §4b vertical-clearance gaps.
  2. No edge clip — quote the bottom-most line in full; if mid-sentence it's clipped. Reels additionally respect the safe-area (top ~14% / bottom ~20% / right ~12% clear of Reels UI).
  3. Density within §8 budget for this surface — count the on-frame words; over budget → trim (don't shrink type).
  4. Legibility — meets §4a minimums at the ~37% squint test.
  5. Hierarchy / contrast — one clear focal point; palette-correct contrast (no grey-on-navy, etc.).

Why reels can afford QA now: frame QA happens at Step 2 (render the PNG), before Step 3 (upload to HeyGen) and Step 4 (generate_reel). So adding the loop costs only local re-renders — no extra HeyGen jobs, no extra money.

9. Schema & code detail — the merge

Before (current refinements, zod.ts:184–210)

sceneType: z.enum(["face","chart","visual"])
// face ≤30, chart/visual ≤45 word VO budget        [KEEP]
// chart  ⇒ chartRef !== null && visualBrief == null  [REPLACE]
// visual ⇒ visualBrief non-empty && chartRef == null [REPLACE]
// face   ⇒ chartRef == null && visualBrief == null    [KEEP, retype]
// emphasize ⇒ chart-only                              [DELETE]

After

sceneType: z.enum(["face","visual"])

// VO budget unchanged (face ≤30, visual ≤45):
.refine(s => reelWordCount(s.voiceover) <= (s.sceneType === "face" ? 30 : 45))

// face ⇒ no chartRef, no visualBrief:
.refine(s => s.sceneType !== "face" ||
            (s.chartRef === null && (s.visualBrief ?? null) === null))

// visual ⇒ XOR(chartRef, visualBrief) — exactly one present:
.refine(s => s.sceneType !== "visual" || (
   (s.chartRef !== null) !==
   (typeof s.visualBrief === "string" && s.visualBrief.trim().length > 0)
))

// emphasize field removed from the object entirely.

Renaming it to image_url is the fully-consistent move, but it touches the wrapper's external tool signature. Decide in the plan whether to rename now (cleaner) or keep chart_url as a now-generic "image URL" (smaller diff). Either is fine; this spec doesn't force it.

10. Non-goals / out of scope

  • No separate critic agent or P2b orchestrator step. Critique stays inside the existing workers (D1).
  • No machine scan of on-frame text. Density & the no-numbers rule remain worker-self-critique + HG3-human enforced (the text isn't in the DB).
  • No change to the produce stage's branching logic (renderScenes keys on format, not scene type) beyond removing the dropped field.
  • No change to deriveSpecs channel/copy derivation, the distribute stage, or the compliance footer assembly.
  • No more than 1 retry per frame/scene. A second failure is a signal to record honestly and move on; the conductor decides re-spawn.
