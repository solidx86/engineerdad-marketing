1. Context

The first real end-to-end loop walk (run_1779446750) is stuck at produce / P1-creative. Two stages still ship the Phase-6 single-spawn shape that Design §8 explicitly flagged for fan-out and never came back to:

  • content.ts C1-write — one content-writer spawn that queries every approved Brief, then authors a hook bank + 3 scripts for each one + 1–2 cross-Brief articles, all in one context window.
  • produce.ts P1-creative — one creative-director spawn that takes the run's Scripts (typically ~12) and produces 4 creatives per Script, all in one context window.

The single spawn fails for an architectural reason, not a model one:

Evidence — run_1779446750 walk, 2026-05-22..23.
  • 5 subagent dispatches; the 2 heaviest (content-writer, creative-director) both failed with Tool result missing due to internal error; the 3 lighter ones succeeded.
  • The content-writer that did return reported 144,027 tokens — ~72% of Sonnet's 200k context window for one run. Zero headroom. The earlier attempt that ran ~2× longer exceeded the window.
  • Root cause: a subagent that exhausts its ~200k window cannot emit a final message; the harness surfaces that as Tool result missing due to internal error. A subagent has no compaction — unlike the main session it gets one window and dies if it fills.
  • Model swap is not a fix: content-writer + creative-director switched sonnet → opus (2026-05-23); creative-director still failed identically. The window is the wall, not the model.

The fix is the fan-out the design always called for: replace each heavy single spawn with N narrow workers, each working in its own fresh window on a single unit of work (one Brief; one Script).

2. Goal

Make the content and produce stages survive a real run at v1 scale (12 Briefs / 12 Scripts) without any worker approaching its context window.

Definition of done:

  • run_1779446750 resumes and clears produce end-to-end on the new shape.
  • Cold-start /loop-once walks content and produce without any single worker exceeding ~60k tokens (≈30% of Sonnet's window).
  • pnpm -r build + pnpm vitest run + pnpm sync:agents:check green.
  • The C1 and P1 verifiers' contracts (hook-bank §8 Piliero rule, CreativePlan shape, 80% proof rule) hold under aggregation.

3. Non-goals

  • Re-doing register-level learning (E-028 is its own spec).
  • Changing what content-writer or creative-director produce. The artifact shapes (ContentResult, CreativePlan, hook bank entries, scene cards) are unchanged. Only dispatch granularity changes.
  • Removing the general-purpose worker pattern used by P2-render. P2 already works; it stays.

4. Current shape (what's wrong)

content stage — packages/orchestrator/src/stages/content.ts

C1-write  spawn  content-writer   → queries Briefs, authors EVERYTHING
C2-gate   gate   HG2

C1's spawnPrompt: "Query this run's human-approved Briefs and author the hook banks, scripts, and AEO/GEO authority articles from them." The agent does:

  • query Notion for N Briefs
  • for each Brief: ≥30 bilingual hooks × 6 registers + value bank + 3 scripts + Notion writes
  • 1–2 cross-Brief Authority Articles (800–1500 words each, bilingual)
  • batch proofRatio validation

For 12 Briefs that's ≥360 bilingual hooks + 36 scripts + 2 articles in one window. This is what hit 144k tokens.

produce stage — packages/orchestrator/src/stages/produce.ts

P1-creative  spawn   creative-director → CreativePlan for ALL Scripts
P2-render    fanout  general-purpose workers, one per static creative
P3-persist   write   create_page per variant + articles query
P4-enrich    write   article packaging fill-only-if-empty
P5-confirm   write   re-query Scripts + CreativeVariants; verifyProduce
P6-gate      gate    HG3

P1's spawnPrompt: "Query this run's human-approved Scripts and decompose each into the 4 distinct creatives." With 12 Scripts that's 48 creatives, each carrying a 3–8 scene shotlist × 2 languages + thumbnail brief, plus the full hookBanks JSON from C1. P2 already fans out cleanly; the wall is P1 itself.

5. Target shape

Two structural moves, parallel between stages:

  1. Add a "read approved units" write step before the heavy work. This is the cheap Notion query already inside the agents today. Moving it to the orchestrator turns the next step's plan-time build into a pure function of run state — fan-out granularity is known at plan time.
  2. Replace the heavy single spawn with a fanout — one worker per unit (one Brief; one Script). The worker keeps its agent identity (content-writer, creative-director) but gets a scoped spawnPrompt: one unit's data plus the contract for just that unit's output.

content stage — target

C0-briefs    write   query approved Briefs for this run
C1-fanout    fanout  content-writer worker, one per Brief
                      → emits { briefId, hooks[], scripts[] }
C2-articles  spawn   content-writer (article-mode) → 1–2 cross-Brief articles
C3-gate      gate    HG2
  • C0-briefs. Single mcp__notion__query for Briefs filtered to Run ID = runId AND Approval Status = Approved. The trailing-call pattern is already proven by P3-persist's articles query.
  • C1-fanout. kind: "fanout", worker: "content-writer". The build function maps approved Brief rows to one unit per Brief; each unit's spawnPrompt carries one Brief's properties + the C1-worker contract (≥30 hooks across 6 registers, ≥3 scripts per Brief, Notion writes for scripts only, proofRefs as multi_select). No Notion query of its own — the Brief data is in the prompt. The verifier aggregates over the fanout's array result and re-runs the verifyContent §8 Piliero check on the unioned hookBanks and the unioned scripts.
  • C2-articles. One small spawn for the 1–2 cross-Brief AEO/GEO articles. These need the cross-Brief view (themes spanning Briefs) so cannot be per-Brief; but the workload — 2 articles at 800–1500 words — is well under one window even at 2× headroom.
  • C3-gate. Unchanged HG2.

produce stage — target

P0-scripts   write   query approved Scripts for this run; load hookBanks from C1
P1-fanout    fanout  creative-director worker, one per Script
                      → emits one CreativeUnit[4] per Script
P2-render    fanout  general-purpose, one per static creative   (UNCHANGED)
P3-persist   write   create_page per variant + articles query   (UNCHANGED)
P4-enrich    write   article packaging                          (UNCHANGED)
P5-confirm   write   re-query + verifyProduce                   (UNCHANGED)
P6-gate      gate    HG3                                        (UNCHANGED)
  • P0-scripts. Single Notion query for the run's approved Scripts with the filter_properties projection the creative-director uses today. Reads C1-fanout's hookBanks from run.steps via stepResult — no new state shape.
  • P1-fanout. kind: "fanout", worker: "creative-director". One unit per Script. Each unit's prompt carries: that Script's properties, its parent Brief's hook bank (looked up by briefId), and the contract for producing exactly 4 CreativeUnits for that one Script. The worker never re-queries Notion. The aggregator unions the per-Script outputs into a single CreativePlan.
  • P2–P6 unchanged. P2's render-worker fanout already operates on plan.creatives flat — it's indifferent to whether that array came from one spawn or N. P3's deriveSpecs is pure. P5's verifier reads from the aggregated plan and Notion ground truth.

6. Aggregation contract

The fanout step's result is an array of per-unit worker outputs. A verifier reduces it back to the shape the downstream code already expects:

StepPer-unit outputFolded into
C1-fanout { briefId, hooks[], scripts[], proofRefs[], notes[] } ContentResult = { scripts: flat, hookBanks: [{briefId, hooks}], proofRatio: recomputed, ok: all-ok }
P1-fanout { scriptId, creatives[4] } CreativePlan = { runId, creatives: flat } — no new fields; deriveSpecs consumes it unchanged.

The §8 Piliero hook-bank check runs per hookBanks entry exactly as today; only the assembly point moves. Downstream stepResult<ContentResult> / stepResult<CreativePlan> reads are unchanged — they see the verifier-folded shape rather than a single agent's self-reported shape.

7. Agent prompt split

content-writer.md and creative-director.md are written today as "do all units in one batch." The fanout shape inverts that. Two options were weighed:

OptionHowVerdict
1 — One agent file, prompt-driven mode Add a "Single-Brief / Single-Script worker mode" section gated by the spawnPrompt shape; the agent operates only on the named unit. Chosen. Keeps agent identity (subagent_type), keeps the tools allowlist sized correctly (Notion writes stay per-unit privileges of the specialist agent), smallest diff.
2 — Per-unit worker prompts under corpus/templates/worker-prompts/ Mirror render-worker.md: fanout uses worker: "general-purpose" with a prompt that says "read this worker file and follow it." Rejected — forces general-purpose to gain Notion-write access just to materialize Scripts; breaks the specialist tool-allowlist boundary.

Batch-level validations that genuinely need the cross-unit view (proofRatio < 0.80, article generation, cross-Script hook-register diversity) move out of the per-unit agent and into either the C1-fanout aggregator or the dedicated C2-articles spawn.

8. The 80% proof rule under fan-out

Spec decision. Each C1 worker enforces proofRatio ≥ 0.80 on its own Brief's scripts. The C1-fanout aggregator re-checks the unioned ratio as defence in depth — it is mathematically guaranteed by per-worker enforcement, but the explicit aggregate check protects against a future worker that surfaces the violation only via notes.

The 20% brand-spot budget thus becomes a per-Brief allowance rather than a shared batch resource — a tighter contract, but it removes the only cross-Brief coupling C1-fanout would otherwise need to coordinate.

10. Test plan

  • C1 verifier per-unit. A single-Brief worker output with 29 hooks fails with the existing §8 message naming that Brief. With 30 hooks but missing a register, fails with the existing missing-register message.
  • C1 verifier aggregation. Three per-Brief outputs, all individually ok, produce a ContentResult whose hookBanks.length === 3 and whose scripts is the flat union with no duplicates.
  • C1 verifier 80% proof. Three workers each at exactly 80% proof produce an aggregate at ≥80%. One worker below 80% is surfaced even though the union may still be ≥80% — defence in depth.
  • P1 aggregation. Twelve per-Script outputs produce a CreativePlan whose creatives.length === 48 (12 × 4 formats). deriveSpecs over this aggregate produces the same VariantSpec[] it would have for an unbatched run.
  • Engine integration. A fixture run drives content from C0 through C3 and produce from P0 through P6 with the new shapes. Stub workers return canned per-unit outputs; the test verifies stage advancement, the verifier-folded results, and that no unit prompt exceeds a sentinel size (a coarse proxy for "no worker overloads its window").
  • No regression in pnpm vitest run — current 459 passing must hold.

11. Risks

Per-worker context still has to fit. A single Brief produces ≥30 hooks × ~20 tokens × 2 languages + 3 scripts at 30/60/90s + per-Brief Notion writes. At ~3–5k tokens for instructions + ~10k for outputs this is far under the window. But if per-worker output grows (E-028 register-rich hooks; long-form scripts), this needs to be re-checked. The engine test's per-unit size proxy catches obvious drift; the worst case is one Brief fails at its verifier and the run parks at C1 — recoverable, not catastrophic.
Notion rate-limiting. N workers each writing M Scripts is N×M parallel create_page calls. Notion's published rate limit is ~3 rps; 12 workers × 3 scripts = 36 writes inside one fanout window. Mitigation: each worker serialises its own Notion writes (a tight for-loop, not parallel). The fanout's parallelism is at the per-Brief boundary, not inside one Brief.
Stage step IDs. Step IDs change: C1-write → C1-fanout; P1-creative → P1-fanout. The blocked run_1779446750 has C1-write already done in its persisted state. Resume path is decided in the plan: either (a) treat C1-write as satisfying C1-fanout via an oldStepIds alias in the stage spec, or (b) introduce new IDs alongside the old via a registry-versioned stage. Option (a) is simpler if state lookup is by ID.

12. Open questions

  1. Stage step IDs. Rename in place (C1-write → C1-fanout) and migrate persisted state, or layer new IDs alongside? Plan picks before the build.
  2. Worker count cap. Should the fanout cap at N (e.g. max 12 parallel) to bound Notion write pressure? P2-render currently has no cap and works at run-scale, so probably no — but worth a sanity check in the plan.
  3. C2-articles overload. 1–2 articles at 800–1500 words bilingual is ~6–10k output tokens. Fine for one spawn. But the article-mode prompt currently re-reads compliance + house-style. If those grow we may need a per-article fanout. Out of scope for E-027 unless the first walk surfaces it.

13. What this unblocks

  • run_1779446750 can resume at produce once P1-fanout lands (with the stage-ID migration decided in §11).
  • A cold-start /loop-once no longer dies at C1.
  • The remaining v1.5 enhancements that build on register-level learning (E-028) gain a stable produce stage to attach to.
